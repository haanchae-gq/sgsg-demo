## 개요

### 목적
플랫폼 운영팀이 전체 시스템을 효율적으로 관리하고 모니터링할 수 있는 관리자 콘솔입니다. 역할 기반 접근 제어(RBAC)를 통한 세분화된 권한 관리와 실시간 데이터 모니터링을 제공합니다.

### 핵심 가치
- **통제력**: 전체 시스템 관리 및 제어
- **가시성**: 실시간 모니터링 및 통계 시각화
- **효율성**: 빠른 의사결정 지원
- **안정성**: 안전한 운영 환경

### 주요 기능 모듈
1. **대시보드**: 주요 지표(KPI), 실시간 주문 현황, 승인 대기 알림
2. **사용자 관리**: 고객, 전문가, 관리자 계정 관리 및 전문가 승인 프로세스
3. **주문 관리**: 전체 주문 모니터링, 상태 변경, 전문가 재배정, 환불 처리
4. **서비스 관리**: 3단계 카테고리(대분류→중분류→서비스 항목) 및 가격 관리
5. **정산 관리**: 전문가 주간 정산 처리, 승인/거절, 명세서 생성
6. **통계 및 리포트**: 매출/주문/사용자 분석, 인기 서비스 TOP 5, 리포트 다운로드
7. **시스템 설정**: 관리자 계정, 권한 관리, 시스템 설정, 로그 조회

### 기술 스택
- **프론트엔드**: React 18 + TypeScript
- **UI 프레임워크**: Ant Design Pro (기업용 관리 콘솔 최적화)
- **상태 관리**: Zustand (간단한 글로벌 상태)
- **라우팅**: React Router v6
- **HTTP 클라이언트**: Axios
- **차트 라이브러리**: Recharts
- **테이블 컴포넌트**: Ant Design Table + TanStack Table 통합
- **폼 관리**: React Hook Form + Ant Design Form
- **빌드 도구**: Vite
- **테스트**: Vitest + React Testing Library

## 역할 기반 접근 제어 (RBAC)

### 권한 레벨 정의
1. **슈퍼 관리자 (Super Admin)**: 시스템 전체 권한 (관리자 계정 생성/삭제, 모든 설정 변경)
2. **운영 관리자 (Operation Admin)**: 사용자, 주문, 서비스, 정산 관리 (전문가 승인, 주문 모니터링)
3. **CS 관리자 (Customer Support Admin)**: 고객 지원 전용 (주문 조회, 고객 정보 확인, 환불 처리)
4. **재무 관리자 (Finance Admin)**: 정산 관리, 통계 조회, 리포트 다운로드

### 권한 매트릭스 (Ant Design Pro 메뉴 기준)

| 대시보드 | 대시보드 | ✓ | ✓ | ✓ | ✓ |
| 사용자 관리 | 사용자 목록 | ✓ | ✓ | ✓ | - |
| | 전문가 승인 | ✓ | ✓ | - | - |
| | 관리자 계정 | ✓ | - | - | - |
| 주문 관리 | 주문 목록 | ✓ | ✓ | ✓ | - |
| | 주문 상세 | ✓ | ✓ | ✓ | - |
| | 환불 처리 | ✓ | ✓ | ✓ | - |
| 서비스 관리 | 서비스 카테고리 | ✓ | ✓ | - | - |
| | 서비스 항목 | ✓ | ✓ | - | - |
| 정산 관리 | 정산 목록 | ✓ | ✓ | - | ✓ |

| 시스템 설정 | 시스템 설정 | ✓ | - | - | - |
| | 로그 조회 | ✓ | ✓ | - | - |

### 권한 구현 방식
1. **API 레벨 권한**: JWT 토큰에 `role` 클레임 포함, 백엔드에서 역할별 엔드포인트 접근 제어
2. **프론트엔드 메뉴 권한**: Ant Design Pro의 `access` 프로퍼티로 동적 메뉴 렌더링
3. **컴포넌트 레벨 권한**: `usePermission` 훅으로 버튼/액션 표시 제어
4. **라우팅 가드**: `ProtectedRoute` 컴포넌트로 권한 없는 페이지 접근 차단

### 모니터링 위젯 (시나리오 기반)

#### 신규 주문 모니터링
- **위젯 유형**: 실시간 카운트 카드 + 최근 주문 목록
- **데이터 소스**: `order_status = NEW` 필터 적용
- **지표**: 현재 신규 주문 수, 시간당 신규 주문 평균, 전일 대비 변화율
- **알림**: 신규 주문 발생 시 실시간 알림 (브라우저 알림, 사운드)

#### 미배정 주문 모니터링
- **위젯 유형**: 통계 카드 + 대기 시간 추이 차트
- **데이터 소스**: `order_status = UNASSIGNED` 필터 적용
- **지표**: 미배정 주문 수, 평균 대기 시간, 최장 대기 시간
- **알림**: 대기 시간 임계치 초과 시 관리자 알림

#### 배정 성공률 모니터링
- **위젯 유형**: 게이지 차트 + 일별 추이 라인 차트
- **데이터 소스**: 배정 시도/성공 로그 집계
- **지표**: 당일 배정 성공률, 전일 대비 변화, 주간 평균
- **분석**: 실패 원인 분류 (전문가 거절, 타임아웃, 조건 불일치)

#### 권역별 배정 성과 분석
- **위젯 유형**: 지도 기반 히트맵 + 지역별 테이블
- **데이터 소스**: 지역 그룹별 배정 성과 데이터
- **지표**: 지역별 배정 건수, 성공률, 평균 응답 시간
- **필터**: 기간, 서비스 카테고리, 전문가 등급별 필터

#### 취소/환불 현황 모니터링
- **위젯 유형**: 도넛 차트 + 추이 라인 차트 + 사유 분석 테이블
- **데이터 소스**: 취소/환불 주문 집계
- **지표**: 취소율, 환불율, 전월 대비 변화, 주요 취소 사유
- **알림**: 취소율 급증 시 관리자 알림

#### 채널별 주문 성과 모니터링

- **데이터 소스**: 채널별 주문 데이터
- **지표**: 채널별 주문 수, 매출, 평균 주문 금액, 전환율
- **비교**: 채널 간 성과 비교, 투자 대비 수익률 분석

#### 서비스 카테고리별 성과 분석

- **인사이트**: 인기 서비스, 성장 추세, 수익성 분석

## 전문가 관리

### 개요
전문가 관리 모듈은 플랫폼에 서비스를 제공하는 전문가(마스터 계정 및 서브 계정)의 전 생애주기를 관리하는 기능을 제공합니다. 마스터 계정은 독립적인 사업자로서 팀을 구성할 수 있으며, 서브 계정은 마스터 계정 하위에서 실제 서비스를 수행하는 수행자 계정입니다.

**핵심 개념**:
- **마스터 계정**: 사업자 단위의 전문가 계정으로, 멤버십 구좌 설정, 서브 계정 관리, 팀 정산 통합 관리 기능 보유
- **서브 계정**: 마스터 계정 소속의 개별 수행자 계정으로, 주문 수행만 가능 (독립적 배정 불가)
- **멤버십 구좌**: 특정 서비스 중분류에서 우선 배정을 받을 수 있는 가중치 기반 할당 시스템
- **패널티 시스템**: 전문가의 품질 이슈(거절률, 타임아웃, 고객 클레임 등)에 따른 배정 제한 조치
- **일일 배정 상한**: 전문가의 과부하 방지를 위한 일일 최대 배정 건수 제한

**관련 데이터 모델**: [02_DATABASE_SCHEMA.md](../specs/02_DATABASE_SCHEMA.md)의 `master_accounts`, `sub_accounts`, `master_membership`, `penalty_history`, `assignment_history`, `master_assignment_policy`, `region_groups` 테이블 참조

### 전문가 목록 페이지 (Ant Design ProTable)
**탭 필터**:
- **전체**: 모든 전문가 (마스터 + 서브)
- **마스터 계정**: `account_type = MASTER` 필터
- **서브 계정**: `account_type = SUB` 필터
- **승인대기**: `approval_status = PENDING` 필터
- **활성 전문가**: `active_status = ACTIVE` 필터
- **비활성 전문가**: `active_status = INACTIVE` 필터
- **패널티 적용 중**: `penalty_status = ACTIVE` 필터
- **멤버십 활성**: `membership_enabled = true` 필터

**상태 필터**:
- **계정 상태**: 활성(ACTIVE), 비활성(INACTIVE)
- **승인 상태**: 승인대기(PENDING), 승인완료(APPROVED), 승인거절(REJECTED)
- **패널티 상태**: 정상(NONE), 소프트리밋(SOFT_LIMIT), 하드블록(HARD_BLOCK), 카테고리제한(CATEGORY_LIMIT), 권역제한(REGION_LIMIT), 멤버십정지(MEMBERSHIP_SUSPEND)
- **멤버십 상태**: 활성(ACTIVE), 정지(SUSPENDED), 비활성(INACTIVE), 만료(EXPIRED)

**검색 기능**:
- 전문가명/수행자명
- 연락처
- 이메일
- 로그인 ID
- 사업자명
- 사업자등록번호
- 마스터 계정 ID

**고급 필터**:
- 가입일 범위
- 서비스 가능 카테고리 (중분류)
- 서비스 가능 지역 (시/군/구)
- 지역 그룹 (region_group)
- 멤버십 구좌 수 범위
- 일일 배정 상한 범위

**일괄 작업**:
- 선택 전문가 활성화/비활성화
- 선택 전문가 승인 처리 (대기 → 승인)
- 선택 전문가 패널티 적용
- 선택 전문가 멤버십 활성화/비활성화
- 선택 전문가 내보내기 (CSV)

**액션 버튼**:
- **전문가 등록**: 마스터 계정 또는 서브 계정 등록 모달 오픈

#### 시나리오 1: 전문가 마스터 계정 등록
**UI 컴포넌트**: Ant Design Modal, Form, Steps, Select, Input, Cascader (지역 선택)
**데이터 검증**: 클라이언트 측 Form Validation + 서버 측 중복 검증

**단계별 흐름**:
1. **가입 유형 선택**: 마스터 계정 가입 선택
2. **기본 정보 입력**:
   - 전문가명 (필수, 2~50자)
   - 연락처 (필수, 휴대폰 번호 형식, 중복 불가)
   - 이메일 (필수, 이메일 형식, 중복 불가)
   - 로그인 ID (필수, 4~20자 영문/숫자, 중복 불가)
   - 비밀번호 (필수, 8자 이상 영문/숫자/특수문자 조합)
   - 비밀번호 확인
3. **사업자 정보 입력**:
   - 사업자 유형 선택: 개인사업자 / 법인사업자
   - 사업자명 (필수)
   - 사업자등록번호 (필수, 10자리 숫자, 중복 불가)
   - 법인등록번호 (법인인 경우)
   - 사업장 주소: 주소 검색 API 연동
4. **서비스 수행 가능 정보 설정**:
   - 서비스 카테고리 선택: 메뉴 03 서비스관리 기준 활성 중분류 목록 (다중 선택 가능)
   - 서비스 가능 지역 선택: 시/도 → 시/군/구 Cascader (다중 선택 가능)
5. **시스템 자동 처리**:
   - 선택된 지역코드를 `region_group` 테이블 기준으로 자동 매핑
   - 초기 상태 설정: `approval_status = PENDING`, `active_status = INACTIVE`, `membership_enabled = false`, `membership_slot_count = 0`
   - `master_accounts` 테이블에 레코드 생성, `account_type = MASTER`
6. **배정 로직 연동**:
   - `approval_status = PENDING` 상태이므로 자동배정 후보군 제외
   - 수동배정 불가 상태
7. **전문가웹앱 연동**:
   - 로그인 허용
   - 주문 노출 없음
   - 승인 대기 안내 배너 노출
8. **감사 로그**: 생성 관리자 ID, 생성 시각, 생성 데이터 스냅샷 저장

**비즈니스 규칙**:
- 사업자등록번호는 시스템 내 유일해야 함 (중복 가입 방지)
- 서비스 카테고리와 서비스 가능 지역은 최소 1개 이상 선택해야 함
- 승인 대기 상태에서는 멤버십 설정 불가
- 마스터 계정 생성 후 24시간 내 승인 처리 권장

#### 시나리오 2: 전문가 서브 계정 등록
**UI 컴포넌트**: Ant Design Modal, Form, Select (마스터 계정 검색), Input
**데이터 검증**: 마스터 계정 존재 여부 확인, 중복 로그인 ID 검증

**단계별 흐름**:
1. **가입 유형 선택**: 서브 계정 가입 선택
2. **기본 정보 입력**:
   - 수행자명 (필수)
   - 연락처 (필수, 중복 불가)
   - 이메일 (필수, 중복 불가)
   - 로그인 ID (필수, 중복 불가)
   - 비밀번호 (필수)
   - 비밀번호 확인
3. **소속 마스터 계정 선택**:
   - 사업자명 또는 사업자등록번호로 마스터 계정 검색
   - `account_type = MASTER`이고 `active_status = ACTIVE`인 계정만 노출
   - 마스터 계정 선택
4. **시스템 검증**:
   - 선택한 마스터 계정 존재 여부 확인
   - 마스터 계정 삭제 상태 여부 확인
5. **초기 상태 설정**:
   - `approval_status = APPROVED` (서브 계정은 승인 자동 완료)
   - `active_status = ACTIVE`
   - `account_type = SUB`
   - `membership_enabled` 필드 없음 (마스터 계정 기준)
6. **시스템 저장 처리**:
   - `sub_accounts` 테이블 생성, `master_account_id` FK 저장
   - `assigned_worker_id` 후보군 등록
7. **배정 로직 연동**:
   - 자동배정 후보군에는 영향 없음 (배정 기준은 마스터 계정)
   - 주문 배정 시 `assigned_master_id`는 마스터 계정, 실제 수행자 지정 시 `assigned_worker_id`에 서브 계정 저장 가능
8. **전문가웹앱 연동**:
   - 로그인 허용
   - `assigned_worker_id`가 본인인 주문만 조회 가능
   - 팀 전체 주문 조회 권한 없음
9. **수행 가능 서비스/지역 제한**:
   - 서브 계정은 독립적 서비스 설정 불가
   - 마스터 계정의 `service_category_mid_available_list` 범위 내 수행
   - 마스터 계정의 `region_group` 범위 내 수행
10. **멤버십 구조 영향**:
    - 멤버십은 마스터 계정 기준 (서브 계정 수와 무관)
    - `membership_slot_count`와 무관
11. **마스터 계정 상태 영향**:
    - 마스터 계정이 `INACTIVE` 상태가 되면 해당 서브 계정 자동 배정 제외 (기존 수행 주문은 유지)
12. **감사 로그**: 생성 관리자 ID, 생성 시각, 소속 마스터 계정 ID 기록

**비즈니스 규칙**:
- 서브 계정은 승인 과정 없이 즉시 활성화
- 서브 계정의 서비스 범위는 상위 마스터 계정에 종속
- 마스터 계정 비활성화 시 소속 서브 계정도 배정 제외 (단, 기존 주문은 유지)

#### 시나리오 4: 전문가 정보 수정
**UI 컴포넌트**: 전문가 상세 화면 수정 모달, 폼 필드 그룹화, 실시간 검증
**데이터 검증**: 중복 검증 (사업자등록번호, 연락처, 이메일), 멤버십 정합성 검증

**수정 가능 항목**:
- 기본 정보 (전문가명, 연락처, 이메일)
- 사업자 정보 (사업자명, 사업장 주소, 사업자 유형)
- 서비스 수행 가능 카테고리 (중분류 추가/제거)
- 서비스 가능 지역 (지역 추가/제거, region_group 자동 재매핑)
- 계약 정보 (계약 기간, 이행보증 정보)
- 내부 메모

1. **수정 모달 오픈**: 현재 값으로 폼 프리필
2. **변경 사항 입력**: 각 필드 수정
3. **멤버십 정합성 검증**:
   - 멤버십 적용 중분류가 제거될 경우 경고
   - `membership_slot_count > 0` 상태에서 해당 중분류 제거 시 멤버십 자동 비활성 처리 또는 수정 불가
4. **저장 처리**: `master_accounts` 또는 `sub_accounts` 테이블 업데이트, 변경 전 데이터 스냅샷 저장, 변경 이력 테이블 기록
5. **배정 로직 연동 처리**:
   - 서비스 카테고리 변경 시 신규 주문부터 후보군 필터 재적용
   - 지역 변경 시 region_group 기준 후보군 재계산
   - 멤버십 구좌 수 변경 시 가중치 재반영
6. **기존 주문 보호 원칙**: 이미 배정된 주문에는 영향 없음, 진행 중 주문은 그대로 유지
7. **전문가웹앱 반영**: 프로필 정보 즉시 반영, 수행 가능 서비스 목록 즉시 반영, 신규 배정 범위 즉시 반영
8. **주문관리 연동**: 주문 상세 화면에 표시되는 전문가 정보는 당시 스냅샷 유지, 통계 집계는 최신 master_account_id 기준
9. **감사 로그**: 변경 전 값, 변경 후 값, 변경 관리자 ID, 변경 시각 기록

**비즈니스 규칙**:
- 사업자등록번호 변경 시 중복 검증 필수
- 서비스 카테고리 0개 상태 허용 여부 정책에 따라 검증 (기본값: 허용 안함)
- 지역 0개 상태 허용 여부 정책에 따라 검증 (기본값: 허용 안함)

#### 시나리오 5: 전문가 활성/비활성 전환
**UI 컴포넌트**: 상태 변경 액션 버튼, 사유 입력 팝업, 진행 중 주문 경고 표시
**데이터 검증**: 진행 중 주문 존재 여부 확인, 멤버십 상태 확인

**비활성화 사유 코드**:
- `TEMPORARY_STOP`: 일시 휴업
- `QUALITY_ISSUE`: 품질 이슈/클레임 누적
- `CONTRACT_END`: 계약 종료
- `POLICY_VIOLATION`: 내부 정책 위반
- `INACTIVE_BY_SYSTEM`: 장기 미사용/장기 미배정

**단계별 흐름**:
1. **대상 전문가 검색**: 전문가명, 연락처, 사업자명, 사업자번호 등으로 검색, 마스터/서브 여부 필터, active_status 기준 필터
2. **비활성화 액션**: 비활성화 버튼 클릭 → 사유 입력 팝업 노출 (사유코드 선택, 상세 사유 메모 필수)
3. **시스템 사전 검증**:
   - `approval_status` 확인 (PENDING 상태에서 비활성화는 "가입 반려" 성격)
   - 현재 진행 중 주문 존재 여부 확인 (`ASSIGNED/IN_SERVICE/RESERVED` 상태 주문 건수 조회)
   - `membership_enabled = true`인 경우 "멤버십 혜택도 즉시 중단" 안내
4. **비활성화 확정**: 관리자 확인 후 처리
5. **시스템 상태 변경 처리 – 마스터 계정**:
   - `active_status: ACTIVE → INACTIVE`
   - 자동배정 후보군에서 해당 master_account 제거
   - 수동배정 대상 목록에서도 제외
   - `membership_enabled = true`인 경우 `membership_status = SUSPENDED` 또는 `INACTIVE`로 변경, `membership_slot_count` 값 보관
6. **시스템 상태 변경 처리 – 서브 계정**:
   - 해당 마스터 계정 소속 서브 계정 전체 조회
   - 서브 계정 `active_status`는 유지 (계정 자체는 살아있음)
   - 마스터 계정 `INACTIVE` 상태면 서브 계정만 단독으로 배정 대상이 될 수 없음 (`assigned_worker_id`로만 사용 가능)
7. **진행 중 주문 처리 정책**:
   - 이미 `ASSIGNED/IN_SERVICE` 상태인 주문: 기존 `assigned_master_id`, `assigned_worker_id` 유지
   - 새로운 주문/배정: 마스터 계정 `INACTIVE` 이후 생성되는 주문은 해당 전문가 자동/수동 배정 모두 불가
   - `HOLD` 상태 멤버십 리드: 운영자가 수동으로 재배정/취소/다른 멤버십 전문가로 변경 필요
8. **전문가웹앱 반영**:
   - 마스터 계정: 로그인 허용 (정책 선택 가능), "운영 중단 상태" 배너 노출, 새로운 배정 알림 없음
   - 서브 계정: 로그인 허용, 배정받을 새 주문 없음, 기존 `assigned_worker_id` 기준 진행 주문만 보임
9. **멤버십/배정 통계 연동**:
   - 멤버십 가중치에서 해당 마스터 계정 제외
   - 멤버십 리드 점유율 통계에서 "비활성 기간" 구간 분리 가능하도록 플래그 기록
   - 배정 성과 대시보드에서 "비활성 전문가"로 표시
10. **감사 로그**: before_active_status, after_active_status, reason_code, reason_detail, 변경자 admin_id, 변경 일시 기록
11. **재활성화 시나리오**:
    - `approval_status`가 `APPROVED`인지 재검증
    - 계약 기간이 유효한지 확인
    - region_group, service_category_mid 설정이 남아 있는지 확인

7. **통계 및 리포트 반영**: 과거 주문은 기존 카테고리 기준 유지, 신규 배정 건수 집계는 변경 이후 기준 반영

   - 다른 region_group에서는 정상 적용
6. **전문가웹앱 반영**: 신규 배정 주문 중 해당 지역 주문은 수신 불가, 기존 주문은 정상 표시, 프로필 내 서비스 가능 지역 즉시 반영
7. **엣지 케이스**:

   - 멤버십 활성 상태에서 region_group이 0개가 되면 membership_status 자동 SUSPENDED 처리 권장

**비즈니스 규칙**:

## 주문 관리

### 개요
주문 관리 모듈은 주문의 전 생애주기를 관리하는 기능을 제공합니다. 주문 상태는 3계층 상태 시스템으로 관리됩니다:
- **주문 상태 (order_status)**: NEW, CHECKED, UNASSIGNED, ASSIGNED, CANCELLED, REFUNDED 등
- **배정 전달 상태 (assignment_delivery_status)**: HOLD, SENT - 멤버십 전문가 배정 시 사용
- **전문가 진행 상태 (expert_progress_status)**: 전문가 웹앱에서 업데이트되는 서비스 진행 상태

상태 머신의 주요 흐름:
```
NEW → CHECKED → [UNASSIGNED 또는 ASSIGNED(with HOLD/SENT)] → 전문가 진행 상태 변경 → COMPLETED → SETTLED

취소 경로: 어느 단계에서나 → 취소 요청 → 관리자 검토 → 취소 완료
환불 경로: 완료 후 → 환불 요청 → 관리자 검토 → 환불 완료 (전체/부분)
```

### 신규 주문 검수 및 배정

#### 시나리오 1: 신규 주문 검수 처리
1. 관리자 로그인 후 주문관리 메뉴 진입
2. 주문 목록에서 `order_status = NEW` 필터 적용
3. 신규 주문 리스트 확인 (주문번호, 채널주문번호, 고객정보, 서비스카테고리, 금액정보 등)
4. 특정 주문 클릭하여 상세 화면 진입
5. 주문 상세 정보 검토:
   - 기본 정보, 고객 정보, 서비스 정보, 전문가 정보, 금액 정보
   - 관리자 메모란, 전문가 전달 메모란 확인
   - 서비스 항목, 옵션, 금액 계산 오류, 중복 주문 여부 확인
6. 이상 여부 판단:
   - **이상 없음**: 검수 완료 처리 (`NEW → CHECKED`), 배정 로직 실행
   - **이상 있음**: 보완 조치 (고객 연락, 주문 정보 수정, 취소 처리)
7. 배정 로직 실행:
   - 자동 배정 대상: 자동 배정 트리거
   - 수동 배정 대상: 수동 배정 트리거
   - 멤버십 대상: HOLD 단계 적용 여부 확인

#### 시나리오 2: 주문 상세 조회 및 상태 확인
1. 관리자 로그인 후 주문관리 메뉴 진입
2. 주문 검색 또는 필터 적용:
   - 주문번호, 채널주문번호 검색
   - 고객명, 고객 연락처 검색
   - 주문 상태, 전문가 진행 상태, 배정 상태, 결제 상태 필터
   - 날짜 필터 (주문신청일시, 서비스희망일, 서비스예정일 등)
   - 주소 필터 (시/군, 시군구)
   - 지역그룹(권역) 필터
   - 채널명 필터
3. 주문 목록에서 대상 주문 선택
4. 주문 상세 화면 진입
5. 주문 기본 정보 확인:
   - 기본 정보 (주문번호, 채널주문번호, 채널명, 주문등록일시)
   - 고객 정보 (고객명, 연락처, 안심번호, 서비스 희망일, 고객 요청사항)
   - 서비스 정보 (서비스 예정일, 서비스 카테고리, 수량, 기본비용, 지역그룹, 서비스 주소)
   - 전문가 정보 및 배정 정보
   - 추가 입력 정보 (서비스 관리에서 설정한 추가입력항목)
6. 상태 정보 확인:
   - 주문 상태, 배정 상태, 전달 상태(HOLD/SENT), 전문가 진행 상태, 결제 상태
7. 금액 정보 확인:
   - 예약금(선결제금액), 기본비용, 현장비용, 할인금액, 미결제금액, 총 결제금액
   - 결제 여부, 예약금결제일, 잔금결제일, 구매확정일, 환불 정보
8. 배정 이력 확인:
   - `order_assignment_history` 조회 (배정 전문가, 사업자명, 배정일시)
   - 배정 지역(권역), 배정 시도 횟수, 거절/타임아웃 여부, 재배정 필요 여부
9. 진행 이력 로그 확인:
   - 상태 변경 로그, 운영 메모, 환불/취소 이력
10. 현재 주문 진행 상황 판단:
    - 정상 진행 중: 모니터링 종료
    - 지연 또는 문제 발생: 추가 조치 검토 (전문가 재배정, 고객 연락, 결제 재요청, 환불/취소 처리)

#### 시나리오 3: 수동 전문가 배정
1. 관리자 로그인 후 주문관리 메뉴 진입
2. `order_status = UNASSIGNED` 필터 적용하여 미배정 주문 목록 확인
3. 배정이 필요한 주문 선택 후 상세 화면 진입
4. 배정 가능 여부 확인:
   - 현재 `order_status`가 `UNASSIGNED`인지 확인
   - 기존 `expert_master_id`가 null인지 확인
   - `assignment_delivery_status`가 `NONE`인지 확인
5. "배정" 버튼 클릭
6. 배정 대상 전문가 목록 조회:
   - 서비스 주소(고객)와 서비스 가능지역(전문가) 일치 여부
   - 서비스 카테고리 수행 가능 여부
   - 멤버십 카테고리 여부 및 멤버십/일반 전문가 구분
   - 활성 상태 여부
   - 최근 배정 이력, 거절률, 체결률 참고
7. 특정 `expert_master_id` 선택
8. 배정 확정 처리:
   - `order_status`: `UNASSIGNED` → `ASSIGNED`
   - `expert_master_id` 저장
   - `assignment_delivery_status`: `NONE` → `SENT`
   - `expert_progress_status`: `NEW_ASSIGN`으로 초기화
9. 배정 이력 기록:
   - `order_assignment_history`에 row 생성, `assignment_type = MANUAL`
   - 배정 시각, 전문가 정보, 배정 횟수 기록
10. 전문가에게 자동 알림 전송 (앱 푸시, SMS, 알림톡 등)
11. 배정 결과 확인:
    - 주문 상세 화면에서 `ASSIGNED` 상태 확인
    - `assignment_delivery_status = SENT` 확인
    - 배정 성공 시 모니터링 종료, 실패 시 실패 사유 확인 및 재시도

#### 시나리오 4: 자동배정 후 멤버십 전문가 배정 주문 HOLD 처리
1. 자동 배정 로직 실행 시 멤버십 전문가가 배정된 경우:
   - `order_status`: `CHECKED` → `ASSIGNED`
   - `expert_master_id` 세팅
   - `assignment_delivery_status`: `NONE` → `HOLD`
   - `expert_progress_status`는 아직 설정하지 않음
   - 주문이 "멤버십 상담 대기 주문"으로 분류
2. 운영팀 1차 상담 진행:
   - 고객 요구사항 상세 확인
   - 현장 추가비용 발생 가능성 확인
   - 방문 일정 재확인
   - 서비스 범위 확정
   - 상담 내용 메모 기록
3. 상담 결과에 따른 분기 처리:
   - **기존 자동배정된 멤버십 전문가에게 전달**:
     - `assignment_delivery_status`: `HOLD` → `SENT`
     - `expert_progress_status`: `NEW_ASSIGN` 설정
     - 전문가에게 자동 알림 발송
     - HOLD 종료 시각 기록
     - 전문가웹/앱에 주문 노출
   - **기존 자동배정 전문가에게 전달하지 않음**:
     - 배정 취소 처리: `order_status`: `ASSIGNED` → `UNASSIGNED`
     - `expert_master_id` 초기화
     - `assignment_delivery_status`: `HOLD` → `NONE`
     - `order_assignment_history`에 `REJECTED_BY_OPERATION` 기록
     - 재배정 로직 실행 (새 전문가 자동/수동 배정)
   - **상담 후 주문 취소**:
     - `order_status`: `ASSIGNED` → `CANCELED`
     - `assignment_delivery_status` HOLD 상태 종료
     - 취소 사유 기록
     - 필요 시 환불 처리

**운영 리스크 통제 포인트**:
- HOLD 상태 주문은 전문가웹/앱에 절대 노출되지 않아야 함
- HOLD 상태에서는 자동 재배정이 실행되지 않아야 함
- HOLD 상태에서는 `expert_progress_status`가 설정되지 않아야 함
- HOLD 장기 미처리 주문은 대시보드에서 별도 카운트되어야 함
- HOLD → SENT 전환 시 기존 자동 배정된 `expert_master_id`를 그대로 유지해야 함
- HOLD 중 전문가 변경은 반드시 `ASSIGNED` → `UNASSIGNED` 전환 후 재배정으로만 처리해야 함

#### 시나리오 5: HOLD 해제 후 전문가 전달
1. 멤버십 HOLD 상태 주문 조회:
   - 필터 적용: `order_status = ASSIGNED`, `assignment_delivery_status = HOLD`, `expert_master_id` not null
   - HOLD 상태 주문 목록 확인 (주문번호, 고객명, 서비스 카테고리, 요청 일정, 자동배정된 전문가, HOLD 시작 시각)
2. 특정 주문 선택 후 상세 화면 진입
3. 사전 조건 검증:
   - 상태 확인: `order_status = ASSIGNED`, `assignment_delivery_status = HOLD`, `expert_master_id` 존재
   - 해당 전문가의 `membership_flag = true` 확인
   - 운영 상담 완료 여부 확인 (상담 메모 존재, 금액/일정 수정 여부, 고객 진행 의사 확정)
4. 전달 여부 결정:
   - **기존 자동배정된 멤버십 전문가에게 전달**:
     - HOLD 해제 실행: "전달" 버튼 클릭
     - 시스템 처리:
       - `assignment_delivery_status`: `HOLD` → `SENT`
       - `expert_progress_status`: `null` → `NEW_ASSIGN`
       - HOLD 종료 시각 기록
       - `order_assignment_history`에 `type = HOLD_RELEASE` 기록
       - 전달 실행자 `admin_id` 기록
     - 전문가 알림 발송 (앱 푸시, SMS, 카카오 알림톡)
     - 전문가웹/앱 노출 (해당 주문이 전문가 신규 탭에 표시)
     - 전달 완료 상태 확인: `assignment_delivery_status = SENT`, `expert_progress_status = NEW_ASSIGN`
   - **기존 자동배정 전문가에게 전달하지 않음**:
     - 배정 취소 및 재배정 준비:
       - 배정 취소 실행: `order_status`: `ASSIGNED` → `UNASSIGNED`
       - `expert_master_id` 초기화
       - `assignment_delivery_status`: `HOLD` → `NONE`
       - `expert_progress_status` 초기화(null)
       - `order_assignment_history`에 `type = REJECTED_BY_OPERATION` 기록
     - 재배정 로직 실행
     - 신규 자동배정 또는 수동 배정 수행
     - 새 전문가에게 `SENT` 처리
   - **상담 후 주문 취소 결정**:
     - 주문 취소 실행: `order_status`: `ASSIGNED` → `CANCELED`
     - `assignment_delivery_status` HOLD 종료 처리
     - 취소 사유 코드 저장
     - `order_assignment_history`에 `type = CANCELED` 기록
     - 필요 시 환불 처리

**상태 정합성 검증 포인트**:
- HOLD 상태에서는 `expert_progress_status`가 설정되면 안 됨
- HOLD 해제 시 반드시 `expert_master_id`가 존재해야 함
- HOLD 해제 후 `SENT` 상태에서만 전문가웹/앱 노출이 가능함
- HOLD 중에는 자동 재배정 로직이 동작하지 않아야 함
- HOLD 해제 로그와 최초 자동배정 로그는 `order_assignment_history`에서 구분 가능해야 함
- HOLD 상태에서 장기 미처리 주문은 대시보드에서 별도 모니터링 대상이어야 함

    - 생성된 중분류 선택  
    - “소분류 생성” 클릭  
    - 소분류명 입력

#### 대분류 관리 (Level 1)
- **생성**: "대분류 생성" 버튼 → 이름, 설명, 활성화 여부 입력 → 2자리 숫자 코드 자동 생성
- **수정/삭제**: 트리 노드 우클릭 또는 편집 버튼
- **비즈니스 규칙**: 
  - 코드 자동 생성 (예: "04") 후 수정 불가
  - `is_active=false` 시 하위 중분류/소분류 모두 비활성화 상태로 표시
  - 변경 이력 로그 저장 (`audit_logs`)

#### 중분류 관리 (Level 2)
- **생성**: 대분류 선택 → "중분류 생성" → 이름, 설명, 활성화 여부, 멤버십 적용 가능 여부 설정
- **멤버십 적용 가능 여부 (`membership_available`)**:
  - `true` 설정 시 멤버십 배정 로직에서 우선 배정 대상
  - 전문가관리 메뉴에서 멤버십 가입 가능 대상으로 노출
  - 변경 시 영향 범위 안내 팝업 (활동 전문가 수, 예상 수요)
- **비즈니스 규칙**:
  - 코드 자동 생성 (상위 대분류 코드 + 2자리 숫자, 예: "0401")
  - `membership_available` 기본값 `false`

#### 소분류 관리 (Level 3)
- **생성**: 중분류 선택 → "소분류 생성" → 이름, 설명, 활성화 여부 설정
- **비즈니스 규칙**:
  - 코드 자동 생성 (상위 중분류 코드 + 2자리 숫자, 예: "040101")

    - 해당 소분류 주문 생성  
    - 주문 상세에서 “현장비용 추가” 클릭  
    - 매핑된 현장비용 항목만 노출되는지 확인  
20. 전문가웹앱 연동 확인  
    - 서비스 진행 중 현장비용 추가 화면 진입  
    - 매핑된 항목만 드롭다운에 표시되는지 확인  
    - 수량 기반 항목은 수량 입력 필드 표시 확인  
21. 금액 계산 연동 확인  
    - base_price + 선택된 현장비용 합계 = total_price 반영 확인  
    - remaining_payable_amount 재계산 확인  
22. 정산관리 연동 확인  
    - 정산 반영 여부 true인 현장비용만 정산 계산식에 포함 확인  
    - 채널 수수료 계산 시 포함 여부 검증  
23. 엑셀 업로드 연동 확인  
    - 해당 소분류 템플릿 다운로드  
    - 매핑된 현장비용 컬럼 생성 여부 확인  
    - 업로드 시 해당 항목만 허용되는지 확인  
24. 매핑 변경 테스트  
    - 특정 현장비용 매핑 해제  
    - 신규 주문에서는 노출되지 않음 확인  
    - 기존 주문 데이터는 유지됨 확인  
25. 비활성화 테스트  
    - 현장비용 카테고리 is_active=false 설정  
    - 신규 주문 및 전문가웹앱에서 선택 불가 확인  
    - 기존 주문에는 영향 없음 확인  
26. 예외 처리  
    - 매핑 없이 현장비용 입력 시 시스템 차단  
    - 정산 반영 false 항목은 수익 계산에서 제외  
    - 중복 매핑 저장 불가  
27. 최종 상태  
    - 현장비용 카테고리 마스터 생성 완료  
    - 서비스 카테고리 기준 매핑 완료  
    - 주문·전문가웹앱·정산과 정합성 유지  
    - 매핑 레벨 우선순위 규칙 확정  

##### 시나리오 5: 서비스–현장비용 매핑 수정 및 우선순위 변경 운영

1. 관리자 로그인  
2. 백오피스 → 메뉴 03 서비스관리 진입  
3. “서비스–현장비용 매핑 관리” 탭 선택  
4. 매핑 기준 선택  
   - 중분류 기준 매핑 조회  
   - 소분류 기준 매핑 조회  
5. 특정 서비스 카테고리 선택  
6. 현재 매핑 상태 확인  
   - 매핑된 현장비용 목록  
   - 매핑 레벨(LEVEL2 또는 LEVEL3)  
   - 필수 여부  
   - 최대 수량 제한  
   - 적용 시작일  
7. 수정 필요 사유 확인  
   - 특정 현장비용 항목 제거 필요  
   - 신규 현장비용 추가 필요  
   - 필수 여부 변경 필요  
   - 중분류 매핑을 소분류 예외 매핑으로 전환 필요  
8. 매핑 수정 액션 실행  
9. 현장비용 항목 추가  
   - 추가할 현장비용 선택  
   - 필수 여부 설정  
   - 수량 제한 설정  
   - 적용 시작일 설정  
10. 현장비용 항목 제거  
    - 기존 매핑 항목 선택  
    - 제거 클릭  
    - 제거 사유 입력  
11. 우선순위 변경 처리  
    - 중분류 매핑이 존재하는 상태에서  
      · 특정 소분류에 별도 매핑 추가 시  
      · 해당 소분류는 중분류 매핑을 덮어씀  
12. 시스템 우선순위 규칙 적용  
    - 소분류 매핑 존재 시 → 해당 매핑 우선 적용  
    - 소분류 매핑 미존재 시 → 중분류 매핑 적용  
13. 저장 버튼 클릭  
14. 시스템 사전 검증  
    - 중복 매핑 여부 확인  
    - 필수 항목 제거 시 정책 위반 여부 확인  
    - 적용 시작일이 과거 주문에 영향 주지 않는지 확인  
15. 저장 처리  
    - 기존 매핑 row 종료일 자동 설정  
    - 신규 매핑 row 생성  
    - mapping_version 증가  
    - 변경 이력 로그 저장  
16. 기존 주문 영향 검증  
    - 이미 생성된 주문에는 기존 매핑 스냅샷 유지  
    - 신규 주문부터 변경된 매핑 적용  
17. 주문관리 연동 확인  
    - 해당 소분류 신규 주문 생성  
    - 현장비용 추가 화면에서 변경된 매핑 반영 확인  
18. 전문가웹앱 연동 확인  
    - 서비스 진행 중 현장비용 입력 화면 진입  
    - 매핑 변경 내용 즉시 반영 확인  
19. 정산 연동 확인  
    - 정산 반영 여부가 변경된 경우  
    - 신규 주문 정산 계산식 반영 확인  
20. 엑셀 템플릿 연동 확인  
    - 해당 소분류 엑셀 템플릿 다운로드  
    - 매핑된 현장비용 컬럼 업데이트 여부 확인  
21. 중분류 매핑 → 소분류 예외 전환 시나리오  
    - 기존 중분류 매핑 존재  
    - 특정 소분류만 예외 처리 필요  
    - 해당 소분류에 별도 매핑 생성  
    - 시스템은 자동으로 해당 소분류에 중분류 매핑 적용 제외  
22. 소분류 매핑 제거 시 처리  
    - 소분류 매핑 삭제  
    - 자동으로 중분류 매핑 적용 복원  
23. 비활성화 테스트  
    - 현장비용 카테고리 is_active=false 처리  
    - 매핑은 유지하되 신규 주문에서는 선택 불가 확인  
24. 감사 로그 확인  
    - 변경 전 매핑 목록  
    - 변경 후 매핑 목록  
    - 변경 사유  
    - 변경 관리자  
    - 변경 시각  
25. 예외 처리  
    - 매핑 수정 중 주문 동시 생성 충돌 방지 위해 트랜잭션 처리  
    - 매핑 없는 상태에서 필수 입력 요구하지 않도록 검증  
    - 정산 확정 주문에는 영향 없음  
26. 최종 상태  
    - 서비스–현장비용 매핑 정책 최신화  
    - 우선순위 규칙 명확화  
    - 신규 주문에만 적용  
    - 기존 주문 데이터 보존  

#### 매핑 생성/수정
- **매핑 기준**: 중분류 기준 매핑 (LEVEL2) 또는 소분류 기준 매핑 (LEVEL3)
- **매핑 항목**: 현장비용 카테고리 선택, 필수 여부(`is_required`), 최대 수량 제한(`max_quantity`)
- **우선순위 규칙**:
  - 소분류 매핑 > 중분류 매핑 (동일 중분류 하위 소분류에 별도 매핑 존재 시 소분류 매핑 우선 적용)
  - 소분류 매핑 없을 경우 중분류 매핑 적용
- **버전 관리**:
  - 신규 매핑 생성 시 기존 매핑 `effective_end_date` 자동 설정
  - `mapping_version` 증가
  - 변경 이력 로그 저장

#### 매핑 변경 운영
- **항목 추가/제거**: 특정 현장비용 항목 제거, 신규 항목 추가, 필수 여부 변경
- **중분류 → 소분류 예외 전환**: 기존 중분류 매핑 존재 시 특정 소분류에 별도 매핑 생성으로 예외 처리
- **연동 확인**:
  - 주문관리: 신규 주문에서 매핑된 현장비용 항목만 노출
  - 전문가웹앱: 매핑 변경 내용 즉시 반영
  - 엑셀 템플릿: 매핑된 현장비용 컬럼 업데이트 여부 확인

### 5. 채널별 수수료 관리 - 시나리오 6
**UI 구성**: 소분류 상세 화면 → "채널별 수수료 관리" 탭

#### 상세 작업 흐름

1. 관리자 로그인  
2. 백오피스 → 메뉴 03 서비스관리 진입  
3. 대·중·소분류 트리에서 수수료를 설정할 소분류 선택  
4. 소분류 상세 화면 진입  
5. “채널별 수수료 관리” 탭 클릭  
6. 현재 수수료 설정 현황 확인  
   - channel_code  
   - channel_name  
   - commission_type  
   - commission_value  
   - 적용 시작일(effective_start_date)  
   - 적용 종료일(effective_end_date)  
7. 신규 채널 수수료 설정 클릭  
8. 수수료 정보 입력  
   - channel_code 선택  
   - commission_type 선택  
     · RATE: 퍼센트 수수료  
     · FIXED: 고정 수수료  
   - commission_value 입력  
   - 적용 시작일 설정  
   - 정산 기준 옵션 선택  
     · VAT 포함 기준  
     · VAT 제외 기준  
9. 저장 버튼 클릭  
10. 시스템 사전 검증  
    - 동일 소분류 + 동일 channel_code + 동일 시작일 중복 여부 확인  
    - commission_value ≥ 0 검증  
    - RATE일 경우 0 ≤ value ≤ 100 검증  
    - 적용 시작일이 과거 확정 정산 주문에 영향을 주지 않는지 검증  
11. 저장 처리  
    - service_category_channel_commission 테이블에 row 생성  
    - 기존 동일 채널 수수료가 존재할 경우 종료일 자동 설정  
    - commission_version 증가  
    - 변경 이력 로그 저장  
12. 주문관리 연동 확인  
    - 해당 소분류 + 해당 채널로 신규 주문 생성  
    - 주문 생성 시 channel_code 스냅샷 저장 확인  
    - 정산 예정 금액 계산 시 수수료 반영 여부 확인  
13. 정산관리 연동 확인  
    - 정산관리 메뉴에서 해당 주문 조회  
    - 전문가 정산 금액 계산식 확인  
      · total_price 기준 계산  
      · 현장비용 포함 여부 확인  
      · 수수료 적용 방식 확인  
14. 금액 수정 연동 확인  
    - 주문 금액 수정 시  
    - 수수료 계산이 최신 price_snapshot 기준으로 재계산되는지 확인  
15. 환불 연동 확인  
    - 부분 환불 발생 시  
    - 환불 금액에 대한 수수료 재계산 정책 확인  
    - 수수료 반환 여부 정책 적용 확인  
16. 엑셀 업로드 연동 확인  
    - 엑셀 업로드 시 channel_code 입력  
    - 업로드된 주문에 해당 수수료 자동 적용 확인  
17. 채널별 예외 정책 설정  
    - 특정 채널은 수수료 0원 설정 가능  
    - 특정 채널은 FIXED + RATE 혼합 구조 불가  
18. 수수료 수정 시나리오  
    - 기존 채널 수수료 값 변경  
    - 적용 시작일을 미래로 설정  
    - 기존 주문 영향 없음 확인  
19. 수수료 비활성화 처리  
    - 특정 채널 수수료 row 종료일 설정  
    - 이후 신규 주문에 적용되지 않음 확인  
20. 감사 로그 확인  
    - 변경 전 수수료 값  
    - 변경 후 수수료 값  
    - 변경 사유  
    - 변경 관리자  
    - 변경 시각  
21. 예외 처리  
    - 수수료 미설정 채널 주문 생성 시 경고 표시  
    - channel_code 없는 주문 생성 불가 처리  
    - 정산 확정 주문에는 수수료 변경 소급 적용 불가  
22. 최종 상태  
    - 소분류 기준 채널별 수수료 정책 확정  
    - 신규 주문에만 적용  
    - 기존 주문 스냅샷 유지  
    - 정산관리와 정합성 유지  

#### 수수료 설정
- **입력 항목**: `channel_code`, `commission_type` (RATE/FIXED), `commission_value`, `vat_basis`, `effective_start_date`
- **시스템 검증**:
  - 동일 소분류 + 동일 `channel_code` + 동일 `effective_start_date` 중복 불가
  - `commission_value ≥ 0`, RATE일 경우 0 ≤ value ≤ 100
  - 적용 시작일이 과거 확정 정산 주문에 영향 없는지 검증
- **버전 관리**:
  - 신규 수수료 생성 시 기존 수수료 `effective_end_date` 자동 설정
  - `commission_version` 증가
  - 변경 이력 로그 저장

#### 연동 확인
- **주문관리**: 주문 생성 시 `channel_code` 스냅샷 저장, 정산 예정 금액 계산 시 수수료 반영
- **정산관리**: 전문가 정산 금액 계산식에 수수료 적용 방식 확인
- **금액 수정**: 주문 금액 수정 시 수수료 계산이 최신 `price_snapshot` 기준 재계산
- **환불 연동**: 부분 환불 발생 시 환불 금액에 대한 수수료 재계산 정책 적용

### 6. 추가입력항목 관리 - 시나리오 7
**UI 구성**: 서비스 카테고리 상세 화면 → "추가입력항목" 탭
#### 상세 작업 흐름

1. 관리자 로그인
2. 백오피스 → 메뉴 03 서비스관리 진입
3. 추가입력항목을 설정할 서비스 카테고리 선택
  - 대·중·소분류 트리 확인
  - 기준 정책
    · 중분류 기준으로 공통 추가입력항목을 관리할지
    · 특정 소분류에만 예외 항목을 둘지 결정
  - 운영자는
    · “중분류 기준으로 공통 필드 설정 → 일부 소분류에서 예외 필드 추가/제거” 패턴을 기본 전략으로 사용
4. 추가입력항목 탭 진입
  - 선택한 카테고리(중분류 또는 소분류)의 상세 화면으로 이동
  - 상단 탭에서 “추가입력항목” 탭 클릭
  - 현재 정의된 항목 목록 확인
    · field_key
    · 표시명(label)
    · 필드 타입(field_type: TEXT/NUMBER/DATE/SELECT 등)
    · 필수 여부(required)
    · 노출 위치(주문관리/전문가웹앱/양쪽)
    · 사용 여부(is_active)
    · 정렬 순서(sort_order)
5. 신규 추가입력항목 생성
  - “필드 추가” 버튼 클릭
  - 필수 정보 입력
    · field_key 입력
      - 영문/숫자 조합, 고유값
      - 한번 생성되면 수정 불가
    · 표시명(label) 입력
      - 운영/CS/전문가 모두 이해 가능한 용어 사용
    · 필드 타입 선택
      - TEXT: 자유 입력
      - NUMBER: 숫자 입력
      - DATE: 날짜 선택
      - SELECT: 사전 정의 옵션 중 선택
    · 필수 여부 설정(required_flag)
      - 주문 생성/수정 시 반드시 입력되어야 하는지
    · 노출 위치 설정
      - ORDER_BACKOFFICE_ONLY
      - EXPERT_APP_ONLY
      - BOTH
    · 정렬 순서(sort_order) 지정
    · SELECT 타입일 경우 옵션 목록 입력
      - option_key / option_label 리스트
  - 저장 클릭

6. 시스템 사전 검증
  - field_key 중복 여부 확인
  - 필드 타입에 따른 필수 속성 검증
    · NUMBER 타입: 최소/최대값 설정 시 min ≤ max
    · SELECT 타입: 최소 1개 이상의 옵션 필수
  - 동일 카테고리 내 동일 label을 허용할지 정책에 따라 경고 또는 허용
  - 저장 가능하면 DB에 신규 필드 정의 생성

7. 저장 처리 및 다른 메뉴 연동 준비
  - service_category_extra_field 테이블에 row 생성
    · category_id (중분류/소분류 ID)
    · field_key, field_type, label, options, required_flag, is_active, sort_order
  - 엑셀 템플릿 정의에 해당 필드 포함 플래그 설정
  - 전문가웹/앱 주문 상세 화면 스키마에 즉시 반영

8. 추가입력항목 변경(수정)
  - 기존 필드 행 선택 후 “수정” 클릭
  - 수정 가능 항목
    · label (표시명)
    · 설명/헬프 텍스트
    · required_flag (필수 여부) – 단, 이미 대량 사용 중이면 경고
    · 노출 위치
    · sort_order
    · SELECT 옵션 label (option_key는 가능하면 불변)
  - 수정 불가 항목
    · field_key
    · field_type (TEXT → NUMBER 등 타입 변경은 별도 신규 필드 생성 권장)
  - 저장 클릭

9. 필드 수정의 영향도 안내
  - required_flag를 true로 변경하는 경우
    · 기존 주문 중 해당 값이 비어 있는 주문 수를 조회하여 안내
  - 노출 위치를 EXPERT_APP_ONLY → BOTH로 변경 시
    · 주문 생성 화면에서도 입력 필드가 생김을 안내
  - SELECT 옵션을 제거/변경하는 경우
    · 기존 주문에 저장된 option_key는 그대로 유지되며,
  화면에서는 “(이전 옵션)” 등으로 표시 필요

10. 추가입력항목 비활성화
  - 더 이상 사용하지 않을 필드의 “사용 여부”를 OFF로 변경
  - 시스템 처리
    · 신규 주문 및 수정 화면에서 입력 불가
    · 전문가웹/앱에서도 입력/수정 불가
    · 기존 주문 상세에서는 “읽기 전용”으로 값만 표시
  - 필드 삭제는 원칙적으로 허용하지 않음
    · 데이터 손실 방지
    · 필요 시 “완전 숨김” 플래그 추가 고려(슈퍼관리자 전용)
11. 중분류 vs 소분류 기준 상속/우선순위
  - 중분류에 정의된 추가입력항목은
    · 기본적으로 해당 중분류 하위 모든 소분류에 공통으로 적용
  - 소분류에 별도 추가입력항목을 정의한 경우
    · 해당 소분류에는 중분류 공통 필드 + 소분류 전용 필드가 모두 노출
  - 소분류에서 “중분류 공통 필드를 비활성화”하는 예외 정책이 필요하다면
    · 소분류 수준에서 override_flag를 설정해
  특정 중분류 필드를 숨김 처리할 수 있도록 확장 가능
  - 우선순위 규칙
    · “보이는 필드 = (중분류 공통 필드 – 소분류 숨김 지정) + 소분류 전용 필드”
12. 메뉴 02 주문관리와의 연동 확인
  - 해당 카테고리 소분류로 신규 주문 생성
  - 주문 생성 화면에서 추가입력항목이 노출되는지 확인
  - 필수 필드 누락 시 저장 불가 여부 확인
  - 주문 상세 화면에서도 동일 필드들이 조회되는지 확인
  - 주문 수정 시에도 동일 검증이 작동하는지 확인


13. 전문가웹/앱과의 연동 확인
  - 전문가에게 배정된 동일 카테고리 주문 확인
  - 전문가웹/앱 주문 상세 화면에서 추가입력항목 노출 확인
  - 노출 위치가 EXPERT_APP_ONLY 필드만 별도로 표시되는지 확인
  - 전문가가 현장에서 값을 수정/추가 가능한 필드와
  백오피스에서만 수정 가능한 필드의 구분이 올바른지 검증


14. 엑셀 템플릿 연동 확인
  - 메뉴 02 → 엑셀 템플릿 다운로드(해당 카테고리 기준)
  - 추가입력항목 필드가 컬럼으로 포함되는지 확인
  - 필드 비활성화 시 템플릿에서 컬럼 유지 여부 정책 결정
    · 컬럼은 남기되 “더 이상 사용하지 않음” 안내 주석 추가
    · 신규 템플릿 버전부터는 컬럼 자체 제거 등 정책 선택


15. 데이터 저장 구조 확인
  - 주문 생성 시
    · order_extra_field_value 테이블에
  · order_id
  · field_key
  · field_value
  형태로 저장
  - 중분류/소분류 필드 모두 동일한 구조로 저장
  - 향후 분석/리포트 시 field_key 기준으로 통계 가능


16. 변경 이력 및 감사 로그 확인
  - 추가입력항목 생성/수정/비활성화 이력 조회
  - 로그 항목
    · category_id
    · field_key
    · 변경 전/후 값
    · 변경 관리자
    · 변경 시각
  - 필수 여부(required) 변경은 별도 강조 표시


17. 예외 케이스 처리
  - 현장비용과의 혼동 방지
    · 추가입력항목은 “정보 필드”이고
  현장비용은 “금액 필드”임을 명확히 구분
  - 필드 타입 변경 필요 시
    · 기존 필드를 비활성화하고
    · 새로운 field_key로 재생성하는 것을 원칙으로 한다
  - 사람이 직접 문자열로 적는 대신
  SELECT 옵션으로 최대한 구조화하여
  나중에 통계/정책에 활용 가능하도록 유도


18. 최종 상태
  - 서비스 카테고리별 추가입력항목 정의가 완료된다
  - 주문관리, 전문가웹/앱, 엑셀 템플릿이 모두 이 정의를 단일 진실원으로 참조한다
  - 필드 변경/비활성화가 과거 주문 데이터에 피해를 주지 않도록 설계된다


#### 필드 생성/수정
- **입력 항목**: `field_key` (영문/숫자 조합, 고유, 수정 불가), `label`, `field_type` (TEXT/NUMBER/DATE/SELECT 등), `options` (SELECT 타입), `is_required`, `display_location` (ORDER_BACKOFFICE_ONLY/EXPERT_APP_ONLY/BOTH), `sort_order`
- **상속/우선순위 규칙**:
  - 중분류 필드는 하위 소분류에 공통 적용
  - 소분류 전용 필드는 중분류 필드와 병합 노출
  - 소분류에서 중분류 공통 필드 비활성화 가능 (`override_flag`)
- **비활성화 처리**: `is_active=false` 설정 시 신규 주문 및 수정 화면에서 입력 불가, 기존 주문은 읽기 전용 표시

#### 연동 확인
- **주문관리**: 주문 생성/수정 화면에서 추가입력항목 노출, 필수 필드 검증
- **전문가웹앱**: `display_location`에 따른 필드 노출 구분 검증
- **엑셀 템플릿**: 추가입력항목 필드 컬럼 포함 여부 확인
- **데이터 저장**: `order_extra_field_value` 테이블에 `field_key`/`field_value` 형태 저장

### 7. 이미지 관리 (AWS S3 통합)
- **서비스 이미지 업로드**: Ant Design Upload 컴포넌트로 S3 직접 업로드
- **이미지 리사이징**: Lambda@Edge로 다양한 크기 자동 생성 (썸네일, 상세)
- **CDN 캐싱**: CloudFront로 이미지 빠른 전송

### 8. 서비스 검색 및 필터링
- **실시간 검색**: 대분류/중분류/소분류 이름으로 검색
- **활성화 필터**: 활성/비활성 서비스 필터링
- **가격 범위 필터**: 최소/최대 가격으로 서비스 필터링
- **멤버십 가능 필터**: `membership_available=true` 인 중분류만 표시

### 배정 관리

#### 시나리오 6: 전문가 거절 또는 타임아웃 후 재배정 처리
1. **전제 조건 검증**:
   - `order_status = ASSIGNED`
   - `assignment_delivery_status = SENT`
   - `expert_progress_status = NEW_ASSIGN`
   - `expert_master_id` 존재
   - 이 상태에서만 거절 또는 타임아웃 이벤트가 발생할 수 있음

2. **전문가 거절 또는 타임아웃 이벤트 발생**:
   - 전문가웹/앱에서 거절 버튼 클릭
   - 또는 설정된 응답 타이머 초과

3. **시스템 이벤트 수신**:
   - 전문가 서비스에서 거절/타임아웃 이벤트를 백오피스 주문 서비스로 전달
   - 이벤트 타입 구분: `REJECTED` 또는 `TIMEOUT`

4. **상태 유효성 검증**:
   - 현재 `order_status`가 `ASSIGNED`인지 확인
   - `assignment_delivery_status`가 `SENT`인지 확인
   - `expert_master_id`가 이벤트 발신자와 일치하는지 확인
   - 이미 `CANCELED` 또는 `REFUNDED` 상태가 아닌지 확인
   - 조건 불일치 시 처리 중단 및 로그 기록

5. **재배정 상태 전환 처리**:
   - `order_status`: `ASSIGNED` → `UNASSIGNED`
   - `expert_master_id` 초기화
   - `expert_worker_id` 초기화
   - `assignment_delivery_status`: `SENT` → `NONE`
   - `expert_progress_status` 초기화(null)
   - 응답 타이머 종료 처리

6. **배정 이력 기록**:
   - `order_assignment_history`에 새로운 row 생성
     - `type = REJECTED` 또는 `TIMEOUT`
     - `expert_master_id`(이전 값) 기록
     - 발생 시각 기록
     - 재배정 시퀀스 번호 증가
   - 주문 이력 로그에 상태 전이 기록

7. **자동 재배정 로직 실행**:
   - 배정 로직 입력 조건:
     - `region_group`
     - 서비스 카테고리
     - 활성 전문가
     - 기존 거절한 전문가 제외
     - 멤버십 우선 정책 적용
   - 다음 전문가 후보 선정:
     - 멤버십 전문가가 존재하는 경우:
       - 자동배정 후 HOLD 처리 여부 판단
       - `assignment_delivery_status = HOLD` 또는 `SENT` 전환
     - 일반 전문가가 선정된 경우:
       - `order_status`: `UNASSIGNED` → `ASSIGNED`
       - `expert_master_id` 세팅
       - `assignment_delivery_status`: `NONE` → `SENT`
       - `expert_progress_status = NEW_ASSIGN`

8. **재배정 결과 분기**:
   - 재배정 성공 시:
     - 전문가 알림 발송 (앱 푸시, SMS, 카카오 알림톡)
     - 주문 상태 확인: `ASSIGNED`, `assignment_delivery_status = SENT` 또는 `HOLD`
   - 재배정 대상 없음 시:
     - `order_status` 유지 = `UNASSIGNED`
     - 미배정 주문으로 대시보드 노출
     - 운영팀 알림 생성
     - 수동 배정 필요 상태로 전환

**특수 케이스 처리**:
1. **멤버십 전문가가 거절한 경우**:
   - 해당 주문은 일반 전문가 대상으로 재배정될 수 있음
   - 멤버십 우선 정책은 "다음 멤버십 전문가" 존재 시에만 적용

2. **연속 거절 발생**:
   - 동일 주문에 대해 거절 횟수 누적
   - 일정 횟수 초과 시 자동 재배정 중단
   - 운영 개입 필요 플래그 설정

3. **타임아웃 반복 발생**:
   - 해당 전문가 응답 지연 통계에 반영
   - 전문가 패널티 점수 증가 가능

**운영 리스크 통제 포인트**:
- 거절 이벤트가 중복 처리되지 않도록 idempotent 처리 필요
- 상태 전환은 트랜잭션 단위로 처리해야 함
- 재배정 중 주문 취소가 발생할 경우 우선순위는 취소가 우선
- HOLD 상태 주문은 거절 이벤트가 발생할 수 없음
- 재배정 로직 실행 중 강제 수동 배정이 들어오면 자동 배정 중단

#### 시나리오 7: 주문 취소 처리
1. **관리자 로그인 및 주문관리 메뉴 진입**:
   - 관리자 계정으로 로그인
   - 백오피스 주문관리 메뉴 진입

2. **취소 대상 주문 검색**:
   - 검색 조건:
     - 주문번호
     - 고객 전화번호
     - `order_status` IN (`NEW`, `CHECKED`, `UNASSIGNED`, `ASSIGNED`)
     - 필요 시 채널, 서비스 카테고리 필터 적용

3. **주문 상세 화면 진입 및 현재 상태 확인**:
   - `order_status` 확인
   - `expert_progress_status` 확인
   - `assignment_delivery_status` 확인
   - `expert_master_id` 존재 여부 확인
   - `total_price`, `paid_amount_total`, `refund_amount_total` 확인

4. **취소 가능 여부 판단**:
   - 취소 불가 조건:
     - `order_status` IN (`CANCELED`, `REFUNDED`)
     - 이미 정산 확정 완료 상태(정산 잠금 상태)
     - 시스템 정책상 취소 제한 시간 경과
   - 상태별 판단:
     - `NEW` / `CHECKED` / `UNASSIGNED` 상태: 즉시 취소 가능
     - `ASSIGNED` + `assignment_delivery_status = HOLD`: 전문가에게 아직 전달되지 않음 → 취소 가능
     - `ASSIGNED` + `assignment_delivery_status = SENT`: 전문가에게 이미 전달됨 → 전문가 알림 필요
     - `expert_progress_status = IN_SERVICE` 이상: 부분 서비스 제공 가능성 → 환불 정책 별도 검토 필요

5. **취소 사유 입력**:
   - 취소 버튼 클릭
   - 취소 사유 코드 선택:
     - `CUSTOMER_REQUEST`
     - `CUSTOMER_NO_SHOW`
     - `EXPERT_UNAVAILABLE`
     - `INTERNAL_ISSUE`
     - `DUPLICATED_ORDER`
     - 기타 정책 정의 코드
   - 상세 메모 입력:
     - 고객 통화 내용
     - 정책 적용 근거
     - 환불 여부 판단 근거

6. **상태 전환 처리**:
   - `order_status` → `CANCELED`
   - `expert_progress_status` → `CANCELED` 또는 `null`
   - `assignment_delivery_status` 종료 처리 (HOLD 또는 SENT 상태 종료)
   - 자동 재배정 중단 플래그 설정
   - `order_assignment_history`에 `type = CANCELED` 기록
   - 취소 시각 및 처리자(`admin_id`) 기록

7. **일정 및 알림 처리**:
   - 일정 비활성화: 연결된 SERVICE/AS 일정 `is_active = false`
   - 전문가 알림: `SENT` 상태였던 경우 전문가에게 취소 알림 발송
   - 고객 알림: 앱/문자/채널 연동 정책에 따라 취소 완료 통지

8. **결제 상태 기반 환불 판단**:
   - `paid_amount_total = 0` 인 경우:
     - 환불 프로세스 없음
     - `order_status = CANCELED` 유지
   - `paid_amount_total > 0` 인 경우:
     - 환불 정책 판단 단계 진행

9. **환불 정책 판단 단계**:
   - 예약금만 결제된 경우 (`paid_amount_total < total_price`):
     - 환불 정책 분기:
       - 전액 환불 허용: `refund_amount = paid_amount_total`, `refund_type = FULL`
       - 일부 환불: `refund_amount = 정책 계산값`, `refund_type = PARTIAL`
       - 환불 불가: `refund_amount = 0`, 환불 없음 기록
   - 전액 결제 완료 상태에서 취소 (`paid_amount_total = total_price`):
     - 전액 환불: `refund_amount = total_price`, `refund_type = FULL`
     - 일부 서비스 제공 후 취소: 환불 금액 = `total_price - 제공 완료 서비스 금액`, `refund_type = PARTIAL`

10. **환불 실행 처리**:
    - `refund_amount > 0` 인 경우:
      - 결제 게이트웨이(Hecto) 환불 요청
      - 환불 성공 시:
        - `refund_amount_total` 업데이트
        - `refund_type` 기록
        - `order_status` → `REFUNDED`
      - 환불 실패 시:
        - `CANCELED` 유지
        - 환불 실패 로그 기록
    - `refund_amount = 0` 인 경우:
      - `order_status = CANCELED` 유지
      - `refund_type` 기록하지 않음 또는 `NONE`

11. **정산 연동 처리**:
    - 정산 미확정 상태:
      - 환불 금액 차감 반영
      - 전문가 정산 대상에서 제외 또는 조정
    - 정산 확정 이후 취소:
      - 재정산 필요 플래그 설정
      - 운영 승인 후 조정

12. **최종 상태 정합성 확인**:
    - 취소만 발생한 경우: `order_status = CANCELED`, `refund_amount_total = 0`
    - 취소 + 환불 완료: `order_status = REFUNDED`, `refund_amount_total > 0`
    - 부분 환불: `order_status = REFUNDED`, `refund_type = PARTIAL`

**운영 리스크 통제 포인트**:
- 취소 처리 전 반드시 현재 주문 상태와 전문가 진행 상태를 확인해야 함
- 환불 정책 적용 시 고객 통화 내용과 정책 근거를 명확히 기록해야 함
- 정산 확정 이후 취소는 재정산 플래그 설정 및 운영 승인 필수
- 결제 게이트웨이 환불 실패 시 즉시 로그 기록 및 운영 알림 발생 필요

#### 시나리오 8: 환불 처리
본 시나리오는 다음 케이스를 모두 포함합니다: 취소 후 환불, 서비스 일부 진행 후 부분 환불, 예약금(부분 선결제) 환불, 환불 단독 처리(취소 없이 금액 조정 목적).

1. **관리자 로그인 및 주문관리 메뉴 진입**:
   - 관리자 계정으로 로그인
   - 백오피스 주문관리 메뉴 진입

2. **환불 대상 주문 검색 및 필터링**:
   - 검색 조건:
     - 주문번호
     - 고객 전화번호
     - `order_status` IN (`ASSIGNED`, `CANCELED`)
     - `paid_amount_total > refund_amount_total`
   - 환불 가능 잔액 존재 여부 확인: `remaining_refundable_amount = paid_amount_total - refund_amount_total`

3. **주문 상세 화면 진입 및 결제/상태 정보 확인**:
   - 결제 정보 확인:
     - `total_price`, `paid_amount_total`, `refund_amount_total`
     - `remaining_refundable_amount` 계산
     - 결제 트랜잭션 내역 목록
   - 현재 상태 확인:
     - `order_status`, `expert_progress_status`
     - 정산 확정 여부
     - 서비스 제공 여부

4. **환불 가능 여부 판단 및 정책 적용**:
   - 환불 불가 조건:
     - `remaining_refundable_amount = 0`
     - 결제 미완료 상태
     - 이미 FULL 환불 완료 상태
   - 환불 정책 판단:
     - 취소 후 전액 환불
     - 취소 후 일부 환불
     - 서비스 일부 제공 후 부분 환불
     - 예약금 환불
     - 단순 금액 조정 환불

5. **환불 정보 입력 및 시스템 검증**:
   - "환불" 버튼 클릭
   - 환불 정보 입력:
     - `refund_amount` 입력
     - `refund_reason_code` 선택 (`CUSTOMER_REQUEST`, `SERVICE_DEFECT`, `PRICE_ADJUSTMENT`, `INTERNAL_ERROR`, 기타 정책 코드)
     - 상세 메모 입력
   - 시스템 검증:
     - `refund_amount > 0` 확인
     - `refund_amount ≤ remaining_refundable_amount` 확인
     - 정산 확정 상태일 경우 재정산 플래그 설정 필요

6. **환불 처리 실행 (결제 게이트웨이 연동)**:
   - 결제 게이트웨이 환불 요청:
     - 환불 대상 트랜잭션 선택
     - 트랜잭션 단위 환불 실행
     - 환불 성공 응답 확인
   - 환불 실패 처리:
     - 주문 상태 변경하지 않음
     - 환불 실패 로그 기록
     - 관리자에게 오류 메시지 노출

7. **내부 데이터 업데이트 및 상태 처리**:
   - 환불 성공 시:
     - `refund_amount_total += refund_amount`
     - `refund_type` 판단:
       - `refund_amount_total = paid_amount_total` → `refund_type = FULL`
       - `refund_amount_total < paid_amount_total` → `refund_type = PARTIAL`
     - `order_status` 상태 판단:
       - 기존 `order_status = CANCELED` → 환불 발생 시 `order_status = REFUNDED`
       - 기존 `order_status = ASSIGNED`:
         - 환불이 전체 금액에 해당하면 `REFUNDED`
         - 일부 환불이면 `ASSIGNED` 유지
     - 환불 이력 로그 저장 (`refund_id`, 환불 금액, 환불 사유, 처리자 `admin_id`, 처리 시각)
   - 정산 데이터 반영:
     - 미정산 상태면 자동 차감
     - 정산 확정 상태면 재정산 플래그 설정

8. **상태 조합 정리**:
   - 취소 + 전액 환불: `order_status = REFUNDED`, `refund_type = FULL`
   - 취소 + 일부 환불: `order_status = REFUNDED`, `refund_type = PARTIAL`
   - 진행 중 일부 환불: `order_status = ASSIGNED` 유지, `refund_type = PARTIAL`
   - 예약금만 환불: `order_status = REFUNDED` 또는 `CANCELED` 정책 정의 필요, `refund_type = FULL` 또는 `PARTIAL`

9. **예약금/부분결제 특수 케이스 처리**:
   - 예시: `total_price = 100,000`, `paid_amount_total = 30,000` (예약금)
     - 30,000 전액 환불: `refund_type = FULL`, `order_status = REFUNDED`
     - 15,000 환불: `refund_type = PARTIAL`, `order_status = REFUNDED`
     - 환불 없음: `order_status = CANCELED` 유지

10. **정산 연동 처리**:
    - 환불 금액은 전문가 정산 대상 금액에서 차감
    - 부분 환불 시 전문가 수익 재계산 필요
    - 정산 확정 이후 환불은 재정산 승인 절차 필요

**운영 리스크 통제 포인트**:
- 환불과 취소는 별도 액션이지만 연속적으로 발생할 수 있음
- 환불은 반드시 트랜잭션 단위로 관리되어야 함
- 부분 환불이 여러 번 발생할 수 있도록 누적 구조 필요
- HOLD 상태 주문에 대한 환불은 취소 이후에만 허용하는 것이 안전
- `order_status`와 `refund_type`의 조합은 BI/통계에 명확히 반영되어야 함

#### 시나리오 9: 부분 환불 처리
1. **관리자 로그인 및 주문관리 메뉴 진입**:
   - 관리자 계정으로 로그인
   - 백오피스 주문관리 메뉴 진입

2. **부분 환불 대상 주문 검색**:
   - 검색 조건:
     - `paid_amount_total > 0`
     - `refund_amount_total < paid_amount_total`
     - `order_status` IN (`ASSIGNED`, `CANCELED`, `REFUNDED`)
   - 부분 환불 가능 잔액 확인: `remaining_refundable_amount = paid_amount_total - refund_amount_total`

3. **주문 상세 화면 진입 및 결제/상태 정보 확인**:
   - 결제 정보 확인:
     - `total_price`, `paid_amount_total`, `refund_amount_total`
     - `remaining_refundable_amount` 계산
     - 결제 트랜잭션 목록
   - 서비스 진행 상태 확인:
     - `expert_progress_status`
     - 서비스 완료 여부
     - 정산 확정 여부

4. **부분 환불 필요 사유 확인 및 정책 검토**:
   - 대표 케이스:
     - 서비스 일부 미제공
     - 현장 추가 비용 미발생
     - 고객 컴플레인 보상
     - 가격 조정
     - 예약금 일부 반환
   - 환불 정책 기준 검토:
     - 환불율
     - 서비스 제공 범위 대비 차감 금액
     - 내부 승인 필요 여부

5. **부분 환불 정보 입력 및 시스템 검증**:
   - "부분 환불" 버튼 클릭
   - 환불 정보 입력:
     - `refund_amount` 입력
     - `refund_reason_code` 선택
     - 상세 메모 입력
   - 시스템 검증:
     - `refund_amount > 0` 확인
     - `refund_amount < remaining_refundable_amount` 확인
     - `refund_amount ≤ paid_amount_total` 확인
     - 정산 확정 상태인 경우 경고 메시지 표시

6. **부분 환불 처리 실행 (결제 게이트웨이 연동)**:
   - 결제 게이트웨이 부분 환불 요청:
     - 특정 트랜잭션 선택 또는 자동 분배
     - 환불 요청 실행
     - 환불 성공 응답 수신

7. **내부 데이터 업데이트 및 상태 처리**:
   - 환불 성공 시:
     - `refund_amount_total += refund_amount`
     - `refund_type = PARTIAL` 설정
     - 환불 이력 테이블에 row 생성 (`refund_id`, 환불 금액, 환불 사유, 처리자, 처리 시각)
   - `order_status` 처리:
     - 기존 `order_status = ASSIGNED` → `ASSIGNED` 유지
     - 기존 `order_status = CANCELED` → `REFUNDED`로 변경
     - 기존 `order_status = REFUNDED` → `REFUNDED` 유지

8. **정산 반영**:
   - 정산 미확정 상태:
     - 전문가 정산 대상 금액에서 환불 금액 차감
     - 수수료 재계산
   - 정산 확정 상태:
     - 재정산 플래그 설정
     - 운영 승인 후 정산 조정

9. **다회차 부분 환불 처리**:
   - `refund_amount_total < paid_amount_total` 인 경우:
     - 추가 부분 환불 가능
   - `refund_amount_total = paid_amount_total` 인 경우:
     - 더 이상 부분 환불 불가
     - 사실상 FULL 환불 상태
   - 마지막 환불로 `paid_amount_total`과 동일해질 경우:
     - `refund_type = FULL`로 자동 전환 가능
     - `order_status = REFUNDED` 유지

10. **예약금 특수 케이스 및 부분 환불 예시**:
    - 예시: `total_price = 100,000`, `paid_amount_total = 30,000` (예약금)
      - 10,000 환불: `refund_amount_total = 10,000`, `refund_type = PARTIAL`, `order_status = REFUNDED`
      - 이후 20,000 추가 환불: `refund_amount_total = 30,000`, `refund_type = FULL`, 상태 정합성 유지

11. **상태 정합성 정리**:
    - 부분 환불은 `refund_type = PARTIAL`로 관리
    - `order_status`는 환불이 발생하면 `REFUNDED`가 될 수 있음
    - 진행 중 일부 환불은 `order_status = ASSIGNED` 유지 가능
    - `CANCELED` 상태에서 부분 환불 발생 시 `REFUNDED` 전환

**운영 리스크 통제 포인트**:
- 부분 환불은 반드시 `remaining_refundable_amount` 기반 검증 필요
- 다회 환불 누적 오류 방지
- 정산 확정 이후 환불은 승인 프로세스 필요
- 서비스 완료 후 부분 환불은 전문가 수익 재계산 필요
- BI 집계 시 `PARTIAL`과 `FULL` 환불 분리 집계 필요

#### 시나리오 10: 결제 링크 재발송
본 시나리오는 다음 상황을 포함합니다: 상담 후 결제 대기 상태, 예약금 결제 링크 재발송, 잔금 결제 링크 재발송, 결제 링크 만료 후 재생성, 고객 분실/미수신으로 인한 재전송.

1. **관리자 로그인 및 주문관리 메뉴 진입**:
   - 관리자 계정으로 로그인
   - 백오피스 주문관리 메뉴 진입

2. **결제 대기 주문 검색**:
   - 검색 조건:
     - `expert_progress_status = WAIT_PAYMENT`
     - 또는 `payment_status = UNPAID`
     - `order_status = ASSIGNED`

3. **주문 상세 화면 진입 및 결제 정보 확인**:
   - 결제 정보 확인:
     - `total_price`, `paid_amount_total`, `remaining_payable_amount`
     - 기존 결제 링크 발송 이력
     - 결제 링크 만료 여부
     - 결제 트랜잭션 상태

4. **재발송 필요 사유 확인**:
   - 고객이 링크 미수신
   - 결제 링크 만료
   - 금액 수정 후 신규 링크 필요
   - 예약금 → 잔금 결제 전환
   - 일부 결제 후 잔여 금액 결제

5. **결제 링크 재생성 여부 판단**:
   - 기존 링크 유효 여부 확인:
     - 만료되지 않은 링크: 단순 재전송 가능
     - 만료된 링크: 신규 링크 생성 필요
   - 금액 변경 여부 확인:
     - 금액 변경 없음: 기존 결제 금액 기준
     - 금액 변경 있음: 결제 금액 재산정, 신규 링크 필수 생성

6. **결제 링크 재발송 실행 및 시스템 검증**:
   - "결제 링크 재발송" 버튼 클릭
   - 시스템 검증:
     - `remaining_payable_amount > 0` 확인
     - `order_status = ASSIGNED` 확인
     - `CANCELED` 또는 `REFUNDED` 상태가 아님 확인

7. **결제 링크 생성 로직**:
   - 결제 대상 금액 계산:
     - 예약금 결제 단계: `payable_amount = 예약금`
     - 잔금 결제 단계: `payable_amount = total_price - paid_amount_total`
     - 부분 조정 결제: `payable_amount = 수정된 잔여 금액`
   - 결제 트랜잭션 생성:
     - `payment_intent` 생성
     - `order_id` 매핑
     - 금액 및 만료시간 설정
     - 결제 URL 생성
   - 결제 링크 이력 저장: `payment_link_id`, 생성 시각, 만료 시각, 생성자 `admin_id`, 결제 금액

8. **고객에게 링크 발송**:
   - 발송 채널 선택: SMS, 카카오 알림톡, 이메일, 앱 푸시
   - 발송 실행
   - 발송 성공 여부 기록

9. **상태 유지 및 대기**:
   - `expert_progress_status` 유지 = `WAIT_PAYMENT`
   - `order_status` 유지 = `ASSIGNED`
   - 결제 완료 이벤트 대기

10. **결제 완료 시 후속 처리 (연동 시나리오)**:
    - 결제 성공 이벤트 수신
    - `paid_amount_total` 업데이트
    - `remaining_payable_amount` 재계산
    - 전액 결제 완료 시: `expert_progress_status` → `CONFIRMED`, 구매확정 단계로 전환
    - 일부 결제 완료 시: `expert_progress_status` 유지 또는 단계별 정책 적용

11. **예외 케이스 처리**:
    - 주문이 `CANCELED` 상태일 경우: 결제 링크 재발송 불가
    - `REFUNDED` 상태일 경우: 신규 결제 링크 생성 불가
    - `HOLD` 상태 주문: `assignment_delivery_status = HOLD` 인 경우, 결제 링크 발송은 원칙적으로 상담 후에만 허용
    - 금액 수정 직후: 반드시 기존 결제 링크 만료 처리 후 신규 링크 생성

**운영 리스크 통제 포인트**:
- 동일 주문에 대해 복수 활성 결제 링크가 동시에 존재하면 안 됨
- 결제 금액 변경 시 이전 링크는 자동 비활성화 필요
- 부분 결제 구조에서는 `remaining_payable_amount` 기반 계산 필수
- 결제 링크 만료 정책 명확화 필요 (예: 24시간)
- 결제 완료 이벤트는 반드시 idempotent 처리 필요

#### 시나리오 11: 주문 메모 및 이력 기록
1. **관리자 로그인 및 주문관리 메뉴 진입**:
   - 관리자 계정으로 로그인
   - 백오피스 주문관리 메뉴 진입

2. **메모/이력 확인이 필요한 주문 검색**:
   - 검색 조건 설정:
     - 주문번호
     - 고객 전화번호
     - `order_status` (예: 문제 주문이면 `CANCELED`, `REFUNDED` 등)
     - 최근 수정일/상담일 기준 정렬
   - 검색 결과에서 특정 주문 선택

3. **주문 상세 화면 진입 및 정보 확인**:
   - 기본 탭에서 주문 정보 확인:
     - 상태 정보: `order_status`, `expert_progress_status`, `assignment_delivery_status`
     - 금액 정보: `total_price`, `paid_amount_total`, `refund_amount_total`
     - 배정 정보: `expert_master_id`, `region_group` 등
   - "메모/이력" 영역 또는 탭으로 이동

4. **기존 메모 및 이력 조회**:
   - 메모 목록 확인:
     - 메모 작성 시각
     - 작성자(`admin_id` 또는 이름)
     - 메모 타입(예: 상담, 이슈, 시스템, 내부 공유 등)
     - 메모 내용(읽기 전용)
   - 상태 이력 로그 확인:
     - 상태 변경 이력: 이전 상태 → 이후 상태, 변경 시각, 변경자
     - 배정 이력: 배정/재배정/거절/타임아웃 기록
     - 결제/환불 이력: 결제/환불 금액, 사유, 처리자

5. **신규 메모 작성 필요 여부 판단**:
   - 고객과 새 통화/문자 상담이 발생한 경우
   - 주문 정보 수동 수정(예외 처리, 강제 상태 변경 등)을 수행한 경우
   - 전문가와 별도 조율(시간 변경, 현장 특이사항 등)이 발생한 경우
   - 환불/부분 환불/보상 처리 근거를 남길 필요가 있는 경우

6. **신규 메모 작성**:
   - "메모 추가" 버튼 클릭
   - 메모 타입 선택:
     - `CUSTOMER_CALL`: 고객 상담
     - `EXPERT_CALL`: 전문가 상담
     - `ISSUE`: 클레임/이슈 처리
     - `ADJUSTMENT`: 금액/일정 조정
     - `INTERNAL_NOTE`: 내부 공유용 메모
   - 메모 내용 입력:
     - 통화 요약(누가, 언제, 무엇을 요청/설명했는지)
     - 운영 판단(어떤 정책을 적용했는지, 예외 허용 여부)
     - 후속 액션(재연락 필요, 재방문 필요, 추후 확인 일자 등)
   - 메모 저장 클릭

7. **메모 저장 시 시스템 처리**:
   - 메모 테이블에 신규 row 생성:
     - `order_id`, `admin_id`, `memo_type`, `memo_content`
     - `created_at`, `updated_at`
   - 주문 이력 로그에 "메모 추가" 이벤트 기록:
     - 이벤트 타입: `NOTE_CREATED`
     - 메모 ID 참조
   - 메모는 작성 후 수정 여부 정책에 따라 처리:
     - 권장: 내용은 수정 가능하되, 수정 이력 별도 기록
     - 삭제는 제한(슈퍼관리자만 가능 또는 완전 금지)

8. **예외 처리 또는 운영 개입과 메모 연동**:
   - 주문 상태를 수동으로 변경한 경우(예: 강제 `ASSIGNED`, 강제 `CANCELED` 등):
     - 상태 변경 액션 직후 "사유 메모 작성" 팝업 노출
     - 메모 없이 상태 강제 변경이 불가능하도록 정책 설정 가능
   - 특별 환불/보상 처리 시 메모:
     - 일반 정책과 다른 환불율·보상금을 적용한 경우
     - 왜 예외를 허용했는지 상세 기록
   - 반복 이슈 고객/전문가에 대한 패턴 기록:
     - 특정 고객이 반복 클레임을 제기하는 경우
     - 특정 전문가가 반복적으로 일정 변경/노쇼 등 발생하는 경우
     - 메모를 통해 이후 결정(블랙리스트, 패널티 등)의 근거로 활용

9. **주문 이력 기반 회귀 분석/모니터링**:
   - 운영팀장이 이슈 주문을 되돌아보며 메모 및 이력을 분석:
     - 클레임 많은 주문 패턴 파악
     - 특정 채널/카테고리에서 반복되는 문제 파악
   - QA/CS 교육용 사례 발굴:
     - 잘 처리된 케이스, 잘못 처리된 케이스를 메모 이력 기반으로 리뷰
   - 정책 개선 근거로 활용:
     - 취소/환불 정책, 예약금 정책, 배정 정책 개선 시 메모 데이터 활용

**운영 리스크 및 정책 포인트**:
- 메모는 "나중에 법적/분쟁 대응"의 근거가 될 수 있으므로, 감정적인 표현/비속어/개인 감정 기록 금지 가이드 필요
- 고객에게 공유되지 않는 내부 메모라는 점을 명시하되, 항상 외부에 노출되어도 문제 없는 수준으로 작성하도록 교육 필요
- 메모 삭제는 원칙적으로 금지하거나, 최소한 "삭제 이력"은 남겨야 함
- 중요한 예외 조치(정책 외 환불, 서비스 무료 제공 등)는 반드시 메모와 함께 태그(예: `IMPORTANT`, `EXCEPTION`)를 붙여 검색이 용이하도록 설계

#### 시나리오 12: 엑셀 업로드로 주문 일괄 등록
1. **관리자 로그인**:
 - 관리자는 백오피스 로그인 페이지에서 인증을 완료한다
 - 로그인 성공 후 기본 진입 화면으로 이동한다

2. **주문관리 메뉴 진입**:
 - 사이드 메뉴에서 메뉴 02 주문관리를 선택한다
 - 주문 목록 화면이 기본으로 표시된다

3. **엑셀 업로드 기능 진입**:
 - 주문 목록 상단에서 "엑셀 업로드" 버튼을 클릭한다
 - 엑셀 업로드 전용 팝업 화면이 표시된다

4. **서비스 카테고리 선택**:
 - 업로드 화면에서 "서비스 카테고리 선택" 드롭다운을 노출한다
 - 운영자는 다음 구조로 카테고리를 선택한다
   - 대분류 카테고리 선택
   - 중분류 카테고리 선택
 - 선택된 카테고리 정보는 service_category_id로 내부 매핑된다

5. **카테고리별 엑셀 템플릿 다운로드**:
 - 운영자는 "템플릿 다운로드" 버튼을 클릭한다
 - 시스템은 선택된 service_category_id를 기준으로 서비스관리 메뉴의 설정을 조회한다
 - 서비스관리 메뉴의 추가입력항목 탭에서 해당 카테고리의 추가 필드 정의를 조회한다
   - 필드명
   - 필드 타입 예: TEXT, NUMBER, DATE, SELECT
   - 필수 여부
   - 표시 순서
 - 시스템은 공통 컬럼과 카테고리별 추가입력항목 컬럼을 합쳐 엑셀 템플릿 컬럼 구조를 생성한다
- 공통 컬럼 구성 예시
- 필드1(필수값) : 채널코드(채널관리 메뉴에 등록된 채널의 코드를 기입한다.예>CH_700,CH_888,CH_900등)
- 필드2(선택값) : 채널주문번호(제휴 채널의 주문번호, 중복불가)
- 필드3(필수값) : 고객명(고객이름) 
- 필드4(필수값) : 고객연락처(고객전화번호, 010-1234-5678 형식)
- 필드5(필수값) : 서비스코드(서비스 카테고리 소분류 코드)
- 필드6(필수값) : 서비스수량(서비스 수량, 기본값:1)
- 필드7(필수값) : 서비스 주소(서비스 제공 주소)
- 필드8(선택값) : 상세 주소(동,호수 등)
- 필드9(선택값) : 서비스 희망일
- 필드10(선택값) : 고객요청사항(고객의 특별 요청사항)
- 필드11(선택값) : 결제금액(예약금액or선결제 금액, 숫자만입력)
- 필드12(선택값) : 예약금결제일
- 필드13(선택값) : 관리자메모(관리자용 메모)
- 필드14(선택값) : 전문가전달메모(전문가에게 전달할 메모)
- 필드15(필수값) : 주문신청 일시
- 카테고리별 추가입력항목 컬럼 구성 예시
   - 추가입력항목 탭에 정의된 필드 코드 또는 필드 키를 컬럼명으로 사용한다
   - 예: 실내기대수, 실외기설치위치, 층수, 평수 등
   - 필수 항목은 템플릿 내 주석 또는 예시 행으로 "필수" 표시를 한다
   - 카테고리별 추가입력 항목 컬럼은 필수가 아닌 선택값으로 넣는다.
- 시스템은 동적으로 생성된 컬럼 구조를 기반으로 .xlsx 파일을 생성하고 다운로드를 시작한다

6. **템플릿 작성**:
 - 운영자는 다운로드 받은 카테고리별 템플릿 파일을 연다
 - 각 행에 주문 데이터를 입력한다
   - 공통 컬럼에 고객/주문 기본 정보를 입력한다
   - 추가입력항목 컬럼에 현장에 필요한 상세 정보를 입력한다
 - 필수 컬럼이 비어 있지 않도록 내부 가이드에 따라 작성한다
 - 작성이 완료된 파일을 저장한다

7. **엑셀 파일 업로드**:
 - 다시 백오피스 주문관리의 엑셀 업로드 화면으로 돌아온다
 - 동일한 서비스 카테고리가 선택되어 있는지 확인한다
 - "파일 선택" 버튼을 클릭하여 작성 완료된 엑셀 파일을 선택한다
 - "업로드" 버튼을 클릭한다

8. **업로드 1차 검증 처리**:
 - 시스템은 업로드된 파일의 기본 형식을 검증한다
   - 파일 확장자가 .xlsx인지 확인한다
   - 템플릿 구조 버전 또는 hidden metadata를 사용해 템플릿 호환성을 검증한다
 - 선택된 service_category_id와 템플릿 내부의 카테고리 메타 정보가 일치하는지 확인한다
 - 컬럼 구조 검증을 수행한다
   - 공통 컬럼이 모두 존재하는지 확인한다
   - 서비스관리 → 추가입력항목 탭에 정의된 필드들이 모두 컬럼으로 존재하는지 확인한다
   - 정의되지 않은 추가 컬럼이 있는 경우 경고 정책을 적용한다

9. **업로드 데이터 행 단위 유효성 검증**:
 - 시스템은 각 행을 순회하며 유효성 검증을 수행한다
 - 공통 컬럼 검증
   - `customer_name`, `customer_phone`, `address`, `requested_date`, `total_price` 등의 필수값 존재 여부를 확인한다
   - 전화번호 포맷, 날짜 포맷, 금액 숫자 포맷을 검증한다
   - `channel_order_id` 중복 여부를 검사한다 동일 채널 내 중복인지 확인한다
 - 추가입력항목 검증
   - 서비스관리 → 추가입력항목 탭에서 필수로 정의된 항목들이 모두 입력되었는지 확인한다
   - 각 항목 타입에 맞는 데이터 형태인지 검증한다
     - NUMBER 타입은 숫자인지
     - DATE 타입은 날짜 포맷인지
     - SELECT 타입은 허용된 옵션 값인지
 - 검증 결과
   - 이상이 없는 행은 "유효"로 분류한다
   - 오류가 있는 행은 오류 사유와 함께 "오류 행" 리스트로 분류한다

10. **업로드 정책 선택**:
 - 일부 행에서 오류가 발생한 경우 시스템은 업로드 정책을 사용자에게 안내한다
 - 정책 옵션 
   - 전체 실패: 오류 행이 하나라도 있으면 전체 업로드를 취소하고 오류 리포트를 제공한다
 - 운영 정책에 따라 기본값을 정하고, 필요 시 설정에서 변경 가능하도록 한다

11. **주문 생성 처리**:
 - 유효 행에 대해 주문 생성 로직을 수행한다
 - 주문 생성 시 기본 상태 설정
   - `order_status` = `NEW`
   - `assignment_delivery_status` = `NONE`
   - `expert_progress_status` = null
   - `service_category_id` = 선택된 카테고리 ID
 - 고객 정보 생성 또는 매핑
   - `customer_phone` 기준으로 기존 고객 여부를 확인한다
   - 기존 고객이 있으면 customer_id를 재사용한다
   - 없으면 새 customer_id를 생성한다
 - 금액 정보 저장
   - total_price를 저장한다
   - paid_amount가 입력된 경우 paid_amount_total로 반영하고, 결제 트랜잭션 정보를 `type` = IMPORTED로 생성한다
 - 추가입력항목 데이터 저장
   - 서비스관리 → 추가입력항목 탭에 정의된 필드 키를 기준으로 주문 추가 정보 테이블에 값을 저장한다
   - 저장 구조 예시: `order_id`, `field_key`, `field_value`
 - 주문 생성 이력 기록
   - 주문 로그에 `CREATED_BY_EXCEL` 이벤트를 기록한다
   - 생성자 admin_id와 업로드 batch_id를 함께 기록한다

12. **업로드 결과 요약 및 피드백**:
 - 시스템은 업로드 결과 요약을 화면에 표시한다
   - 전체 행 수
   - 성공 생성 주문 수
   - 실패 행 수
 - 실패 행이 있을 경우
   - 실패 행만 포함된 "오류 행 엑셀 파일" 다운로드 링크를 제공한다
   - 각 행마다 오류 사유를 컬럼으로 포함한다
 - 생성된 주문으로 바로 이동할 수 있는 링크를 제공한다 예: "성공 건 주문 목록 보기"

13. **후속 프로세스 연계**:
 - 생성된 주문에 대해 운영팀은 일반 신규 주문과 동일하게 검수 및 배정 프로세스를 수행한다
 - paid_amount가 존재하는 주문은 총결제 금액과 동이하면 결제 완료 상태로 간주되며, 총결제 금액보다 작을 경우 부분 결제 완료 상태로 간주된다. 이후 환불·정산 로직과 연계된다
 - 추가입력항목 데이터는 전문가웹/앱에서 현장 준비 정보로 활용된다

14. **운영 및 정책 관점 주의사항**:
 - 서비스관리에서 추가입력항목 정의를 변경하면, 이후 생성되는 템플릿 컬럼 구조도 함께 변경되므로 변경 시점 관리에 유의해야 한다
 - 기존에 배포된 템플릿 파일을 사용하는 운영자가 있을 수 있으므로, 템플릿 버전 또는 생성일 표시를 통해 호환성 문제를 줄여야 한다
 - 카테고리별 템플릿을 잘못 선택하여 업로드하는 것을 방지하기 위해, 업로드 시 템플릿 내 카테고리 메타 정보와 현재 선택된 카테고리를 반드시 교차 검증해야 한다
 - 대량 업로드는 운영 리스크가 크므로, 특정 권한 이상에서만 사용 가능하도록 권한 제어가 필요하다

#### 시나리오 12-1: 한 고객의 다중/다수 서비스 구매 처리 (엑셀 업로드 확장)
본 시나리오는 한 명의 고객이 동일 시점에 두 개 이상의 서비스를 구매한 경우를 엑셀 업로드 구조에 반영하는 방식을 정의한다.
1. **기본 원칙 정의**:
- 한 행(row)은 하나의 서비스 주문을 의미한다
 - 한 고객이 여러 서비스를 구매한 경우, 서비스 개수만큼 여러 행으로 작성한다
 - 시스템 내부에서는 "서비스 1개 = 주문 1건"으로 생성한다
 - 여러 주문을 하나의 묶음으로 식별하기 위해 `order_group_key` 및 `order_group_id`를 사용한다

2. **엑셀 템플릿 컬럼 확장 정의**:
- 공통 컬럼에 다음 항목을 추가한다
   - `order_group_key`: 동일 고객의 다중 서비스 주문을 하나의 묶음으로 식별하기 위한 그룹 키(선택 입력)
- `order_group_key` 작성 규칙
   - 동일 고객의 동일 묶음 주문에는 동일한 값을 입력한다
   - 묶음이 아닌 단일 주문인 경우 비워둘 수 있다
   - 예시 값
     - 20250301-홍길동-1
     - 외부 채널 주문번호와 동일 값

3. **업로드 시 그룹 처리 로직**:
- 시스템은 업로드된 각 행을 순회하며 `order_group_key` 값을 확인한다
 - `order_group_key`가 존재하는 경우
   - 동일한 `order_group_key`를 가진 행들을 하나의 그룹으로 인식한다
   - 내부적으로 `order_group_id`를 생성한다
   - 동일한 `order_group_key`를 가진 모든 주문에 동일한 `order_group_id`를 저장한다
- `order_group_key`가 없는 경우
   - 해당 주문은 개별 주문으로 생성한다
   - `order_group_id`는 `null` 또는 단독 생성된 값으로 처리한다

4. **서로 다른 카테고리 서비스 동시 구매 처리**:
- 서로 다른 서비스 카테고리의 경우
   - 각 카테고리별 템플릿을 사용하여 개별 업로드를 수행한다
   - 동일 고객의 묶음 주문일 경우 동일한 `order_group_key` 값을 사용한다
- 업로드 완료 후 시스템 처리
   - 서로 다른 업로드 batch에서 생성된 주문이라도
   - `order_group_key`가 동일하면 동일한 `order_group_id`로 매핑한다

5. **주문 생성 시 데이터 저장 구조**:
- 주문 테이블에 다음 필드를 추가한다
   - `order_group_id`: 내부 묶음 식별자
   - `order_group_key`: 외부 입력 기반 그룹 키
- 주문 생성 시 처리
   - `order_group_key`가 존재하면 기존 동일 키의 `order_group_id` 조회
   - 존재하면 동일 `order_group_id` 사용
   - 존재하지 않으면 신규 `order_group_id` 생성

6. **운영 화면에서의 활용**:
- 주문 목록 화면
   - `order_group_id`가 동일한 주문은 묶음 표시 아이콘 또는 태그로 표시한다
   - 그룹 클릭 시 해당 그룹의 주문 목록만 필터링하여 조회 가능하도록 한다
- 주문 상세 화면
   - 동일 `order_group_id`를 가진 다른 주문 목록을 하단에 표시한다
   - 운영자가 "묶음 전체 보기" 기능을 통해 관련 주문을 한 번에 확인할 수 있도록 한다

7. **배정 및 정산과의 관계**:
- 배정은 주문 단위로 수행한다
   - 같은 그룹이라도 각 주문은 독립적으로 배정된다
   - 동일 전문가에게 배정되도록 운영자가 수동 조정 가능
- 정산은 주문 단위로 계산한다
   - 그룹 단위 정산은 기본 제공하지 않는다
   - 필요 시 그룹 단위 합산 리포트만 제공한다

8. **취소 및 환불 시 그룹 처리 원칙**:
- 기본 원칙
   - 취소 및 환불은 주문 단위로 처리한다
   - 한 그룹의 일부 주문만 취소 가능
- 운영 편의 기능
   - "묶음 전체 취소" 버튼은 향후 확장 기능으로 고려 가능
   - 해당 기능 구현 시 그룹 내 모든 주문에 동일 상태 전환을 일괄 적용한다

9. **운영 리스크 통제 포인트**:
- 서로 다른 카테고리 템플릿 업로드 시 동일 `order_group_key`를 정확히 입력하도록 운영 가이드 필요
 - 동일 `channel_order_id`와 `order_group_key` 혼동 방지 필요
 - 그룹 단위 취소/환불 오처리 방지를 위해 기본 단위는 항상 "주문 단위"임을 유지한다
 - BI 및 통계 집계 시 그룹 단위와 주문 단위를 구분하여 정의한다


#### 시나리오 13: 주문 데이터 엑셀 다운로드
본 시나리오는 주문 데이터를 엑셀(.xlsx) 형식으로 다운로드하여 운영 분석, 정산 검토, 외부 보고, CS 이력 관리 등에 활용하는 흐름을 정의한다.
1. **관리자 로그인**:
 - 관리자는 백오피스 로그인 페이지에서 인증을 완료한다
 - 로그인 성공 후 기본 화면으로 이동한다
2. **주문관리 메뉴 진입**:
 - 상단 또는 사이드 메뉴에서 메뉴 02 주문관리를 선택한다
 - 주문 목록 화면이 표시된다
3. **다운로드 대상 주문 조회**:
- 검색 조건 설정
   - 기간 필터: 주문 생성일, 예약일, 결제일, 취소일 등
   - `order_status` 필터
   - `expert_progress_status` 필터
   - `assignment_delivery_status` 필터
   - 서비스 카테고리(대분류 > 중분류)
   - 채널(`channel_code`)
   - `region_group`
   - 전문가 ID
   - `order_group_id` 또는 `order_group_key`
- 검색 실행
   - 조건에 맞는 주문 목록이 화면에 표시된다
   - 총 건수가 상단에 표시된다

4. **엑셀 다운로드 버튼 클릭**:
- "엑셀 다운로드" 버튼 클릭
 - 다운로드 옵션 선택 팝업 노출

5. **다운로드 옵션 선택**:
- 다운로드 범위 선택
   - 현재 검색 조건 전체 다운로드
   - 현재 페이지 다운로드
   - 선택 체크박스 주문만 다운로드
- 컬럼 구성 선택
  - 기본 컬럼만 다운로드
   - 기본 컬럼 + 추가입력항목 포함
   - 사용자 정의 컬럼 선택 기능(확장 기능)
- 파일 분할 정책
   - 건수 초과 시 자동 분할 다운로드
   - 예: 10,000건 초과 시 분할 파일 생성

6. **엑셀 컬럼 구성 정의**:
- 기본 공통 컬럼
  - `order_id`
   - `order_group_id`
   - `order_group_key`
   - `channel_code`
   - `channel_order_id`
   - `customer_name`
   - `customer_phone`
   - `address`
   - `service_category_name`
   - `region_group`
   - `expert_master_id`
   - `expert_worker_id`
   - `order_status`
   - `expert_progress_status`
   - `assignment_delivery_status`
   - `total_price`
   - `paid_amount_total`
   - `refund_amount_total`
   - `refund_type`
   - `created_at`
   - `assigned_at`
   - `service_date`
   - `canceled_at`
   - `refunded_at`
  - 이외에 주문관리에 포함된 모든 컬럼을 선택 다운로드 할수있게 함. 
- 추가입력항목 컬럼 구성
  - 서비스관리 → 추가입력항목 탭에서 정의된 필드 목록 조회
   - 선택된 주문들의 서비스 카테고리를 기준으로 필요한 필드 집합 계산
   - 동적으로 컬럼 생성
   - 각 주문의 추가입력항목 값 매핑
   - 값이 없는 경우 공란 처리

7. **데이터 추출 처리**:
- 현재 검색 조건을 기준으로 DB 조회
 - 대량 데이터의 경우 비동기 처리
   - 다운로드 요청 시 background job 생성
   - 처리 완료 시 알림 또는 다운로드 링크 제공
- 데이터 정합성 보장
   - 조회 시점 기준 스냅샷 처리
   - 다운로드 도중 상태 변경이 발생하더라도 동일 스냅샷 유지

8. **엑셀 파일 생성**:
- 파일 포맷은 .xlsx
 - 첫 번째 행은 컬럼 헤더
 - 금액 필드는 숫자 형식 유지
 - 날짜 필드는 ISO 포맷 또는 yyyy-mm-dd hh:mm:ss 형식 유지
 - 상태 값은 내부 코드 또는 표시명 정책에 따라 결정

9. **다운로드 완료 처리**:
- 파일 다운로드 시작
 - 다운로드 이력 기록
   - admin_id
   - 다운로드 조건
   - 다운로드 건수
   - 다운로드 시각
- 보안 정책
   - 고객 개인정보 포함 파일은 암호화 다운로드 옵션 고려
   - 일정 기간 이후 다운로드 링크 자동 만료

10. **특수 케이스 처리**:
- 서로 다른 카테고리가 혼합된 경우
  - 모든 추가입력항목 컬럼을 union 형태로 생성
   - 특정 주문에 해당 없는 컬럼은 공란 처리
- 묶음 주문(`order_group_id`)
  - 동일 `order_group_id`를 가진 주문은 각각 별도 행으로 출력
   - 필요 시 그룹 단위 집계 리포트는 별도 기능으로 확장
- 대량 다운로드
  - 동시 다운로드 요청 제한
   - 시스템 부하 방지를 위한 rate limit 적용

11. **운영 리스크 통제 포인트**:
- 개인정보 대량 유출 방지를 위해 권한 기반 접근 제어 필요
 - 다운로드 로그는 최소 1년 이상 보관 권장
 - 추가입력항목 변경 시 과거 주문 데이터와 컬럼 구조 불일치 가능성 고려
 - BI 집계용 다운로드와 운영용 다운로드를 분리하는 것이 장기적으로 바람직


#### 시나리오 14: 주문 강제 상태 수정(운영 개입)
본 시나리오는 정상적인 상태머신 흐름을 우회하여
 운영자가 특정 주문의 상태를 강제로 변경해야 하는 예외 상황을 정의한다.
본 기능은 고위 권한 관리자에게만 허용한다.

1. **관리자 로그인**:
- 운영 권한이 있는 계정으로 로그인한다
 - 해당 계정이 "강제 상태 수정 권한"을 보유했는지 검증한다
 - 권한이 없을 경우 해당 기능은 노출되지 않는다

2. **주문 상세 화면 진입**:
- 주문관리 메뉴에서 특정 주문을 조회한다
 - 주문 상세 화면에 진입한다
 - 현재 상태 확인
   - `order_status`
   - `expert_progress_status`
   - `assignment_delivery_status`
   - 결제 상태
   - 정산 확정 여부

3. **강제 상태 수정 버튼 클릭**:
- "강제 상태 수정" 버튼 클릭
 - 경고 팝업 노출
   - 정상 상태 흐름을 우회한다는 경고 문구 표시
   - 해당 변경이 정산 및 통계에 영향을 줄 수 있음을 안내
   - 관리자 재확인 필요

4. **수정 대상 상태 선택**:
- 수정 가능한 상태 목록은 현재 상태에 따라 제한적으로 노출한다
- 수정 가능한 대상
  - `order_status` 변경
   - `expert_progress_status` 변경
   - `assignment_delivery_status` 변경
- 허용 예시
  - ASSIGNED → UNASSIGNED
   - ASSIGNED → CANCELED
   - WAIT_PAYMENT → CONFIRMED
   - IN_SERVICE → CONFIRMED
   - HOLD → SENT
- 절대 허용 불가 예시
  - REFUNDED → ASSIGNED
   - CANCELED → ASSIGNED (결제/배정 정합성 깨짐)
   - 정산 확정 완료 주문을 서비스 진행 상태로 되돌리는 것

5. **강제 변경 사유 입력**:
- 사유 코드 선택
  - SYSTEM_ERROR
   - PG_SYNC_ERROR
   - EXPERT_APP_ERROR
   - DATA_CORRECTION
   - INTERNAL_POLICY_EXCEPTION
- 상세 사유 메모 입력 필수
  - 변경 필요 배경 설명
   - 고객/전문가 통화 여부
   - 관련 정책 근거
- 메모 미입력 시 저장 불가

6. **시스템 사전 정합성 검증**:
- 결제 관련 검증
  - `paid_amount_total` > 0 인데 `order_status`를 NEW로 변경 시 차단
   - `refund_amount_total` > 0 인 상태에서 ASSIGNED로 변경 시 차단
- 배정 관련 검증
  - `assignment_delivery_status` = SENT 상태에서 `expert_master_id`가 null이면 차단
   - HOLD 상태에서 전문가 미존재 시 차단
- 정산 관련 검증
  - 정산 확정 상태인 경우 일부 상태 변경 차단
   - 변경 시 재정산 플래그 자동 설정

7. **상태 변경 실행**:
- 트랜잭션 단위 처리
  - `order_status` 변경
   - `expert_progress_status` 변경
   - `assignment_delivery_status` 변경
   - 필요 시 배정 정보 초기화
- 자동 후처리 로직
  - ASSIGNED → UNASSIGNED 변경 시 자동 재배정 비활성화
   - HOLD → SENT 변경 시 전문가 알림 발송
   - WAIT_PAYMENT → CONFIRMED 변경 시 결제 완료 로직 트리거 여부 정책 정의

8. **이력 기록**:
- 주문 이력 로그에 기록
  - `event_type` = FORCE_STATUS_CHANGE
   - 이전 상태 값 기록
   - 변경 후 상태 값 기록
   - `admin_id` 기록
   - 변경 시각 기록
- 메모 자동 생성
  - `memo_type` = FORCE_CHANGE
   - 입력한 사유 자동 저장

9. **알림 및 연동 처리**:
- 상태에 따라 고객 알림 여부 판단
  - CANCELED로 변경 시 고객 알림
   - CONFIRMED로 변경 시 전문가 알림
- 외부 채널 연동 여부 확인
  - 채널 주문의 경우 상태 동기화 정책 별도 정의

10. **운영 리스크 통제 포인트**:
- 강제 상태 수정은 감사 로그 대상 기능으로 분류한다
 - 변경 이력은 삭제 불가
 - 하루 강제 수정 횟수 제한 또는 알림 체계 필요
 - 강제 수정 주문은 BI에서 별도 플래그로 집계

11. **정책적 권장사항**:
- 강제 수정은 최후 수단으로 사용한다
 - 시스템 버그로 반복 발생하는 경우 로직 수정이 우선이다
 - 강제 수정 빈도는 월 단위 리포트로 관리한다

#### 시나리오 15: 일정 변경에 따른 주문 정보 수정
본 시나리오는 예약/방문 일정이 변경될 때
 주문 정보와 일정(캘린더) 정보를 어떻게 함께 수정·동기화할지에 대한 운영 절차를 정의한다.
1. **관리자 로그인**:
 - 운영 계정으로 백오피스에 로그인한다
 - 로그인 성공 후 메뉴 02 주문관리로 이동한다
2. **일정 변경 요청이 들어온 주문 검색**:
- 검색 조건 설정
   - 주문번호로 검색
   - 고객 전화번호로 검색
   - `service_date`(예약일) 기준으로 오늘/내일 일정 필터
   - `expert_progress_status` IN (RESERVED, IN_SERVICE, AS_REQUEST) 등 일정이 연관된 상태만 필터
- 검색 결과에서 일정 변경 요청이 들어온 주문을 선택한다
3. **주문 상세 화면 진입 및 현재 상태 확인**:
- 주문 상세 화면 진입
 - 기본 정보 확인
   - `order_status` (ASSIGNED, CANCELED 등)
   - `expert_progress_status` (RESERVED, IN_SERVICE 등)
   - `assignment_delivery_status` (SENT/HOLD 등)
   - 현재 `service_date`, `service_time` 정보
   - 배정된 `expert_master_id`, `expert_worker_id`
- 일정 이벤트 연동 여부 확인
   - 해당 주문과 연결된 캘린더 이벤트 존재 여부
   - `event_type` (SERVICE / AS)
   - `event_start_time`, `event_end_time`
4. **일정 변경 요청 내용 파악**:
- 고객 또는 전문가로부터 전달받은 변경 내용 확인
   - 날짜 변경 요청 (예: 3/10 → 3/12)
   - 시간대 변경 요청 (예: 오전 → 오후)
   - 동일 일자 내 시간만 이동
   - 날짜 + 시간 모두 변경
- 추가 조건 확인
   - 당일/당일 직전 변경인지 여부 (페널티/정책 연동 가능)
   - 주소 변경 수반 여부 (`region_group` 변경 여부 확인)
   - 기존 전문가 일정과의 충돌 가능성
5. **변경 가능 여부 검토**:
- 상태 기반 검토
   - `expert_progress_status` = RESERVED 인 경우: 일반적인 일정 변경 허용
   - `expert_progress_status` = IN_SERVICE 인 경우: 원칙적으로 일정 변경이 아닌 "실제 서비스 중"이므로, 시간 변경보다는 이력 메모 위주 처리
   - `expert_progress_status` = CONFIRMED 인 경우: 서비스 종료로 간주, 일정 변경 불가 (예외 시 강제 상태 수정 시나리오 연계)
- 전문가 스케줄 기반 검토
   - 동일 전문가의 해당 시간대 다른 서비스 일정 유무 확인
   - 충돌 시, 다른 시간 제안 또는 재배정 필요 여부 판단
- 채널/정책 기반 검토
   - 채널별 일정 변경 가능 마감 시간 확인
   - 일정 변경 수수료/제한 정책 존재 시 해당 여부 확인
6. **일정 변경 실행 – 기본 케이스 (주소/권역 불변, 전문가 유지)**:
- "일정 변경" 액션 클릭
 - 새로운 날짜/시간 입력
   - `service_date`_new, `service_time`_new 선택
   - 시스템에서 과거 시점, 영업 불가 시간대 차단
- 시스템 처리 – 주문 정보 업데이트
   - 주문의 `service_date`, `service_time`을 새로운 값으로 업데이트
   - `expert_progress_status`는 RESERVED 상태 유지
   - `order_status`, `assignment_delivery_status`는 변경 없음
- 시스템 처리 – 캘린더 이벤트 업데이트
   - 기존 이벤트를 삭제하지 않고 start_time, end_time을 수정한다
   - event 변경 이력을 event_history에 기록한다(이전 시간 → 변경 시간)
   - 전문가웹/앱의 일정 화면에서 해당 이벤트가 새 시간으로 표시된다
- 알림 발송
   - 고객에게 일정 변경 알림 발송 (SMS/알림톡 등)
   - 전문가에게 일정 변경 알림 발송 (앱 푸시 등)
- 메모 기록
   - 주문 메모에 일정 변경 사유 및 변경 전/후 시간을 기록한다
   - `memo_type` = SCHEDULE_CHANGE 로 저장
7. **일정 변경 실행 – 주소/권역 변경이 수반되는 케이스**:
- 고객 주소 변경 요청 확인
   - `address`, 상세주소, 동/호수 변경 여부 확인
   - `region_group`이 변경되는지 여부 확인
- `region_group` 변경 여부 판단
   - 기존 주소 기반 `region_group`_old
   - 변경된 주소 기반 `region_group`_new
   - `region_group`_old ≠ `region_group`_new 인 경우, 기존 전문가가 해당 권역을 지원하는지 확인
- 기존 전문가가 새 권역 지원 불가인 경우
   - 현재 배정을 해제해야 할지 판단
   - 정책에 따라 다음 중 하나 선택
     - 기존 전문가 유지(예외 허용)
     - 기존 배정 취소 후 UNASSIGNED로 되돌려 재배정(재배정 시나리오 연계)
- 주소 정보 및 `region_group` 업데이트
   - 주문의 `address`, `region_group` 값을 새로운 값으로 수정
   - 고객 기본 주소와 동기화 여부는 별도 정책에 따름
- 캘린더 이벤트 반영
   - 동일 전문가 유지 시: 이벤트 시간/장소 메모만 수정
   - 전문가 변경 시: 기존 이벤트 is_active = false 처리, 새 전문가 기준으로 신규 이벤트 생성
- 메모 기록
   - 주소/권역 변경 사유 및 전문가 유지/변경 여부 기록
   - 정책 예외가 있었다면 반드시 명시
8. **일정 변경 실행 – 전문가 교체가 수반되는 케이스**:
- 일정 변경 요청 + 기존 전문가 일정 충돌 발생
   - 새 시간대에 해당 전문가의 다른 예약이 이미 존재
   - 또는 전문가가 해당 시간에 불가 응답
- 전문가 재배정 필요 여부 판단
   - 고객이 날짜/시간 변경보다 전문가 교체를 우선하는지 확인
   - 동일 권역 내 다른 전문가 후보 탐색
- 전문가 변경 처리
   - 기존 주문 배정 해제
     - `order_status`: ASSIGNED → UNASSIGNED (필요 시)
     - `expert_master_id`, `expert_worker_id` 초기화 또는 변경
     - `assignment_delivery_status`: SENT → NONE
     - `order_assignment_history`에 REASSIGN_FROM_SCHEDULE_CHANGE 기록
  - 새 전문가 배정
     - 자동/수동 배정 로직을 사용해 새 전문가 선택
     - ASSIGNED 및 SENT 상태로 전환
     - 새 전문가 기준 캘린더 이벤트 생성
- 고객/전문가 알림
   - 전문가 교체 사실 + 변경된 방문 일정 고객에게 안내
   - 새 전문가에게 신규 일정 알림 발송
- 메모 기록
   - 일정 변경으로 인한 전문가 교체 사유 기록
   - 고객 동의 여부, 통화 내용 저장

9. **일정 변경 불가 케이스 처리**:
- 일정 변경 요청이 정책 범위를 벗어나거나
   - 너무 임박한 시간(예: 방문 1시간 전)
   - 이미 서비스 완료/구매확정 상태
   - 정산 확정 이후 특별 변경 요청
- 변경 불가 안내
   - 고객에게 불가 사유 설명
   - 필요 시 취소/환불/재예약 대안 제시
- 메모 기록
   - 일정 변경 요청 거절 사유 기록
   - 향후 CS 이력 확인용
10. **운영 리스크 및 정책 포인트**:
- 일정 변경은
   - 주문의 `service_date`/time
   - 전문가 캘린더 이벤트
   - 전문가 배정 상태
 를 동시에 다루므로, 반드시 트랜잭션/동기화 고려가 필요하다
- 일정 변경 이력이 많을수록 CS/품질 이슈 가능성이 있으므로
   - 주문별 일정 변경 횟수를 집계
   - 특정 고객/전문가/채널에서 빈번하게 발생하는지 통계로 관리
- 당일/임박 변경에 대한 수수료, 패널티, 페널티 점수 등은 별도 정책 문서와 연동
- 강제 일정 변경(운영이 임의로 전문가나 일정을 바꾼 경우)은
   - 반드시 메모와 함께 남기고
   - 필요 시 "강제 상태 수정" 시나리오와 연계해 감사 가능해야 한다

#### 시나리오 16: 서비스 금액 수정(상담 후 금액 조정)
본 시나리오는 상담 과정 또는 운영 개입에 의해 주문의 `total_price`를 변경해야 하는 경우를 정의한다.
본 기능은 결제·환불·정산에 직접적인 영향을 미치므로 권한 기반 접근 제어가 필요하다.
1. **관리자 로그인**:
- 금액 수정 권한을 보유한 관리자 계정으로 로그인한다
 - 권한이 없는 계정에는 "금액 수정" 버튼이 노출되지 않는다
2. **주문 상세 화면 진입**:
- 주문관리 메뉴에서 대상 주문을 검색한다
 - 주문 상세 화면으로 이동한다
 - 현재 금액 상태 확인
  - `total_price`
   - `paid_amount_total`
   - `refund_amount_total`
   - `remaining_payable_amount` = `total_price` - `paid_amount_total`
   - 결제 트랜잭션 내역
3. **금액 수정 가능 여부 판단**:
- 금액 수정 불가 상태
  - `order_status` = REFUNDED 이고 `refund_amount_total` = `paid_amount_total` 인 경우
   - 정산 확정 완료 상태에서 환불 없이 금액 하향 조정 시
   - 법적 분쟁으로 잠금 처리된 주문
- 금액 수정 허용 상태
  - 결제 전 상태
   - 일부 결제 상태
   - 전액 결제 상태이지만 환불 가능 상태
   - 서비스 진행 전 상태
4. **금액 수정 액션 실행**:
- "금액 수정" 버튼 클릭
 - 수정 입력 팝업 노출
  - 기존 `total_price` 표시
   - 신규 `total_price` 입력 필드
   - 변경 사유 코드 선택
    - OPTION_CHANGE
     - PRICE_ADJUSTMENT
     - DISCOUNT_APPLIED
     - INPUT_ERROR
     - INTERNAL_EXCEPTION
  - 상세 사유 메모 입력 필수
5. **시스템 사전 검증**:
- 신규 `total_price` ≥ 0 확인
- `paid_amount_total`와의 관계 검증
  - 신규 `total_price` ≥ `paid_amount_total` 인 경우
 → 추가 결제 필요 또는 정상 유지
  - 신규 `total_price` < `paid_amount_total` 인 경우
 → 초과 결제 발생
 → 부분 환불 필요 경고 표시
- `refund_amount_total`와의 관계 검증
  - `refund_amount_total` > 신규 `total_price` 인 경우 차단
6. **금액 수정 분기 처리**:
- 케이스 A: 결제 전 금액 수정
  - `total_price`를 신규 값으로 업데이트
   - `remaining_payable_amount` 재계산
   - 기존 발송된 결제 링크 자동 만료 처리
   - 필요 시 신규 결제 링크 발송 가능 상태 유지
   - `order_status`, `expert_progress_status` 변경 없음
- 케이스 B: 일부 결제 상태에서 금액 상향 조정
  - `total_price` 증가
   - `remaining_payable_amount` 증가
   - 기존 결제 트랜잭션 유지
   - 추가 결제 링크 발송 필요
   - `expert_progress_status` = WAIT_PAYMENT 유지
- 케이스 C: 일부 결제 상태에서 금액 하향 조정
  - 신규 `total_price` ≥ `paid_amount_total` 인 경우
 → `remaining_payable_amount` 감소
 → 추가 환불 없음
  - 신규 `total_price` < `paid_amount_total` 인 경우
 → 초과 결제 금액 = `paid_amount_total` - 신규 `total_price`
 → 자동 부분 환불 실행 여부 선택
    - 즉시 부분 환불 실행
     - 환불 예약 상태로 전환
- 케이스 D: 전액 결제 완료 상태에서 금액 하향 조정
  - 초과 금액 발생
   - 초과 금액 = `paid_amount_total` - 신규 `total_price`
   - 부분 환불 시나리오 9로 자동 연계
   - `refund_type` = PARTIAL
7. **데이터 업데이트 처리**:
- `total_price` 업데이트
 - `remaining_payable_amount` 재계산
 - 필요 시 `refund_amount_total` 업데이트
 - 금액 변경 이력 기록
  - `event_type` = PRICE_UPDATED
   - old_`total_price`
   - new_`total_price`
   - `admin_id`
   - 변경 시각
- 주문 메모 자동 생성
  - `memo_type` = PRICE_CHANGE
   - 변경 전/후 금액 기록
   - 사유 코드 및 상세 사유 저장
8. **정산 연동 처리**:
- 정산 미확정 상태
  - 전문가 정산 예정 금액 재계산
   - 수수료 재산정
- 정산 확정 상태
  - 재정산 필요 플래그 설정
   - 관리자 승인 후 반영
9. **알림 처리**:
- 고객 알림
  - 금액 증가 시 추가 결제 요청 알림
   - 금액 감소 및 환불 시 환불 안내 알림
- 전문가 알림
  - 금액 변경이 정산 금액에 영향을 주는 경우 알림
10. **운영 리스크 통제 포인트**:
- 금액 수정은 환불/정산과 직결되므로 감사 로그 필수
 - 결제 링크가 존재하는 경우 반드시 기존 링크 무효화 처리 필요
 - 금액 변경 후 결제/환불 불일치가 발생하지 않도록 트랜잭션 처리
 - BI 통계에서 "원래 금액"과 "최종 금액"을 구분 저장 권장
 - 반복 금액 변경 주문은 이상 패턴으로 모니터링 필요

#### 시나리오 17: 수동 전문가 변경(운영 직접 교체)
본 시나리오는 자동 재배정이 아닌,  운영자가 특정 전문가를 지정하여 배정을 교체하는 경우를 정의한다.
본 기능은 일반 배정과 달리 "강제 배정" 성격을 가지며, 배정 정책을 우회할 수 있으므로 권한 통제가 필요하다.
1. **관리자 로그인**:
- "전문가 수동 변경 권한"을 보유한 계정으로 로그인한다
 - 권한이 없는 계정에는 "전문가 변경" 버튼이 노출되지 않는다
2. **주문 상세 화면 진입**:
- 주문관리 메뉴에서 대상 주문을 검색한다
 - 주문 상세 화면에 진입한다
 - 현재 배정 상태 확인
  - `order_status`
   - `expert_progress_status`
   - `assignment_delivery_status`
   - `expert_master_id`
   - `region_group`
   - `service_date` / `service_time`
3. **수동 변경 가능 여부 판단**:
- 변경 불가 상태
  - `order_status` = CANCELED
   - `order_status` = REFUNDED
   - `expert_progress_status` = CONFIRMED (서비스 종료)
   - 정산 확정 완료 상태
- 변경 가능 상태
  - ASSIGNED 상태
   - `assignment_delivery_status` = HOLD 또는 SENT
   - UNASSIGNED 상태에서도 직접 지정 가능
4. **전문가 변경 액션 실행**:
- "전문가 변경" 버튼 클릭
 - 전문가 선택 팝업 노출
  - 필터 조건
     - `region_group` 일치
     - 서비스 카테고리 수행 가능 여부
     - 활성 전문가만 표시
     - 멤버십 여부 표시
  - 일정 충돌 체크 기능 제공
     - 해당 날짜/시간 기존 예약 여부 표시
     - 충돌 시 경고 메시지 노출
5. **변경 사유 입력**:
- 사유 코드 선택 필수
  - CUSTOMER_REQUEST
   - EXPERT_REQUEST
   - SCHEDULE_CONFLICT
   - QUALITY_ISSUE
   - INTERNAL_EXCEPTION
- 상세 사유 메모 입력 필수
6. **상태 전환 처리 로직**:
- 기존 전문가 배정 해제
  - 기존 `expert_master_id` 저장
   - 기존 `assignment_delivery_status` 종료 처리
   - 기존 전문가 일정 이벤트 is_active = false
   - `order_assignment_history`에 type = MANUAL_UNASSIGN 기록
- 신규 전문가 배정
  - `order_status` = ASSIGNED 유지
   - `expert_master_id` = 신규 전문가 ID
   - `assignment_delivery_status` = SENT
   - `expert_progress_status` = NEW_ASSIGN
   - `order_assignment_history`에 type = MANUAL_ASSIGN 기록
   - 배정 시각 기록
7. **HOLD 상태 주문의 수동 변경**:
- `assignment_delivery_status` = HOLD 상태인 경우
  - 기존 HOLD 배정 전문가 제거
   - 신규 전문가 지정 시
     - 멤버십 전문가일 경우 HOLD 유지 가능
     - 일반 전문가일 경우 즉시 SENT 처리
  - HOLD 변경 이력 별도 기록
8. **캘린더 및 일정 처리**:
- 기존 전문가 일정 삭제 또는 비활성화
 - 신규 전문가 기준으로 신규 일정 이벤트 생성
 - 전문가웹/앱 캘린더에 즉시 반영
9. **알림 처리**:
- 기존 전문가에게 배정 해제 알림 발송
 - 신규 전문가에게 신규 배정 알림 발송
 - 고객에게 전문가 변경 안내 발송 여부는 정책 선택
10. **자동 배정 로직과의 관계**:
- 수동 변경은 자동 라운드로빈 포인터에 영향을 주지 않는다
 - 수동 변경된 주문은 재배정 자동 트리거 대상에서 제외 가능
 - 멤버십 우선 정책을 우회할 수 있으므로 변경 이력 필수 기록
11. **정산 연동**:
- 정산은 최종 수행 전문가 기준으로 계산
 - 변경 전 전문가에게는 정산 영향 없음
 - 이미 일부 서비스 진행된 경우 예외 처리 필요
12. **이력 및 감사 로그**:
- `event_type` = MANUAL_EXPERT_CHANGE 기록
 - 이전 전문가 ID
 - 신규 전문가 ID
 - 변경 사유 코드
 - `admin_id`
 - 변경 시각
- 주문 메모 자동 생성
   - `memo_type` = EXPERT_CHANGE

13. **운영 리스크 통제 포인트**:
- 수동 변경은 배정 통계에 왜곡을 줄 수 있음
 - 멤버십 우선 정책을 무력화할 수 있으므로 남용 방지 필요
 - 일정 충돌 무시 배정은 시스템에 명확히 표시 필요
### 관리자 개입 기능
1. **전문가 재배정**: 현재 전문가에서 다른 전문가로 변경 (동일 서비스 지역 필터)
2. **환불 처리**: 전체/부분 환불, 환불 사유 입력, Hecto 환불 API 호출
3. **주문 취소**: 고객/전문가 요청 시 강제 취소, 취소 수수료 정책 적용
4. **수동 상태 변경**: 특수 상황에서 상태 강제 변경 (활동 로그 기록)
5. **엑셀 업로드/다운로드**: 주문 데이터 일괄 등록 및 추출
6. **배치 작업**: 대량 주문 상태 변경, 메모 추가 등

## 정산 관리

### 정산 주기 및 프로세스
- **주간 정산**: 매주 월요일 00:00~23:59 동안 완료된 주문 대상
- **자동 정산 생성**: 매주 월요일 오전 2시 배치 작업 실행
- **정산 상태**: 대기(pending) → 승인(approved) → 지급완료(paid)
- **지급 방식**: 계좌이체 (전문가 등록 통장으로 자동 이체)

### 정산 목록 페이지 (Ant Design Table + 통계)
- **필터**: 상태(대기/승인/지급완료), 기간, 전문가명
- **통계 요약**: 총 정산액, 총 수수료, 총 순수익, 평균 정산액
- **일괄 작업**: 선택 정산 승인, 명세서 다운로드

### 정산 상세 페이지
1. **기본 정보**: 정산번호, 전문가, 정산 기간, 주문 수, 총 매출
2. **수익 계산**:
   - 총 매출: `₩1,500,000`
   - 플랫폼 수수료 (25.8%): `₩387,000`
   - VAT (10%): `₩150,000`
   - 순수익: `₩1,113,750`
3. **주문 내역**: 해당 정산에 포함된 주문 목록 (주문별 수익 상세)
4. **정산 명세서**: PDF 생성 및 다운로드 (React PDF 라이브러리)

### 정산 승인 프로세스
1. **재무 관리자 검토**: 정산 내역 확인, 이상 거래 검토
2. **승인 시**: 전문가에게 정산 명세서 알림 전송, 지급 준비 상태로 변경
3. **거절 시**: 거절 사유 입력, 전문가에게 알림, 정산 재계산 필요 시 주문 제외
4. **지급 완료**: 실제 계좌이체 후 상태 업데이트, 이체 증빙 파일 업로드

- **계정 생성**: 이메일, 이름, 역할 할당, 초기 비밀번호 설정
- **계정 수정**: 역할 변경, 활성화/비활성화, 비밀번호 재설정
- **계정 삭제**: 소프트 삭제 (데이터 보존)

### 권한 관리 인터페이스
- **역할 CRUD**: 새로운 역할 생성, 권한 설정
- **권한 그룹**: 미리 정의된 권한 그룹 (운영, CS, 재무) 적용
- **사용자별 권한**: 특정 관리자에게 개별 권한 추가/제거

### 시스템 설정 페이지
- **기본 설정**: 플랫폼 이름, 로고, 파비콘, 메타데이터
- **정산 설정**: 수수료율 (25.8%), VAT율 (10%), 정산 주기, 지급일
- **알림 설정**: 이메일/SMS 알림 템플릿, 자동 알림 규칙
- **외부 서비스 설정**: Hecto, Aligo, AWS S3 인증 정보 (암호화 저장)
- **캐시 설정**: Redis TTL, 캐시 전략

### 활동 로그 시스템 (Ant Design Table + 필터)
- **로그 유형**: 로그인, 주문 변경, 정산 승인, 설정 변경, 오류
- **필터링**: 사용자, 유형, 기간, IP 주소, 심각도
- **상세 보기**: 로그 상세 정보 (요청 데이터, 응답 데이터)
- **로그 보존**: 90일 보존 후 자동 아카이빙

## API 연동

### 인증 및 권한
- **로그인**: `POST /api/v1/auth/login/admin` (이메일/비밀번호 → JWT)
- **토큰 갱신**: `POST /api/v1/auth/refresh` (Refresh Token 사용)
- **권한 검증**: 모든 API 요청에 `Authorization: Bearer <token>` 헤더
- **역할 검증**: 백엔드에서 `role` 클레임 기반 접근 제어

### 대시보드 API
- `GET /api/v1/admin/dashboard/summary` - 주요 지표 요약
- `GET /api/v1/admin/dashboard/realtime-orders` - 실시간 주문 현황
- `GET /api/v1/admin/dashboard/recent-orders` - 최근 주문 목록
- `GET /api/v1/admin/dashboard/pending-approvals` - 승인 대기 항목
- `GET /api/v1/admin/dashboard/system-status` - 시스템 상태

### 사용자 관리 API
- `GET /api/v1/admin/users` - 사용자 목록 (필터, 검색, 페이징)
- `GET /api/v1/admin/users/:id` - 사용자 상세 정보
- `PUT /api/v1/admin/users/:id/status` - 사용자 상태 변경
- `POST /api/v1/admin/experts/:id/approve` - 전문가 승인
- `POST /api/v1/admin/experts/:id/reject` - 전문가 거절
- `GET /api/v1/admin/master-accounts/:id/team` - 마스터 계정 팀원 목록

### 주문 관리 API
- `GET /api/v1/admin/orders` - 주문 목록 (고급 필터)
- `GET /api/v1/admin/orders/:id` - 주문 상세 정보
- `PUT /api/v1/admin/orders/:id/status` - 주문 상태 변경
- `POST /api/v1/admin/orders/:id/reassign` - 전문가 재배정
- `POST /api/v1/admin/orders/:id/refund` - 환불 처리
- `GET /api/v1/admin/orders/:id/timeline` - 주문 타임라인
- `GET /api/v1/admin/orders/:id/chat-logs` - 채팅 내역

### 서비스 관리 API
- `GET /api/v1/admin/categories` - 카테고리 목록 (트리 구조)
- `POST /api/v1/admin/categories` - 카테고리 생성
- `PUT /api/v1/admin/categories/:id` - 카테고리 수정
- `DELETE /api/v1/admin/categories/:id` - 카테고리 삭제 (소프트)
- `GET /api/v1/admin/service-items` - 서비스 항목 목록
- `PUT /api/v1/admin/service-items/:id/price` - 서비스 가격 변경

### 정산 관리 API
- `GET /api/v1/admin/settlements` - 정산 목록
- `GET /api/v1/admin/settlements/:id` - 정산 상세
- `POST /api/v1/admin/settlements/:id/approve` - 정산 승인
- `POST /api/v1/admin/settlements/:id/reject` - 정산 거절
- `POST /api/v1/admin/settlements/:id/mark-paid` - 지급 완료 표시
- `GET /api/v1/admin/settlements/:id/statement` - 정산 명세서 PDF

### 통계 API
- `GET /api/v1/admin/statistics/sales` - 매출 통계 (기간별)
- `GET /api/v1/admin/statistics/orders` - 주문 통계
- `GET /api/v1/admin/statistics/users` - 사용자 통계
- `GET /api/v1/admin/statistics/top-services` - 인기 서비스 TOP 5
- `GET /api/v1/admin/statistics/export` - 리포트 데이터 내보내기

### 시스템 설정 API
- `GET /api/v1/admin/settings` - 시스템 설정 조회
- `PUT /api/v1/admin/settings` - 시스템 설정 업데이트
- `GET /api/v1/admin/admins` - 관리자 목록
- `POST /api/v1/admin/admins` - 관리자 생성
- `PUT /api/v1/admin/admins/:id` - 관리자 수정
- `GET /api/v1/admin/logs` - 활동 로그 조회

## 보안 및 권한

### 프론트엔드 보안
1. **JWT 저장**: `httpOnly` 쿠키에 저장 (XSS 방지)
2. **토큰 갱신**: 30분 만료, 자동 갱신 메커니즘
3. **CSRF 보호**: `X-CSRF-Token` 헤더 사용
4. **CSP 헤더**: Content Security Policy 적용
5. **XSS 방지**: 모든 사용자 입력 sanitization

### 백엔드 보안
1. **역할 검증**: 모든 관리자 API 엔드포인트에서 `role` 검증
2. **입력 검증**: TypeBox 스키마로 모든 입력값 검증
3. **SQL 인젝션 방지**: Prisma 사용으로 자동 방지
4. **Rate Limiting**: 관리자 API도 적절한 Rate Limit 적용
5. **감사 로그**: 모든 관리자 작업 로깅

### 데이터 접근 제어
1. **수직 권한**: 역할별 데이터 접근 범위 제한
2. **수평 권한**: 자신의 데이터만 접근 가능 (CS 관리자는 자신이 처리한 주문만)
3. **민감 데이터 마스킹**: 주민등록번호, 계좌번호 등 부분 마스킹
4. **데이터 내보내기 제한**: 대량 데이터 내보내기 권한 별도 관리
