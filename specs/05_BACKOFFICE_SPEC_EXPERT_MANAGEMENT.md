## 전문가 관리

### 개요
전문가 관리 모듈은 플랫폼에 서비스를 제공하는 전문가(마스터 계정 및 서브 계정)의 전 생애주기를 관리하는 기능을 제공합니다. 마스터 계정은 독립적인 사업자로서 팀을 구성할 수 있으며, 서브 계정은 마스터 계정 하위에서 실제 서비스를 수행하는 수행자 계정입니다.

**핵심 개념**:
- **마스터 계정**: 사업자 단위의 전문가 계정으로, 멤버십 구좌 설정, 서브 계정 관리, 팀 정산 통합 관리 기능 보유
- **서브 계정**: 마스터 계정 소속의 개별 수행자 계정으로, 주문 수행만 가능 (독립적 배정 불가)
- **멤버십 구좌**: 특정 서비스 중분류에서 우선 배정을 받을 수 있는 가중치 기반 할당 시스템
- **패널티 시스템**: 전문가의 품질 이슈(거절률, 타임아웃, 고객 클레임 등)에 따른 배정 제한 조치
- **일일 배정 상한**: 전문가의 과부하 방지를 위한 일일 최대 배정 건수 제한

**관련 데이터 모델**: [02_DATABASE_SCHEMA.md](../specs/02_DATABASE_SCHEMA.md)의 `master_accounts`, `sub_accounts`, `master_membership`, `penalty_history`, `assignment_history`, `master_assignment_policy`, `region_groups` 테이블 참조

### 전문가 목록 페이지 (Ant Design ProTable)
**탭 필터**:
- **전체**: 모든 전문가 (마스터 + 서브)
- **마스터 계정**: `account_type = MASTER` 필터
- **서브 계정**: `account_type = SUB` 필터
- **승인대기**: `approval_status = PENDING` 필터
- **활성 전문가**: `active_status = ACTIVE` 필터
- **비활성 전문가**: `active_status = INACTIVE` 필터
- **패널티 적용 중**: `penalty_status = ACTIVE` 필터
- **멤버십 활성**: `membership_enabled = true` 필터

**상태 필터**:
- **계정 상태**: 활성(ACTIVE), 비활성(INACTIVE)
- **승인 상태**: 승인대기(PENDING), 승인완료(APPROVED), 승인거절(REJECTED)
- **패널티 상태**: 정상(NONE), 소프트리밋(SOFT_LIMIT), 하드블록(HARD_BLOCK), 카테고리제한(CATEGORY_LIMIT), 권역제한(REGION_LIMIT), 멤버십정지(MEMBERSHIP_SUSPEND)
- **멤버십 상태**: 활성(ACTIVE), 정지(SUSPENDED), 비활성(INACTIVE), 만료(EXPIRED)

**검색 기능**:
- 전문가명/수행자명
- 연락처
- 이메일
- 로그인 ID
- 사업자명
- 사업자등록번호
- 마스터 계정 ID

**고급 필터**:
- 가입일 범위
- 서비스 가능 카테고리 (중분류)
- 서비스 가능 지역 (시/군/구)
- 지역 그룹 (region_group)
- 멤버십 구좌 수 범위
- 일일 배정 상한 범위

**일괄 작업**:
- 선택 전문가 활성화/비활성화
- 선택 전문가 승인 처리 (대기 → 승인)
- 선택 전문가 패널티 적용
- 선택 전문가 멤버십 활성화/비활성화
- 선택 전문가 내보내기 (CSV)

**액션 버튼**:
- **전문가 등록**: 마스터 계정 또는 서브 계정 등록 모달 오픈
- **상세 보기**: 전문가 상세 화면 진입 (탭 구조)
- **수정**: 기본 정보 수정 모달
- **배정 이력**: 해당 전문가의 배정 이력 조회
- **통계 요약**: 전문가 KPI 대시보드

### 전문가 등록 워크플로우

#### 시나리오 1: 전문가 마스터 계정 등록
**UI 컴포넌트**: Ant Design Modal, Form, Steps, Select, Input, Cascader (지역 선택)
**데이터 검증**: 클라이언트 측 Form Validation + 서버 측 중복 검증

**단계별 흐름**:
1. **가입 유형 선택**: 마스터 계정 가입 선택
2. **기본 정보 입력**:
   - 전문가명 (필수, 2~50자)
   - 연락처 (필수, 휴대폰 번호 형식, 중복 불가)
   - 이메일 (필수, 이메일 형식, 중복 불가)
   - 로그인 ID (필수, 4~20자 영문/숫자, 중복 불가)
   - 비밀번호 (필수, 8자 이상 영문/숫자/특수문자 조합)
   - 비밀번호 확인
3. **사업자 정보 입력**:
   - 사업자 유형 선택: 개인사업자 / 법인사업자
   - 사업자명 (필수)
   - 사업자등록번호 (필수, 10자리 숫자, 중복 불가)
   - 법인등록번호 (법인인 경우)
   - 사업장 주소: 주소 검색 API 연동
4. **서비스 수행 가능 정보 설정**:
   - 서비스 카테고리 선택: 메뉴 03 서비스관리 기준 활성 중분류 목록 (다중 선택 가능)
   - 서비스 가능 지역 선택: 시/도 → 시/군/구 Cascader (다중 선택 가능)
5. **시스템 자동 처리**:
   - 선택된 지역코드를 `region_group` 테이블 기준으로 자동 매핑
   - 초기 상태 설정: `approval_status = PENDING`, `active_status = INACTIVE`, `membership_enabled = false`, `membership_slot_count = 0`
   - `master_accounts` 테이블에 레코드 생성, `account_type = MASTER`
6. **배정 로직 연동**:
   - `approval_status = PENDING` 상태이므로 자동배정 후보군 제외
   - 수동배정 불가 상태
7. **전문가웹앱 연동**:
   - 로그인 허용
   - 주문 노출 없음
   - 승인 대기 안내 배너 노출
8. **감사 로그**: 생성 관리자 ID, 생성 시각, 생성 데이터 스냅샷 저장

**비즈니스 규칙**:
- 사업자등록번호는 시스템 내 유일해야 함 (중복 가입 방지)
- 서비스 카테고리와 서비스 가능 지역은 최소 1개 이상 선택해야 함
- 승인 대기 상태에서는 멤버십 설정 불가
- 마스터 계정 생성 후 24시간 내 승인 처리 권장

#### 시나리오 2: 전문가 서브 계정 등록
**UI 컴포넌트**: Ant Design Modal, Form, Select (마스터 계정 검색), Input
**데이터 검증**: 마스터 계정 존재 여부 확인, 중복 로그인 ID 검증

**단계별 흐름**:
1. **가입 유형 선택**: 서브 계정 가입 선택
2. **기본 정보 입력**:
   - 수행자명 (필수)
   - 연락처 (필수, 중복 불가)
   - 이메일 (필수, 중복 불가)
   - 로그인 ID (필수, 중복 불가)
   - 비밀번호 (필수)
   - 비밀번호 확인
3. **소속 마스터 계정 선택**:
   - 사업자명 또는 사업자등록번호로 마스터 계정 검색
   - `account_type = MASTER`이고 `active_status = ACTIVE`인 계정만 노출
   - 마스터 계정 선택
4. **시스템 검증**:
   - 선택한 마스터 계정 존재 여부 확인
   - 마스터 계정 삭제 상태 여부 확인
5. **초기 상태 설정**:
   - `approval_status = APPROVED` (서브 계정은 승인 자동 완료)
   - `active_status = ACTIVE`
   - `account_type = SUB`
   - `membership_enabled` 필드 없음 (마스터 계정 기준)
6. **시스템 저장 처리**:
   - `sub_accounts` 테이블 생성, `master_account_id` FK 저장
   - `assigned_worker_id` 후보군 등록
7. **배정 로직 연동**:
   - 자동배정 후보군에는 영향 없음 (배정 기준은 마스터 계정)
   - 주문 배정 시 `assigned_master_id`는 마스터 계정, 실제 수행자 지정 시 `assigned_worker_id`에 서브 계정 저장 가능
8. **전문가웹앱 연동**:
   - 로그인 허용
   - `assigned_worker_id`가 본인인 주문만 조회 가능
   - 팀 전체 주문 조회 권한 없음
9. **수행 가능 서비스/지역 제한**:
   - 서브 계정은 독립적 서비스 설정 불가
   - 마스터 계정의 `service_category_mid_available_list` 범위 내 수행
   - 마스터 계정의 `region_group` 범위 내 수행
10. **멤버십 구조 영향**:
    - 멤버십은 마스터 계정 기준 (서브 계정 수와 무관)
    - `membership_slot_count`와 무관
11. **마스터 계정 상태 영향**:
    - 마스터 계정이 `INACTIVE` 상태가 되면 해당 서브 계정 자동 배정 제외 (기존 수행 주문은 유지)
12. **감사 로그**: 생성 관리자 ID, 생성 시각, 소속 마스터 계정 ID 기록

**비즈니스 규칙**:
- 서브 계정은 승인 과정 없이 즉시 활성화
- 서브 계정의 서비스 범위는 상위 마스터 계정에 종속
- 마스터 계정 비활성화 시 소속 서브 계정도 배정 제외 (단, 기존 주문은 유지)

#### 시나리오 3: 전문가 승인 처리
**UI 컴포넌트**: 전문가 상세 화면 (탭 구조), 승인 액션 버튼, 상태 표시기
**데이터 검증**: 필수 정보 완성도 검증, 서비스 카테고리/지역 설정 여부 확인

**단계별 흐름**:
1. **승인대기 목록 필터**: `approval_status = PENDING`인 마스터 계정 목록 조회
2. **전문가 상세 화면 진입**: 기본 정보, 사업자 정보, 서비스 수행 가능 정보 검토
3. **필수 정보 검증**:
   - 사업자등록번호 중복 여부 재확인
   - 필수 입력값 누락 여부 확인
   - 서비스 카테고리 미설정 시 승인 불가
   - 서비스 지역 미설정 시 승인 불가
4. **승인 액션**: 승인 버튼 클릭 시 `approval_status: PENDING → APPROVED`, `active_status: INACTIVE → ACTIVE`
5. **자동배정 후보군 포함 처리**:
   - 조건 검증: `approval_status = APPROVED`, `active_status = ACTIVE`, 서비스 카테고리 일치, region_group 일치
   - 조건 충족 시 배정 로직 후보군에 포함
6. **멤버십 설정 가능 상태 전환**: `membership_enabled` 설정 가능, `membership_slot_count` 입력 가능
7. **전문가웹앱 연동**: 로그인 허용 유지, 신규 주문 배정 가능, 승인 대기 안내 배너 제거
8. **감사 로그**: 변경 전/후 approval_status, 변경 관리자 ID, 변경 시각 기록
9. **승인 알림 발송**: 전문가웹앱 이용 가능 안내, 배정 가능 상태 안내

**비즈니스 규칙**:
- 승인 전 필수 정보 검증 실패 시 승인 불가
- 승인 후 즉시 자동배정 후보군에 포함 (다음 배정 사이클부터 적용)
- 승인 알림은 이메일/문자/SNS 중 선택적 발송

#### 시나리오 4: 전문가 정보 수정
**UI 컴포넌트**: 전문가 상세 화면 수정 모달, 폼 필드 그룹화, 실시간 검증
**데이터 검증**: 중복 검증 (사업자등록번호, 연락처, 이메일), 멤버십 정합성 검증

**수정 가능 항목**:
- 기본 정보 (전문가명, 연락처, 이메일)
- 사업자 정보 (사업자명, 사업장 주소, 사업자 유형)
- 서비스 수행 가능 카테고리 (중분류 추가/제거)
- 서비스 가능 지역 (지역 추가/제거, region_group 자동 재매핑)
- 계약 정보 (계약 기간, 이행보증 정보)
- 내부 메모
- 멤버십 구좌 정보 (`membership_slot_count`, 적용 중분류)

**단계별 흐름**:
1. **수정 모달 오픈**: 현재 값으로 폼 프리필
2. **변경 사항 입력**: 각 필드 수정
3. **멤버십 정합성 검증**:
   - 멤버십 적용 중분류가 제거될 경우 경고
   - `membership_slot_count > 0` 상태에서 해당 중분류 제거 시 멤버십 자동 비활성 처리 또는 수정 불가
4. **저장 처리**: `master_accounts` 또는 `sub_accounts` 테이블 업데이트, 변경 전 데이터 스냅샷 저장, 변경 이력 테이블 기록
5. **배정 로직 연동 처리**:
   - 서비스 카테고리 변경 시 신규 주문부터 후보군 필터 재적용
   - 지역 변경 시 region_group 기준 후보군 재계산
   - 멤버십 구좌 수 변경 시 가중치 재반영
6. **기존 주문 보호 원칙**: 이미 배정된 주문에는 영향 없음, 진행 중 주문은 그대로 유지
7. **전문가웹앱 반영**: 프로필 정보 즉시 반영, 수행 가능 서비스 목록 즉시 반영, 신규 배정 범위 즉시 반영
8. **주문관리 연동**: 주문 상세 화면에 표시되는 전문가 정보는 당시 스냅샷 유지, 통계 집계는 최신 master_account_id 기준
9. **감사 로그**: 변경 전 값, 변경 후 값, 변경 관리자 ID, 변경 시각 기록

**비즈니스 규칙**:
- 사업자등록번호 변경 시 중복 검증 필수
- 서비스 카테고리 0개 상태 허용 여부 정책에 따라 검증 (기본값: 허용 안함)
- 지역 0개 상태 허용 여부 정책에 따라 검증 (기본값: 허용 안함)

#### 시나리오 5: 전문가 활성/비활성 전환
**UI 컴포넌트**: 상태 변경 액션 버튼, 사유 입력 팝업, 진행 중 주문 경고 표시
**데이터 검증**: 진행 중 주문 존재 여부 확인, 멤버십 상태 확인

**비활성화 사유 코드**:
- `TEMPORARY_STOP`: 일시 휴업
- `QUALITY_ISSUE`: 품질 이슈/클레임 누적
- `CONTRACT_END`: 계약 종료
- `POLICY_VIOLATION`: 내부 정책 위반
- `INACTIVE_BY_SYSTEM`: 장기 미사용/장기 미배정

**단계별 흐름**:
1. **대상 전문가 검색**: 전문가명, 연락처, 사업자명, 사업자번호 등으로 검색, 마스터/서브 여부 필터, active_status 기준 필터
2. **비활성화 액션**: 비활성화 버튼 클릭 → 사유 입력 팝업 노출 (사유코드 선택, 상세 사유 메모 필수)
3. **시스템 사전 검증**:
   - `approval_status` 확인 (PENDING 상태에서 비활성화는 "가입 반려" 성격)
   - 현재 진행 중 주문 존재 여부 확인 (`ASSIGNED/IN_SERVICE/RESERVED` 상태 주문 건수 조회)
   - `membership_enabled = true`인 경우 "멤버십 혜택도 즉시 중단" 안내
4. **비활성화 확정**: 관리자 확인 후 처리
5. **시스템 상태 변경 처리 – 마스터 계정**:
   - `active_status: ACTIVE → INACTIVE`
   - 자동배정 후보군에서 해당 master_account 제거
   - 수동배정 대상 목록에서도 제외
   - `membership_enabled = true`인 경우 `membership_status = SUSPENDED` 또는 `INACTIVE`로 변경, `membership_slot_count` 값 보관
6. **시스템 상태 변경 처리 – 서브 계정**:
   - 해당 마스터 계정 소속 서브 계정 전체 조회
   - 서브 계정 `active_status`는 유지 (계정 자체는 살아있음)
   - 마스터 계정 `INACTIVE` 상태면 서브 계정만 단독으로 배정 대상이 될 수 없음 (`assigned_worker_id`로만 사용 가능)
7. **진행 중 주문 처리 정책**:
   - 이미 `ASSIGNED/IN_SERVICE` 상태인 주문: 기존 `assigned_master_id`, `assigned_worker_id` 유지
   - 새로운 주문/배정: 마스터 계정 `INACTIVE` 이후 생성되는 주문은 해당 전문가 자동/수동 배정 모두 불가
   - `HOLD` 상태 멤버십 리드: 운영자가 수동으로 재배정/취소/다른 멤버십 전문가로 변경 필요
8. **전문가웹앱 반영**:
   - 마스터 계정: 로그인 허용 (정책 선택 가능), "운영 중단 상태" 배너 노출, 새로운 배정 알림 없음
   - 서브 계정: 로그인 허용, 배정받을 새 주문 없음, 기존 `assigned_worker_id` 기준 진행 주문만 보임
9. **멤버십/배정 통계 연동**:
   - 멤버십 가중치에서 해당 마스터 계정 제외
   - 멤버십 리드 점유율 통계에서 "비활성 기간" 구간 분리 가능하도록 플래그 기록
   - 배정 성과 대시보드에서 "비활성 전문가"로 표시
10. **감사 로그**: before_active_status, after_active_status, reason_code, reason_detail, 변경자 admin_id, 변경 일시 기록
11. **재활성화 시나리오**:
    - `approval_status`가 `APPROVED`인지 재검증
    - 계약 기간이 유효한지 확인
    - region_group, service_category_mid 설정이 남아 있는지 확인
    - `membership_enabled = true`였으면 정책에 따라 membership_status 재활성 또는 재가입 요구

**비즈니스 규칙**:
- 진행 중 주문이 존재해도 비활성화 가능 (단, 신규 배정만 중단)
- 멤버십 활성 상태에서 비활성화 시 멤버십 정지 처리
- 재활성화 시 기존 설정 복원 가능

#### 시나리오 6: 전문가 서비스 가능 카테고리 수정
**UI 컴포넌트**: 서비스 카테고리 관리 영역, 추가/제거 버튼, 멤버십 연동 표시
**데이터 검증**: 중복 추가 여부, 비활성 카테고리 선택 불가, 지역 설정 존재 여부 확인

**단계별 흐름**:
1. **현재 설정 조회**: `service_category_mid_available_list` 조회, 하위 소분류 자동 포함 구조 확인
2. **서비스 카테고리 추가**:
   - "카테고리 추가" 버튼 클릭 → 중분류 목록 표시 (메뉴 03 서비스관리 기준 활성 카테고리만 노출, membership_available 여부 표시)
   - 중분류 선택 → 저장 클릭
   - 시스템 검증: 중복 추가 여부 확인, 해당 중분류 비활성 상태 확인, 지역 설정 존재 여부 확인 (지역이 하나도 설정되어 있지 않으면 경고)
   - 저장 처리: `service_category_mid_available_list`에 중분류 ID 추가, 배정 로직 후보군 재계산, 멤버십 연동 (새 중분류가 membership 적용 대상이면 membership_slot_count 가중치 자동 적용), 감사 로그 기록
3. **서비스 카테고리 제거**:
   - 제거할 중분류 선택 → "카테고리 제거" 버튼 클릭
   - 시스템 사전 검증: 진행 중 주문 존재 여부 확인 (해당 중분류 + assigned_master_id 기준 ASSIGNED/RESERVED/IN_SERVICE 상태 주문 건수 조회), 멤버십 적용 여부 확인
   - 제거 정책 분기:
     - 진행 중 주문 존재하는 경우: 제거 허용, 기존 주문 유지, 신규 배정만 중단
     - 멤버십 적용 중인 중분류인 경우: membership 적용 중분류 목록에서 자동 제거 또는 제거 불가 정책 선택
   - 저장 처리: `service_category_mid_available_list`에서 해당 ID 제거, 배정 로직 후보군 재계산 (해당 중분류에서는 신규 자동배정 대상 제외), membership 중분류 목록에서 자동 제외, 감사 로그 기록
4. **배정 로직 영향**:
   - 제거 이후 생성되는 신규 주문: 자동배정 후보군에서 제외, 수동배정 대상 목록에서도 제외
   - 기존 배정 주문: assigned_master_id 유지, 강제 재배정 없음
5. **멤버십 가중치 영향**:
   - `membership_slot_count` 자체는 유지
   - 해당 중분류에서의 우선 배정 가중치는 0으로 처리
   - 다른 중분류에는 정상 적용
6. **전문가웹앱 반영**: 프로필에서 수행 가능 서비스 목록 즉시 반영, 신규 배정 주문만 노출 대상, 기존 주문은 정상 표시
7. **통계 및 리포트 반영**: 과거 주문은 기존 카테고리 기준 유지, 신규 배정 건수 집계는 변경 이후 기준 반영
8. **엣지 케이스**:
   - 모든 중분류 제거 시: active_status 자동 INACTIVE 전환 여부 정책 결정 또는 배정 불가 상태 경고 표시
   - `membership_enabled = true`인 상태에서 수행 가능 중분류가 0개가 되면 membership_status 자동 INACTIVE 처리

**비즈니스 규칙**:
- 서비스 카테고리 추가 시 반드시 서비스 가능 지역이 설정되어 있어야 함
- 멤버십 적용 중인 카테고리 제거 시 멤버십에서 자동 제외
- 모든 카테고리 제거 시 전문가 비활성화 권장

#### 시나리오 7: 전문가 서비스 가능 권역 변경
**UI 컴포넌트**: 서비스 가능 지역 관리 영역, 지역 추가/제거 버튼, region_group 매핑 표시
**데이터 검증**: 진행 중 주문 존재 여부, HOLD 상태 주문 존재 여부, region_group 내 다른 지역 존재 여부 확인

**단계별 흐름**:
1. **현재 설정 조회**: `service_region_list` 조회, 현재 region_group 자동 매핑 결과 확인
2. **권역 추가**:
   - "지역 추가" 버튼 클릭 → 시/도 선택 → 시/군/구 선택 → 저장 클릭
   - 시스템 처리: `service_region_list`에 지역코드 추가, region_group 자동 재매핑 (region_group 테이블 기준), 배정 로직 후보군 재계산 (해당 중분류 + 신규 region_group 조합에 포함), 멤버십 연동 (해당 중분류 + 신규 region_group에도 가중치 적용), 감사 로그 기록
3. **권역 제거**:
   - 제거할 시/군/구 선택 → "지역 제거" 버튼 클릭
   - 시스템 사전 검증: 진행 중 주문 존재 여부 확인 (해당 지역 + assigned_master_id 기준 ASSIGNED/RESERVED/IN_SERVICE 상태 주문 조회), HOLD 상태 주문 존재 여부 확인, 해당 region_group 내 다른 지역이 남아 있는지 확인
   - 제거 정책 분기:
     - 진행 중 주문 존재: 제거 허용, 기존 주문 유지, 신규 배정만 차단
     - HOLD 상태 멤버십 주문 존재: 운영자에게 경고 표시, 재배정 또는 취소 유도
   - 저장 처리: `service_region_list`에서 해당 지역코드 제거, region_group 재계산 (동일 그룹 내 다른 지역이 남아 있으면 group 유지, 해당 그룹에 속한 지역이 0개면 group 제거), 배정 로직 후보군 재계산 (해당 중분류 + region_group 조합에서 제외), 감사 로그 기록
4. **자동배정 로직 영향**:
   - 제거 이후 생성되는 신규 주문: 해당 지역 주문은 자동배정 대상에서 제외, 수동배정 대상 목록에서도 제외
   - 기존 배정 주문은 영향 없음
5. **멤버십 가중치 영향**:
   - `membership_slot_count` 자체는 유지
   - 제거된 region_group에서는 가중치 적용 불가
   - 다른 region_group에서는 정상 적용
6. **전문가웹앱 반영**: 신규 배정 주문 중 해당 지역 주문은 수신 불가, 기존 주문은 정상 표시, 프로필 내 서비스 가능 지역 즉시 반영
7. **엣지 케이스**:
   - 모든 지역 제거 시: 자동배정 후보군 완전 제외, active_status 자동 INACTIVE 전환 여부 정책 결정
   - 멤버십 활성 상태에서 region_group이 0개가 되면 membership_status 자동 SUSPENDED 처리 권장

**비즈니스 규칙**:
- 지역 추가 시 region_group 자동 매핑 (수도권/충청/영남/호남/제주)
- 지역 제거 시 region_group 재계산 (그룹 내 지역 0개면 그룹 제거)
- 모든 지역 제거 시 전문가 비활성화 권장

#### 시나리오 8: 멤버십 구좌 설정 및 변경
**UI 컴포넌트**: Ant Design Tabs (멤버십 관리 탭), Form, Select (중분류 선택), InputNumber (slot_count), DatePicker (적용 기간), Modal (비활성화 확인)
**데이터 검증**: 멤버십 활성화 조건 검증 (approval_status, active_status, 서비스 카테고리/지역 존재), slot_count 범위 검증, 중분류 선택 유효성

**단계별 흐름**:
1. **멤버십 관리 탭 진입**: 현재 멤버십 상태 확인 (membership_enabled, membership_status, membership_slot_count, 적용 중분류 목록, 적용 시작일/종료일)
2. **멤버십 신규 설정**:
   - "멤버십 활성화" 버튼 클릭
   - 적용 중분류 선택: 메뉴 03 서비스관리 기준 membership_available = true인 중분류만 노출, 마스터 계정 service_category_mid_available_list와 교집합만 허용
   - membership_slot_count 입력: 최소값 1, 최대값 정책 제한, 정수만 허용
   - 적용 시작일 입력
   - 저장 버튼 클릭
3. **시스템 사전 검증**: approval_status = APPROVED 확인, active_status = ACTIVE 확인, 수행 가능 중분류 존재 여부 확인, 수행 가능 지역 존재 여부 확인
4. **저장 처리**: master_membership 테이블 생성, membership_enabled = true, membership_status = ACTIVE, membership_slot_count 저장, membership_mid_list 저장, 변경 이력 로그 기록
5. **배정 로직 반영**:
   - 중분류 + region_group 단위로 적용
   - 가중치 계산식: 일반 전문가 기본 가중치 = 1, 멤버십 전문가 가중치 = membership_slot_count
   - 라운드로빈 포인터 구조에 가중치 반영
6. **신규 주문 발생 시 흐름**:
   - 자동배정 실행
   - 가중치 기반 마스터 계정 선정
   - 멤버십 대상 주문인 경우 assignment_delivery_status = HOLD, 전문가웹앱 미노출
   - 운영팀 1차 상담 후 SENT 또는 재배정
7. **멤버십 구좌 변경**:
   - membership_slot_count 수정: 기존 값 조회, 신규 값 입력, 저장 클릭
   - 시스템 처리: membership_slot_count 업데이트, 신규 배정부터 가중치 반영, 기존 HOLD 또는 SENT 주문 영향 없음
8. **적용 중분류 변경**:
   - membership_mid_list 수정: 중분류 추가/제거
   - 제거 시 정책: 해당 중분류에서의 가중치 즉시 제거, 기존 배정 주문 유지, HOLD 상태 주문 운영자 재검토 필요
9. **멤버십 비활성화**:
   - "멤버십 비활성화" 버튼 클릭
   - 사유 입력
   - 시스템 처리: membership_enabled = false, membership_status = INACTIVE, 신규 배정부터 가중치 제거, 기존 주문 영향 없음

**비즈니스 규칙**:
- 멤버십은 master_account 단위로 관리, 서브 계정 수와 무관
- 가중치는 중분류 + region_group 조합에서만 적용
- 자동배정 선행 후 HOLD 정책 유지 (운영팀 검증 후 SENT)
- 기존 주문은 항상 보호 (멤버십 변경 시 기존 배정 영향 없음)
- 예외 케이스:
  - 수행 가능 중분류 0개 상태에서 멤버십 활성 불가
  - service_region_list 0개 상태에서 멤버십 활성 불가
  - master_account.active_status = INACTIVE 인 경우 membership_status 자동 SUSPENDED
  - 계약 종료일 경과 시 membership_status 자동 만료 처리 가능
- 통계 및 로그: 멤버십 배정 건수 집계 기준은 master_account_id 기준, membership_slot_count 기준 가중치 반영
- 변경 이력 로그 저장: before_slot_count, after_slot_count, 변경 관리자 ID, 변경 시각
- v2/v3 확장 포인트:
  - 전문가 일일 배정 상한: daily_assignment_limit 필드 추가, 상한 초과 시 후보군 제외
  - 품질 지표 연동: 응답속도, 수락률 등 지표 기반 가중치 조정 (0.5~1.5 범위)
  - 슬롯 상한 설계: membership_slot_count가 N이더라도 슬롯 수 = min(N, max_slot_limit) 형태 조정 가능
  - 포인터 스코프 확장: region_group 외 시/도, 시/군/구 단위 확장 가능 설계
  - 배정 실패 분석: 필터 단계별 탈락 사유 집계 및 진단 화면 설계

참고: 배정 로직 상세 사용자 시나리오는 `/docs/menu10_assignment_logic_spec_ko.md` 참조

---#### 시나리오 9: 전문가 배정 이력 조회
**UI 컴포넌트**: Ant Design Tabs (배정 이력 탭), DatePicker (기간 선택), Select (다중 필터), ProTable (이력 리스트), Modal (상세 이력 팝업), Statistic (통계 카드)
**데이터 검증**: 조회 기간 유효성, 필터 조건 조합 검증

**단계별 흐름**:
1. **배정 이력 탭 진입**: 기본 조회 조건 설정 (기간 선택, 서비스 중분류 필터, region_group 필터, 채널 필터, 배정 유형 필터 [AUTO_ASSIGN, MANUAL_ASSIGN, REASSIGN], 멤버십 배정 여부 필터 [is_membership_assignment = true/false])
2. **조회 버튼 클릭**: 시스템 조회 처리 (assignment_history 테이블 조회, 조건: assigned_master_id = 선택한 master_account_id, 배정 시점 기준, 페이징 처리)
3. **이력 리스트 표시 항목**:
   - 배정 일시
   - 주문번호
   - 서비스 중분류
   - 고객 지역
   - region_group
   - 배정 유형
   - 멤버십 배정 여부
   - 배정 당시 membership_slot_count
   - 배정 당시 가중치 값
   - 배정 결과 상태 (ACCEPTED, REJECTED, TIMEOUT, HOLD, SENT)
4. **상세 이력 조회**: 특정 주문 클릭 시 배정 상세 이력 팝업 표시
   - 배정 시점 order_status
   - 배정 시점 expert_progress_status
   - 배정 시점 assignment_delivery_status
   - 배정 로직 판단 근거 (후보군 총 인원 수, 해당 전문가 가중치, 포인터 위치)
   - 이전 배정 전문가 이력 존재 시 표시
5. **멤버십 관련 조회**:
   - 멤버십 배정 통계 영역 표시 (총 배정 건수, 멤버십 배정 건수, 일반 배정 건수, 멤버십 배정 비율, 중분류별 멤버십 배정 건수)
   - membership_slot_count 변경 이력 연동 (특정 기간 동안 slot 변경 여부 표시, slot 변경 전/후 배정 비율 비교 가능)
6. **HOLD 흐름 확인**:
   - HOLD 상태 배정 이력 필터 (assignment_delivery_status = HOLD, 운영팀 전달 여부 표시, SENT 전환 여부 표시)
   - HOLD → 재배정 케이스 확인 (재배정 횟수 표시, 재배정 사유 표시)
7. **성과 분석 요소**:
   - 거절률 계산 (REJECTED 건수 / 총 배정 건수, 중분류별 거절률)
   - 타임아웃률 계산 (TIMEOUT 건수 / 총 배정 건수)
   - 평균 응답 시간 계산 (배정 시점 → 수락 시점 차이)

**비즈니스 규칙**:
- 이력은 "배정 당시 스냅샷 기준"으로 조회 (현재 service_category_mid_available_list 변경과 무관, 현재 region_group 변경과 무관, membership_slot_count 변경과 무관)
- 과거 이력은 절대 수정 불가
- 예외 케이스:
  - 마스터 계정이 INACTIVE 상태라도 이력 조회 가능
  - 멤버십이 현재 INACTIVE라도 과거 이력은 유지
  - 삭제된 서비스 카테고리도 과거 이력에 표시
- 관리자 입장에서:
  - 해당 전문가의 배정 패턴 확인 가능
  - 멤버십 가중치 효과 검증 가능
  - 품질 문제(거절률/타임아웃률) 판단 가능
  - 패널티 적용 근거 확보 가능

---#### 시나리오 10: 전문가 패널티 적용
**UI 컴포넌트**: Ant Design Tabs (패널티 관리 탭), Select (패널티 유형, 적용 범위, 사유 코드), DatePicker (기간 설정), Input.TextArea (상세 사유), Modal (패널티 적용 확인), ProTable (패널티 이력)
**데이터 검증**: 패널티 유형별 적용 범위 검증, 기간 겹침 검증, 중복 패널티 검증, 멤버십 정합성 검증

**단계별 흐름**:
1. **패널티 관리 탭 진입**: 현재 패널티 상태 확인 (penalty_status, penalty_type, penalty_start_date, penalty_end_date, 적용 범위, 사유 코드)
2. **신규 패널티 적용**:
   - "패널티 적용" 버튼 클릭
   - 패널티 유형 선택:
     - SOFT_LIMIT: 가중치 감소
     - HARD_BLOCK: 자동배정 완전 제외
     - CATEGORY_LIMIT: 특정 중분류 배정 제외
     - REGION_LIMIT: 특정 region_group 배정 제외
     - MEMBERSHIP_SUSPEND: 멤버십 가중치 무효화
   - 적용 범위 설정: 전체 서비스, 특정 중분류 선택, 특정 region_group 선택
   - 패널티 기간 설정: 시작일 입력, 종료일 입력 (종료일 미입력 시 무기한 처리)
   - 사유 코드 선택: 거절률 초과, 타임아웃 과다, 고객 클레임 누적, 정책 위반, 수동 지정
   - 상세 사유 메모 입력
   - 저장 버튼 클릭
3. **시스템 검증**: approval_status 확인, active_status 확인, 중복 패널티 여부 확인, 기존 패널티와 기간 겹침 여부 확인, 멤버십 정합성 검증 (MEMBERSHIP_SUSPEND 선택 시 membership_enabled = true 여부 확인)
4. **저장 및 배정 로직 반영**:
   - penalty_history 테이블 생성, penalty_status = ACTIVE 설정
   - 배정 로직 후보군 재계산
   - 유형별 처리:
     - SOFT_LIMIT: 기본 가중치 = 1, membership_slot_count 무시 또는 절반 적용
     - HARD_BLOCK: 자동배정 후보군 완전 제외, 수동배정 가능 여부 정책 선택
     - CATEGORY_LIMIT: 특정 중분류에서만 제외
     - REGION_LIMIT: 특정 region_group에서만 제외
     - MEMBERSHIP_SUSPEND: membership_slot_count 가중치 0 처리, 일반 전문가와 동일 가중치 적용
5. **기존 주문 보호**: 이미 배정된 주문 영향 없음, 신규 배정부터 적용
6. **전문가웹앱 반영**:
   - HARD_BLOCK 상태: 신규 배정 없음, 기존 주문만 노출
   - MEMBERSHIP_SUSPEND 상태: 멤버십 배지 비활성 표시
   - 패널티 배너 노출 정책 선택 가능
7. **자동 만료 처리**: 종료일 도래 시 penalty_status = EXPIRED, 자동배정 후보군 재포함, 멤버십 가중치 복구, 감사 로그 기록
8. **패널티 이력 조회**: 과거 패널티 이력 목록 조회 가능, 각 패널티별 적용 기간 표시, 적용 당시 거절률/타임아웃률 지표 스냅샷 보관

**비즈니스 규칙**:
- 패널티는 active_status와 별도 레이어 (HARD_BLOCK은 active_status를 변경하지 않음)
- 멤버십 가중치와 패널티 가중치는 분리 계산
- 패널티는 항상 기간 기반 (무기한 가능)
- 과거 이력은 절대 수정 불가
- 패널티 적용 근거는 배정 이력 및 품질 지표와 연동
- 패널티는 배정 품질 관리 체계 완성, 멤버십 악용 방지, 자동배정 통제 가능, 운영 리스크 최소화 목적

---#### 시나리오 11: 전문가 일일 배정 상한 설정
**UI 컴포넌트**: Ant Design Tabs (배정 설정 탭), InputNumber (daily_assignment_limit), DatePicker (적용 시작일), Statistic (오늘 배정된 건수 실시간 표시), Modal (설정 확인)
**데이터 검증**: limit 값 범위 검증 (0 이상 정수), approval_status 및 active_status 검증, 수행 가능 서비스 존재 여부 확인

**단계별 흐름**:
1. **배정 설정 탭 진입**: 현재 배정 정책 조회 (daily_assignment_limit 값, 적용 여부 활성 플래그, 오늘 배정된 건수 실시간 표시)
2. **일일 배정 상한 설정**:
   - "일일 배정 상한 설정" 버튼 클릭
   - daily_assignment_limit 입력: 0 입력 시 무제한, 1 이상 입력 시 상한 적용, 정수만 허용
   - 적용 시작일 설정
   - 저장 버튼 클릭
3. **시스템 검증 및 저장**:
   - approval_status = APPROVED 확인
   - active_status = ACTIVE 확인
   - 수행 가능 서비스 존재 여부 확인
   - 저장 처리: master_assignment_policy 테이블에 limit 값 저장, 변경 이력 기록
4. **배정 로직 반영**:
   - 자동배정 실행 시 필터 단계 추가:
     - 기존 필터: 승인 상태, 활성 상태, 서비스 중분류 일치, region_group 일치, 패널티 여부
     - 신규 필터: today_assigned_count < daily_assignment_limit
   - 상한 초과 시 처리: 자동배정 후보군에서 제외, 멤버십 가중치도 무시, 다음 후보자로 포인터 이동
5. **배정 건수 계산 기준**:
   - today_assigned_count 계산 기준 정의: 배정 시점 기준, order_status가 ASSIGNED 이상인 건, 취소된 주문 제외 여부 정책 정의
6. **멤버십과의 정합성**:
   - membership_slot_count가 높아도 상한 초과 시 배정 불가
   - 멤버십은 후보군 내 우선순위일 뿐, 상한은 최종 필터
7. **기존 주문 보호**: 이미 배정된 주문 영향 없음, 신규 배정부터 제한 적용

**비즈니스 규칙**:
- 일일 상한은 배정 후보군 필터의 최종 단계에서 적용 (다른 모든 조건 통과 후 체크)
- 목적: 전문가 과부하 방지 및 서비스 품질 유지, 멤버십 가중치가 있더라도 상한 초과 시 자동배정 제외
- 과부하 방지, 멤버십 남용 방지, 품질 안정화
- daily_assignment_limit = 0 은 무제한으로 처리
- 상한 초과 시 멤버십 가중치도 무시 (공정성 보장)
- 배정 건수는 매일 00:00 기준 초기화

---#### 시나리오 12: 전문가 통계 요약 대시보드
**UI 컴포넌트**: Ant Design Tabs (통계 요약 탭), DatePicker (기간 선택), Statistic (KPI 카드), Table (중분류별 데이터), LineChart (추이 그래프), Card (패널티 상태)
**데이터 검증**: 기간 선택 유효성, 데이터 범위 검증

**단계별 흐름**:
1. **통계 요약 탭 진입**: 기본 조회 조건 설정 (기간 선택: 오늘, 최근 7일, 최근 30일, 사용자 지정)
2. **주요 KPI 표시**:
   - 배정 지표: 총 배정 건수, 멤버십 배정 건수, 일반 배정 건수, 중분류별 배정 건수
   - 응답 품질 지표: 평균 응답 시간, 거절률, 타임아웃률
   - 완료 품질 지표: 구매확정 전환율, A/S 발생률, 취소율
3. **멤버십 효과 분석**:
   - membership_slot_count 표시
   - 멤버십 가중치 적용 전후 비교 지표
   - 중분류별 멤버십 배정 비율
   - 동일 region_group 내 평균 대비 배정 비율 비교
4. **일일 상한 연동**:
   - 최근 7일 일일 배정 건수 그래프
   - daily_assignment_limit 초과 발생 여부 표시
   - 상한 도달 후 미배정 기회 건수 추정
5. **패널티 연동**:
   - 현재 패널티 상태 표시
   - 패널티 적용 전후 배정률 비교
   - 거절률 추이 그래프
6. **데이터 기준 설명**:
   - 모든 지표는 "배정 당시 스냅샷 기준"
   - 멤버십 가중치는 당시 slot_count 기준
   - 카테고리/권역 변경과 무관

**비즈니스 규칙**:
- 통계 데이터는 실시간 집계 (배정 이력 기반)
- 기간 선택 시 배정 시점 기준 필터링
- 멤버십 효과 분석은 membership_slot_count 변경 이력을 고려
- 패널티 적용 근거 확보를 위해 패널티 적용 전후 지표 비교 제공
- 일일 상한 정책 효과 분석 가능하도록 상한 도달 패턴 시각화
- 관리자 입장에서:
  - 전문가 품질 수준 한눈에 파악
  - 멤버십 효과 검증 가능
  - 패널티 적용 근거 확보
  - 배정 상한 정책 효과 분석 가능
