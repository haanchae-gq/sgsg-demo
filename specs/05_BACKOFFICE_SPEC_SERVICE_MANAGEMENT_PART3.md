### 관리 기능

#### 시나리오 11: 주문 메모 및 이력 기록
1. **관리자 로그인 및 주문관리 메뉴 진입**:
   - 관리자 계정으로 로그인
   - 백오피스 주문관리 메뉴 진입

2. **메모/이력 확인이 필요한 주문 검색**:
   - 검색 조건 설정:
     - 주문번호
     - 고객 전화번호
     - `order_status` (예: 문제 주문이면 `CANCELED`, `REFUNDED` 등)
     - 최근 수정일/상담일 기준 정렬
   - 검색 결과에서 특정 주문 선택

3. **주문 상세 화면 진입 및 정보 확인**:
   - 기본 탭에서 주문 정보 확인:
     - 상태 정보: `order_status`, `expert_progress_status`, `assignment_delivery_status`
     - 금액 정보: `total_price`, `paid_amount_total`, `refund_amount_total`
     - 배정 정보: `expert_master_id`, `region_group` 등
   - "메모/이력" 영역 또는 탭으로 이동

4. **기존 메모 및 이력 조회**:
   - 메모 목록 확인:
     - 메모 작성 시각
     - 작성자(`admin_id` 또는 이름)
     - 메모 타입(예: 상담, 이슈, 시스템, 내부 공유 등)
     - 메모 내용(읽기 전용)
   - 상태 이력 로그 확인:
     - 상태 변경 이력: 이전 상태 → 이후 상태, 변경 시각, 변경자
     - 배정 이력: 배정/재배정/거절/타임아웃 기록
     - 결제/환불 이력: 결제/환불 금액, 사유, 처리자

5. **신규 메모 작성 필요 여부 판단**:
   - 고객과 새 통화/문자 상담이 발생한 경우
   - 주문 정보 수동 수정(예외 처리, 강제 상태 변경 등)을 수행한 경우
   - 전문가와 별도 조율(시간 변경, 현장 특이사항 등)이 발생한 경우
   - 환불/부분 환불/보상 처리 근거를 남길 필요가 있는 경우

6. **신규 메모 작성**:
   - "메모 추가" 버튼 클릭
   - 메모 타입 선택:
     - `CUSTOMER_CALL`: 고객 상담
     - `EXPERT_CALL`: 전문가 상담
     - `ISSUE`: 클레임/이슈 처리
     - `ADJUSTMENT`: 금액/일정 조정
     - `INTERNAL_NOTE`: 내부 공유용 메모
   - 메모 내용 입력:
     - 통화 요약(누가, 언제, 무엇을 요청/설명했는지)
     - 운영 판단(어떤 정책을 적용했는지, 예외 허용 여부)
     - 후속 액션(재연락 필요, 재방문 필요, 추후 확인 일자 등)
   - 메모 저장 클릭

7. **메모 저장 시 시스템 처리**:
   - 메모 테이블에 신규 row 생성:
     - `order_id`, `admin_id`, `memo_type`, `memo_content`
     - `created_at`, `updated_at`
   - 주문 이력 로그에 "메모 추가" 이벤트 기록:
     - 이벤트 타입: `NOTE_CREATED`
     - 메모 ID 참조
   - 메모는 작성 후 수정 여부 정책에 따라 처리:
     - 권장: 내용은 수정 가능하되, 수정 이력 별도 기록
     - 삭제는 제한(슈퍼관리자만 가능 또는 완전 금지)

8. **예외 처리 또는 운영 개입과 메모 연동**:
   - 주문 상태를 수동으로 변경한 경우(예: 강제 `ASSIGNED`, 강제 `CANCELED` 등):
     - 상태 변경 액션 직후 "사유 메모 작성" 팝업 노출
     - 메모 없이 상태 강제 변경이 불가능하도록 정책 설정 가능
   - 특별 환불/보상 처리 시 메모:
     - 일반 정책과 다른 환불율·보상금을 적용한 경우
     - 왜 예외를 허용했는지 상세 기록
   - 반복 이슈 고객/전문가에 대한 패턴 기록:
     - 특정 고객이 반복 클레임을 제기하는 경우
     - 특정 전문가가 반복적으로 일정 변경/노쇼 등 발생하는 경우
     - 메모를 통해 이후 결정(블랙리스트, 패널티 등)의 근거로 활용

9. **주문 이력 기반 회귀 분석/모니터링**:
   - 운영팀장이 이슈 주문을 되돌아보며 메모 및 이력을 분석:
     - 클레임 많은 주문 패턴 파악
     - 특정 채널/카테고리에서 반복되는 문제 파악
   - QA/CS 교육용 사례 발굴:
     - 잘 처리된 케이스, 잘못 처리된 케이스를 메모 이력 기반으로 리뷰
   - 정책 개선 근거로 활용:
     - 취소/환불 정책, 예약금 정책, 배정 정책 개선 시 메모 데이터 활용

**운영 리스크 및 정책 포인트**:
- 메모는 "나중에 법적/분쟁 대응"의 근거가 될 수 있으므로, 감정적인 표현/비속어/개인 감정 기록 금지 가이드 필요
- 고객에게 공유되지 않는 내부 메모라는 점을 명시하되, 항상 외부에 노출되어도 문제 없는 수준으로 작성하도록 교육 필요
- 메모 삭제는 원칙적으로 금지하거나, 최소한 "삭제 이력"은 남겨야 함
- 중요한 예외 조치(정책 외 환불, 서비스 무료 제공 등)는 반드시 메모와 함께 태그(예: `IMPORTANT`, `EXCEPTION`)를 붙여 검색이 용이하도록 설계

#### 시나리오 12: 엑셀 업로드로 주문 일괄 등록
1. **관리자 로그인**:
 - 관리자는 백오피스 로그인 페이지에서 인증을 완료한다
 - 로그인 성공 후 기본 진입 화면으로 이동한다

2. **주문관리 메뉴 진입**:
 - 사이드 메뉴에서 메뉴 02 주문관리를 선택한다
 - 주문 목록 화면이 기본으로 표시된다

3. **엑셀 업로드 기능 진입**:
 - 주문 목록 상단에서 "엑셀 업로드" 버튼을 클릭한다
 - 엑셀 업로드 전용 팝업 화면이 표시된다

4. **서비스 카테고리 선택**:
 - 업로드 화면에서 "서비스 카테고리 선택" 드롭다운을 노출한다
 - 운영자는 다음 구조로 카테고리를 선택한다
   - 대분류 카테고리 선택
   - 중분류 카테고리 선택
 - 선택된 카테고리 정보는 service_category_id로 내부 매핑된다

5. **카테고리별 엑셀 템플릿 다운로드**:
 - 운영자는 "템플릿 다운로드" 버튼을 클릭한다
 - 시스템은 선택된 service_category_id를 기준으로 서비스관리 메뉴의 설정을 조회한다
 - 서비스관리 메뉴의 추가입력항목 탭에서 해당 카테고리의 추가 필드 정의를 조회한다
   - 필드명
   - 필드 타입 예: TEXT, NUMBER, DATE, SELECT
   - 필수 여부
   - 표시 순서
 - 시스템은 공통 컬럼과 카테고리별 추가입력항목 컬럼을 합쳐 엑셀 템플릿 컬럼 구조를 생성한다
- 공통 컬럼 구성 예시
- 필드1(필수값) : 채널코드(채널관리 메뉴에 등록된 채널의 코드를 기입한다.예>CH_700,CH_888,CH_900등)
- 필드2(선택값) : 채널주문번호(제휴 채널의 주문번호, 중복불가)
- 필드3(필수값) : 고객명(고객이름) 
- 필드4(필수값) : 고객연락처(고객전화번호, 010-1234-5678 형식)
- 필드5(필수값) : 서비스코드(서비스 카테고리 소분류 코드)
- 필드6(필수값) : 서비스수량(서비스 수량, 기본값:1)
- 필드7(필수값) : 서비스 주소(서비스 제공 주소)
- 필드8(선택값) : 상세 주소(동,호수 등)
- 필드9(선택값) : 서비스 희망일
- 필드10(선택값) : 고객요청사항(고객의 특별 요청사항)
- 필드11(선택값) : 결제금액(예약금액or선결제 금액, 숫자만입력)
- 필드12(선택값) : 예약금결제일
- 필드13(선택값) : 관리자메모(관리자용 메모)
- 필드14(선택값) : 전문가전달메모(전문가에게 전달할 메모)
- 필드15(필수값) : 주문신청 일시
- 카테고리별 추가입력항목 컬럼 구성 예시
   - 추가입력항목 탭에 정의된 필드 코드 또는 필드 키를 컬럼명으로 사용한다
   - 예: 실내기대수, 실외기설치위치, 층수, 평수 등
   - 필수 항목은 템플릿 내 주석 또는 예시 행으로 "필수" 표시를 한다
   - 카테고리별 추가입력 항목 컬럼은 필수가 아닌 선택값으로 넣는다.
- 시스템은 동적으로 생성된 컬럼 구조를 기반으로 .xlsx 파일을 생성하고 다운로드를 시작한다

6. **템플릿 작성**:
 - 운영자는 다운로드 받은 카테고리별 템플릿 파일을 연다
 - 각 행에 주문 데이터를 입력한다
   - 공통 컬럼에 고객/주문 기본 정보를 입력한다
   - 추가입력항목 컬럼에 현장에 필요한 상세 정보를 입력한다
 - 필수 컬럼이 비어 있지 않도록 내부 가이드에 따라 작성한다
 - 작성이 완료된 파일을 저장한다

7. **엑셀 파일 업로드**:
 - 다시 백오피스 주문관리의 엑셀 업로드 화면으로 돌아온다
 - 동일한 서비스 카테고리가 선택되어 있는지 확인한다
 - "파일 선택" 버튼을 클릭하여 작성 완료된 엑셀 파일을 선택한다
 - "업로드" 버튼을 클릭한다

8. **업로드 1차 검증 처리**:
 - 시스템은 업로드된 파일의 기본 형식을 검증한다
   - 파일 확장자가 .xlsx인지 확인한다
   - 템플릿 구조 버전 또는 hidden metadata를 사용해 템플릿 호환성을 검증한다
 - 선택된 service_category_id와 템플릿 내부의 카테고리 메타 정보가 일치하는지 확인한다
 - 컬럼 구조 검증을 수행한다
   - 공통 컬럼이 모두 존재하는지 확인한다
   - 서비스관리 → 추가입력항목 탭에 정의된 필드들이 모두 컬럼으로 존재하는지 확인한다
   - 정의되지 않은 추가 컬럼이 있는 경우 경고 정책을 적용한다

9. **업로드 데이터 행 단위 유효성 검증**:
 - 시스템은 각 행을 순회하며 유효성 검증을 수행한다
 - 공통 컬럼 검증
   - `customer_name`, `customer_phone`, `address`, `requested_date`, `total_price` 등의 필수값 존재 여부를 확인한다
   - 전화번호 포맷, 날짜 포맷, 금액 숫자 포맷을 검증한다
   - `channel_order_id` 중복 여부를 검사한다 동일 채널 내 중복인지 확인한다
 - 추가입력항목 검증
   - 서비스관리 → 추가입력항목 탭에서 필수로 정의된 항목들이 모두 입력되었는지 확인한다
   - 각 항목 타입에 맞는 데이터 형태인지 검증한다
     - NUMBER 타입은 숫자인지
     - DATE 타입은 날짜 포맷인지
     - SELECT 타입은 허용된 옵션 값인지
 - 검증 결과
   - 이상이 없는 행은 "유효"로 분류한다
   - 오류가 있는 행은 오류 사유와 함께 "오류 행" 리스트로 분류한다

10. **업로드 정책 선택**:
 - 일부 행에서 오류가 발생한 경우 시스템은 업로드 정책을 사용자에게 안내한다
 - 정책 옵션 
   - 전체 실패: 오류 행이 하나라도 있으면 전체 업로드를 취소하고 오류 리포트를 제공한다
 - 운영 정책에 따라 기본값을 정하고, 필요 시 설정에서 변경 가능하도록 한다

11. **주문 생성 처리**:
 - 유효 행에 대해 주문 생성 로직을 수행한다
 - 주문 생성 시 기본 상태 설정
   - `order_status` = `NEW`
   - `assignment_delivery_status` = `NONE`
   - `expert_progress_status` = null
   - `service_category_id` = 선택된 카테고리 ID
 - 고객 정보 생성 또는 매핑
   - `customer_phone` 기준으로 기존 고객 여부를 확인한다
   - 기존 고객이 있으면 customer_id를 재사용한다
   - 없으면 새 customer_id를 생성한다
 - 금액 정보 저장
   - total_price를 저장한다
   - paid_amount가 입력된 경우 paid_amount_total로 반영하고, 결제 트랜잭션 정보를 `type` = IMPORTED로 생성한다
 - 추가입력항목 데이터 저장
   - 서비스관리 → 추가입력항목 탭에 정의된 필드 키를 기준으로 주문 추가 정보 테이블에 값을 저장한다
   - 저장 구조 예시: `order_id`, `field_key`, `field_value`
 - 주문 생성 이력 기록
   - 주문 로그에 `CREATED_BY_EXCEL` 이벤트를 기록한다
   - 생성자 admin_id와 업로드 batch_id를 함께 기록한다

12. **업로드 결과 요약 및 피드백**:
 - 시스템은 업로드 결과 요약을 화면에 표시한다
   - 전체 행 수
   - 성공 생성 주문 수
   - 실패 행 수
 - 실패 행이 있을 경우
   - 실패 행만 포함된 "오류 행 엑셀 파일" 다운로드 링크를 제공한다
   - 각 행마다 오류 사유를 컬럼으로 포함한다
 - 생성된 주문으로 바로 이동할 수 있는 링크를 제공한다 예: "성공 건 주문 목록 보기"

13. **후속 프로세스 연계**:
 - 생성된 주문에 대해 운영팀은 일반 신규 주문과 동일하게 검수 및 배정 프로세스를 수행한다
 - paid_amount가 존재하는 주문은 총결제 금액과 동이하면 결제 완료 상태로 간주되며, 총결제 금액보다 작을 경우 부분 결제 완료 상태로 간주된다. 이후 환불·정산 로직과 연계된다
 - 추가입력항목 데이터는 전문가웹/앱에서 현장 준비 정보로 활용된다

14. **운영 및 정책 관점 주의사항**:
 - 서비스관리에서 추가입력항목 정의를 변경하면, 이후 생성되는 템플릿 컬럼 구조도 함께 변경되므로 변경 시점 관리에 유의해야 한다
 - 기존에 배포된 템플릿 파일을 사용하는 운영자가 있을 수 있으므로, 템플릿 버전 또는 생성일 표시를 통해 호환성 문제를 줄여야 한다
 - 카테고리별 템플릿을 잘못 선택하여 업로드하는 것을 방지하기 위해, 업로드 시 템플릿 내 카테고리 메타 정보와 현재 선택된 카테고리를 반드시 교차 검증해야 한다
 - 대량 업로드는 운영 리스크가 크므로, 특정 권한 이상에서만 사용 가능하도록 권한 제어가 필요하다

#### 시나리오 12-1: 한 고객의 다중/다수 서비스 구매 처리 (엑셀 업로드 확장)
본 시나리오는 한 명의 고객이 동일 시점에 두 개 이상의 서비스를 구매한 경우를 엑셀 업로드 구조에 반영하는 방식을 정의한다.
1. **기본 원칙 정의**:
- 한 행(row)은 하나의 서비스 주문을 의미한다
 - 한 고객이 여러 서비스를 구매한 경우, 서비스 개수만큼 여러 행으로 작성한다
 - 시스템 내부에서는 "서비스 1개 = 주문 1건"으로 생성한다
 - 여러 주문을 하나의 묶음으로 식별하기 위해 `order_group_key` 및 `order_group_id`를 사용한다

2. **엑셀 템플릿 컬럼 확장 정의**:
- 공통 컬럼에 다음 항목을 추가한다
   - `order_group_key`: 동일 고객의 다중 서비스 주문을 하나의 묶음으로 식별하기 위한 그룹 키(선택 입력)
- `order_group_key` 작성 규칙
   - 동일 고객의 동일 묶음 주문에는 동일한 값을 입력한다
   - 묶음이 아닌 단일 주문인 경우 비워둘 수 있다
   - 예시 값
     - 20250301-홍길동-1
     - 외부 채널 주문번호와 동일 값

3. **업로드 시 그룹 처리 로직**:
- 시스템은 업로드된 각 행을 순회하며 `order_group_key` 값을 확인한다
 - `order_group_key`가 존재하는 경우
   - 동일한 `order_group_key`를 가진 행들을 하나의 그룹으로 인식한다
   - 내부적으로 `order_group_id`를 생성한다
   - 동일한 `order_group_key`를 가진 모든 주문에 동일한 `order_group_id`를 저장한다
- `order_group_key`가 없는 경우
   - 해당 주문은 개별 주문으로 생성한다
   - `order_group_id`는 `null` 또는 단독 생성된 값으로 처리한다

4. **서로 다른 카테고리 서비스 동시 구매 처리**:
- 서로 다른 서비스 카테고리의 경우
   - 각 카테고리별 템플릿을 사용하여 개별 업로드를 수행한다
   - 동일 고객의 묶음 주문일 경우 동일한 `order_group_key` 값을 사용한다
- 업로드 완료 후 시스템 처리
   - 서로 다른 업로드 batch에서 생성된 주문이라도
   - `order_group_key`가 동일하면 동일한 `order_group_id`로 매핑한다

5. **주문 생성 시 데이터 저장 구조**:
- 주문 테이블에 다음 필드를 추가한다
   - `order_group_id`: 내부 묶음 식별자
   - `order_group_key`: 외부 입력 기반 그룹 키
- 주문 생성 시 처리
   - `order_group_key`가 존재하면 기존 동일 키의 `order_group_id` 조회
   - 존재하면 동일 `order_group_id` 사용
   - 존재하지 않으면 신규 `order_group_id` 생성

6. **운영 화면에서의 활용**:
- 주문 목록 화면
   - `order_group_id`가 동일한 주문은 묶음 표시 아이콘 또는 태그로 표시한다
   - 그룹 클릭 시 해당 그룹의 주문 목록만 필터링하여 조회 가능하도록 한다
- 주문 상세 화면
   - 동일 `order_group_id`를 가진 다른 주문 목록을 하단에 표시한다
   - 운영자가 "묶음 전체 보기" 기능을 통해 관련 주문을 한 번에 확인할 수 있도록 한다

7. **배정 및 정산과의 관계**:
- 배정은 주문 단위로 수행한다
   - 같은 그룹이라도 각 주문은 독립적으로 배정된다
   - 동일 전문가에게 배정되도록 운영자가 수동 조정 가능
- 정산은 주문 단위로 계산한다
   - 그룹 단위 정산은 기본 제공하지 않는다
   - 필요 시 그룹 단위 합산 리포트만 제공한다

8. **취소 및 환불 시 그룹 처리 원칙**:
- 기본 원칙
   - 취소 및 환불은 주문 단위로 처리한다
   - 한 그룹의 일부 주문만 취소 가능
- 운영 편의 기능
   - "묶음 전체 취소" 버튼은 향후 확장 기능으로 고려 가능
   - 해당 기능 구현 시 그룹 내 모든 주문에 동일 상태 전환을 일괄 적용한다

9. **운영 리스크 통제 포인트**:
- 서로 다른 카테고리 템플릿 업로드 시 동일 `order_group_key`를 정확히 입력하도록 운영 가이드 필요
 - 동일 `channel_order_id`와 `order_group_key` 혼동 방지 필요
 - 그룹 단위 취소/환불 오처리 방지를 위해 기본 단위는 항상 "주문 단위"임을 유지한다
 - BI 및 통계 집계 시 그룹 단위와 주문 단위를 구분하여 정의한다


#### 시나리오 13: 주문 데이터 엑셀 다운로드
본 시나리오는 주문 데이터를 엑셀(.xlsx) 형식으로 다운로드하여 운영 분석, 정산 검토, 외부 보고, CS 이력 관리 등에 활용하는 흐름을 정의한다.
1. **관리자 로그인**:
 - 관리자는 백오피스 로그인 페이지에서 인증을 완료한다
 - 로그인 성공 후 기본 화면으로 이동한다
2. **주문관리 메뉴 진입**:
 - 상단 또는 사이드 메뉴에서 메뉴 02 주문관리를 선택한다
 - 주문 목록 화면이 표시된다
3. **다운로드 대상 주문 조회**:
- 검색 조건 설정
   - 기간 필터: 주문 생성일, 예약일, 결제일, 취소일 등
   - `order_status` 필터
   - `expert_progress_status` 필터
   - `assignment_delivery_status` 필터
   - 서비스 카테고리(대분류 > 중분류)
   - 채널(`channel_code`)
   - `region_group`
   - 전문가 ID
   - `order_group_id` 또는 `order_group_key`
- 검색 실행
   - 조건에 맞는 주문 목록이 화면에 표시된다
   - 총 건수가 상단에 표시된다

4. **엑셀 다운로드 버튼 클릭**:
- "엑셀 다운로드" 버튼 클릭
 - 다운로드 옵션 선택 팝업 노출

5. **다운로드 옵션 선택**:
- 다운로드 범위 선택
   - 현재 검색 조건 전체 다운로드
   - 현재 페이지 다운로드
   - 선택 체크박스 주문만 다운로드
- 컬럼 구성 선택
  - 기본 컬럼만 다운로드
   - 기본 컬럼 + 추가입력항목 포함
   - 사용자 정의 컬럼 선택 기능(확장 기능)
- 파일 분할 정책
   - 건수 초과 시 자동 분할 다운로드
   - 예: 10,000건 초과 시 분할 파일 생성

6. **엑셀 컬럼 구성 정의**:
- 기본 공통 컬럼
  - `order_id`
   - `order_group_id`
   - `order_group_key`
   - `channel_code`
   - `channel_order_id`
   - `customer_name`
   - `customer_phone`
   - `address`
   - `service_category_name`
   - `region_group`
   - `expert_master_id`
   - `expert_worker_id`
   - `order_status`
   - `expert_progress_status`
   - `assignment_delivery_status`
   - `total_price`
   - `paid_amount_total`
   - `refund_amount_total`
   - `refund_type`
   - `created_at`
   - `assigned_at`
   - `service_date`
   - `canceled_at`
   - `refunded_at`
  - 이외에 주문관리에 포함된 모든 컬럼을 선택 다운로드 할수있게 함. 
- 추가입력항목 컬럼 구성
  - 서비스관리 → 추가입력항목 탭에서 정의된 필드 목록 조회
   - 선택된 주문들의 서비스 카테고리를 기준으로 필요한 필드 집합 계산
   - 동적으로 컬럼 생성
   - 각 주문의 추가입력항목 값 매핑
   - 값이 없는 경우 공란 처리

7. **데이터 추출 처리**:
- 현재 검색 조건을 기준으로 DB 조회
 - 대량 데이터의 경우 비동기 처리
   - 다운로드 요청 시 background job 생성
   - 처리 완료 시 알림 또는 다운로드 링크 제공
- 데이터 정합성 보장
   - 조회 시점 기준 스냅샷 처리
   - 다운로드 도중 상태 변경이 발생하더라도 동일 스냅샷 유지

8. **엑셀 파일 생성**:
- 파일 포맷은 .xlsx
 - 첫 번째 행은 컬럼 헤더
 - 금액 필드는 숫자 형식 유지
 - 날짜 필드는 ISO 포맷 또는 yyyy-mm-dd hh:mm:ss 형식 유지
 - 상태 값은 내부 코드 또는 표시명 정책에 따라 결정

9. **다운로드 완료 처리**:
- 파일 다운로드 시작
 - 다운로드 이력 기록
   - admin_id
   - 다운로드 조건
   - 다운로드 건수
   - 다운로드 시각
- 보안 정책
   - 고객 개인정보 포함 파일은 암호화 다운로드 옵션 고려
   - 일정 기간 이후 다운로드 링크 자동 만료

10. **특수 케이스 처리**:
- 서로 다른 카테고리가 혼합된 경우
  - 모든 추가입력항목 컬럼을 union 형태로 생성
   - 특정 주문에 해당 없는 컬럼은 공란 처리
- 묶음 주문(`order_group_id`)
  - 동일 `order_group_id`를 가진 주문은 각각 별도 행으로 출력
   - 필요 시 그룹 단위 집계 리포트는 별도 기능으로 확장
- 대량 다운로드
  - 동시 다운로드 요청 제한
   - 시스템 부하 방지를 위한 rate limit 적용

11. **운영 리스크 통제 포인트**:
- 개인정보 대량 유출 방지를 위해 권한 기반 접근 제어 필요
 - 다운로드 로그는 최소 1년 이상 보관 권장
 - 추가입력항목 변경 시 과거 주문 데이터와 컬럼 구조 불일치 가능성 고려
 - BI 집계용 다운로드와 운영용 다운로드를 분리하는 것이 장기적으로 바람직


#### 시나리오 14: 주문 강제 상태 수정(운영 개입)
본 시나리오는 정상적인 상태머신 흐름을 우회하여
 운영자가 특정 주문의 상태를 강제로 변경해야 하는 예외 상황을 정의한다.
본 기능은 고위 권한 관리자에게만 허용한다.

1. **관리자 로그인**:
- 운영 권한이 있는 계정으로 로그인한다
 - 해당 계정이 "강제 상태 수정 권한"을 보유했는지 검증한다
 - 권한이 없을 경우 해당 기능은 노출되지 않는다

2. **주문 상세 화면 진입**:
- 주문관리 메뉴에서 특정 주문을 조회한다
 - 주문 상세 화면에 진입한다
 - 현재 상태 확인
   - `order_status`
   - `expert_progress_status`
   - `assignment_delivery_status`
   - 결제 상태
   - 정산 확정 여부

3. **강제 상태 수정 버튼 클릭**:
- "강제 상태 수정" 버튼 클릭
 - 경고 팝업 노출
   - 정상 상태 흐름을 우회한다는 경고 문구 표시
   - 해당 변경이 정산 및 통계에 영향을 줄 수 있음을 안내
   - 관리자 재확인 필요

4. **수정 대상 상태 선택**:
- 수정 가능한 상태 목록은 현재 상태에 따라 제한적으로 노출한다
- 수정 가능한 대상
  - `order_status` 변경
   - `expert_progress_status` 변경
   - `assignment_delivery_status` 변경
- 허용 예시
  - ASSIGNED → UNASSIGNED
   - ASSIGNED → CANCELED
   - WAIT_PAYMENT → CONFIRMED
   - IN_SERVICE → CONFIRMED
   - HOLD → SENT
- 절대 허용 불가 예시
  - REFUNDED → ASSIGNED
   - CANCELED → ASSIGNED (결제/배정 정합성 깨짐)
   - 정산 확정 완료 주문을 서비스 진행 상태로 되돌리는 것

5. **강제 변경 사유 입력**:
- 사유 코드 선택
  - SYSTEM_ERROR
   - PG_SYNC_ERROR
   - EXPERT_APP_ERROR
   - DATA_CORRECTION
   - INTERNAL_POLICY_EXCEPTION
- 상세 사유 메모 입력 필수
  - 변경 필요 배경 설명
   - 고객/전문가 통화 여부
   - 관련 정책 근거
- 메모 미입력 시 저장 불가

6. **시스템 사전 정합성 검증**:
- 결제 관련 검증
  - `paid_amount_total` > 0 인데 `order_status`를 NEW로 변경 시 차단
   - `refund_amount_total` > 0 인 상태에서 ASSIGNED로 변경 시 차단
- 배정 관련 검증
  - `assignment_delivery_status` = SENT 상태에서 `expert_master_id`가 null이면 차단
   - HOLD 상태에서 전문가 미존재 시 차단
- 정산 관련 검증
  - 정산 확정 상태인 경우 일부 상태 변경 차단
   - 변경 시 재정산 플래그 자동 설정

7. **상태 변경 실행**:
- 트랜잭션 단위 처리
  - `order_status` 변경
   - `expert_progress_status` 변경
   - `assignment_delivery_status` 변경
   - 필요 시 배정 정보 초기화
- 자동 후처리 로직
  - ASSIGNED → UNASSIGNED 변경 시 자동 재배정 비활성화
   - HOLD → SENT 변경 시 전문가 알림 발송
   - WAIT_PAYMENT → CONFIRMED 변경 시 결제 완료 로직 트리거 여부 정책 정의

8. **이력 기록**:
- 주문 이력 로그에 기록
  - `event_type` = FORCE_STATUS_CHANGE
   - 이전 상태 값 기록
   - 변경 후 상태 값 기록
   - `admin_id` 기록
   - 변경 시각 기록
- 메모 자동 생성
  - `memo_type` = FORCE_CHANGE
   - 입력한 사유 자동 저장

9. **알림 및 연동 처리**:
- 상태에 따라 고객 알림 여부 판단
  - CANCELED로 변경 시 고객 알림
   - CONFIRMED로 변경 시 전문가 알림
- 외부 채널 연동 여부 확인
  - 채널 주문의 경우 상태 동기화 정책 별도 정의

10. **운영 리스크 통제 포인트**:
- 강제 상태 수정은 감사 로그 대상 기능으로 분류한다
 - 변경 이력은 삭제 불가
 - 하루 강제 수정 횟수 제한 또는 알림 체계 필요
 - 강제 수정 주문은 BI에서 별도 플래그로 집계

11. **정책적 권장사항**:
- 강제 수정은 최후 수단으로 사용한다
 - 시스템 버그로 반복 발생하는 경우 로직 수정이 우선이다
 - 강제 수정 빈도는 월 단위 리포트로 관리한다
