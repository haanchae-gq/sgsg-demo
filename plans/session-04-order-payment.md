# 세션 4: 주문 및 결제 API

## 세션 개요
이 세션은 고객의 서비스 주문 생성부터 결제 완료까지의 전체 주문 관리 기능과 결제 처리 기능을 구현합니다. 주문 생성, 조회, 수정, 취소, 메모 관리, 첨부파일 관리와 결제 초기화, 완료, 환불 처리 등을 포함합니다.

**우선순위**: P2 (서비스 카탈로그 후)  
**예상 작업 시간**: 4-5일  
**의존성**: 세션 1 (인증), 세션 3 (서비스 카탈로그)  
**다중 에이전트 작업 가능**: 예 (주문 관리와 결제 관리로 분할)

## 필요한 데이터베이스 테이블/마이그레이션
### 이미 존재하는 테이블 (Prisma schema 확인)
- `Order` - 주문 기본 정보
- `OrderExtraFieldValue` - 주문별 추가입력항목 값
- `OrderNote` - 주문 메모
- `OrderAttachment` - 주문 첨부파일
- `Payment` - 결제 정보
- `ServiceItem` - 서비스 소분류 (참조)
- `Customer` - 고객 정보 (참조)
- `Expert` - 전문가 정보 (참조)

### 필요한 마이그레이션
1. `Order` 테이블 확장: 결제 상태, 배정 상태, 진행 상태 필드 추가
2. `Payment` 테이블 확장: PG사 응답 데이터 저장 필드, 환불 관련 필드
3. 관계 설정: 주문 ↔ 결제, 주문 ↔ 서비스 항목, 주문 ↔ 고객/전문가
4. 인덱스: 주문 조회 성능을 위한 인덱스 (고객ID, 전문가ID, 상태, 생성일시)

## 의존성
### 내부 의존성
- **세션 1 (인증)**: 사용자 인증, 역할 기반 접근 제어
- **세션 3 (서비스 카탈로그)**: 서비스 항목 조회, 가격 정보
- **세션 2 (전문가 관리)**: 전문가 배정 로직 (선택적 - 초기에는 단순 구현)

### 외부 의존성
- **PG(Payment Gateway) 연동**: 결제 처리 (테스트용 샌드박스 먼저)
- **이메일/SMS 알림**: 주문 상태 변경 알림

## 테스트 시나리오
### 단위 테스트
1. `OrderService` 단위 테스트: 주문 생성, 조회, 수정, 취소
2. `OrderNoteService` 단위 테스트: 주문 메모 CRUD
3. `OrderAttachmentService` 단위 테스트: 첨부파일 업로드/조회
4. `PaymentService` 단위 테스트: 결제 초기화, 완료 처리, 환불
5. `PGIntegrationService` 단위 테스트: PG사 연동 모듈

### 통합 테스트 (E2E)
1. 주문 생성 → 결제 초기화 → 결제 완료 전체 흐름
2. 주문 취소 및 부분 환불 시나리오
3. 주문 메모 및 첨부파일 관리
4. 전문가 배정 연계 시나리오 (선택적)

## 다중 에이전트 작업을 위한 특별 지침
1. **작업 분할**: 
   - 에이전트 1: 주문 생성/기본 CRUD + 결제 초기화/완료 + PG 연동
   - 에이전트 2: 주문 메모/첨부파일 + 결제 조회/환불
   
2. **공유 리소스**:
   - `src/routes/v1/orders/`, `/payments/`: 라우트 구조 협의
   - `src/services/order.service.ts`, `payment.service.ts`: 서비스 로직 분리
   - 주문 상태 머신: 상태 전이 로직 협의 필요

3. **통합 포인트**:
   - 주문-결제 상태 동기화: 결제 완료 시 주문 상태 업데이트
   - 에러 처리: 결제 실패 시 롤백 메커니즘
   - 트랜잭션 관리: Prisma 트랜잭션 활용

4. **보안 고려사항**:
   - 결제 정보 암호화 저장
   - PG 콜백 엔드포인트 보안 검증
   - 환불 요청 권한 엄격 관리

5. **테스트 협업**:
   - PG 샌드박스 환경 공유
   - 결제 테스트 시나리오 협의 (성공/실패/부분환불 등)
