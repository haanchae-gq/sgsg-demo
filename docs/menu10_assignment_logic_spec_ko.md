# 메뉴 10 배정 로직 관리: 개발 스펙 수정 포인트

기존 스펙 문서에서 멤버십 구조가 “서브 계정 개수 기반 슬롯”으로 정의되어 있어서, 이부분을 
“마스터 계정 기준 멤버십 구좌 수(slot_count)”로 수정이 필요합니다.

## 1-1. 용어 정의 섹션 수정 : 멤버십 전문가, 슬롯 정의

### 1-1-2. 현재 문제점 :
1-1-2-1. 멤버십 전문가 정의가 “멤버십 서브 계정을 1개 이상 보유한 마스터”로 되어 있음
1-1-2-2. 슬롯 수 = “멤버십 서브 계정 수”로 정의되어 있음

### 1-1-3. 수정 방향 :
1-1-3-1. “멤버십 서브 계정 수”라는 표현을 모두 “멤버십 구좌 수(membership_slot_count)”로 치환
1-1-3-2. 멤버십 여부 판단 기준을 “마스터 계정에 설정된 구좌 수”로 변경

### 1-1-4. 수정 예시(개발 스펙 문서 문구 예) :
1-1-4-1. 멤버십 전문가 :
: 특정 서비스 중분류에 대해 유효한 멤버십 구좌 수(membership_slot_count)가 1개 이상 설정된 마스터 계정을 의미한다.

1-1-4-2. 2-11. 슬롯(Slot) :
: 슬롯이란 자동배정 순번 리스트에서 하나의 마스터 계정이 차지하는 배정 기회 횟수를 의미하며, 멤버십 전문가의 경우 master_account 단위로 설정된 membership_slot_count 값과 동일하다. 일반 전문가는 항상 1개의 슬롯을 가진다.

## 1-2. 멤버십 서브 계정 언급 제거 및 마스터 기준 정리

### 1-2-2. 현재 문제점 :
1-2-2-1. 멤버십을 서브 계정 멤버십 집계 기반으로 계산하도록 되어 있음

### 1-2-3. 수정 방향 :
1-2-3-1. 멤버십 서브 계정 집계 로직을 제거
1-2-3-2. master_account의 membership_slot_count를 단일 소스로 사용

### 1-2-4. 수정 예시 :
1-2-4-1. 멤버십 전문가 조건 :
: 특정 서비스 중분류에 대해 master_account의 membership_slot_count > 0 인 경우, 해당 마스터 계정은 해당 카테고리에서 멤버십 전문가로 간주한다.

1-2-4-2. 5-4-3 멤버십 전문가의 슬롯 수 :
: 특정 중분류에 대해 해당 마스터 계정에 설정된 membership_slot_count 값을 슬롯 수로 사용한다.

## 1-3. 후보군 그룹 분리 로직 정정

### 1-3-2. 현재 문제점 :
1-3-2-1. 그룹 분리 조건이 “멤버십 서브 계정 수 집계”에 의존

### 1-3-3. 수정 방향 :
1-3-3-1. 그룹 분리 기준을 “membership_slot_count > 0”으로 통일

### 1-3-4. 수정 예시 :
1-3-4-1.:
: 멤버십 그룹 = membership_slot_count ≥ 1인 마스터 계정 집합
1-3-4-2. :
: 일반 그룹 = membership_slot_count = 0인 마스터 계정 집합

# 2. 메뉴 10 고도화 설계 정리(v2/v3 확장 포인트)

이 섹션은 “지금 바로 개발하지 않지만, 구조는 이렇게 갈 것”을 명시

## 2-1. 전문가 일일 배정 상한 확장 설계

### 2-1-1. 도메인 모델 확장 :
2-1-1-1. master_assignment_policy 테이블에 daily_assignment_limit 필드 추가
2-1-1-2. 0 : 제한 없음, 1 이상 : 일일 상한
2-1-1-3. today_assigned_count는 통계/캐시에서 실시간 제공

### 2-1-2. 필터 단계 반영 :
2-1-2-1. 후보군 필터(서비스·권역·상태·멤버십) 이후
2-1-2-2. daily_assignment_limit > 0 이고 today_assigned_count ≥ daily_assignment_limit 인 마스터는 자동배정 후보군에서 제외

### 2-1-3. 멤버십과의 관계 :
2-1-3-1. membership_slot_count가 아무리 커도 상한을 넘으면 배정 불가
2-1-3-2. 멤버십은 “후보군 내부 우선순위”일 뿐, 상한 필터보다 뒤에 서지 않음

### 2-1-4. 구현 시기 :
2-1-4-1. v1에서는 “1-8-2-3 전문가별 일/주간 상한제” 하위에 설계만 명시
2-1-4-2. v2 이상에서 구현 플래그 ON

## 2-2. 전문가 통계/배정 품질 지표 연동 설계

### 2-2-1. 통계 모듈 역할 :
2-2-1-1. master_account_id + 기간 조건으로
2-2-1-2. 배정 건수, 거절률, 타임아웃률, 평균 응답 시간, 멤버십 배정 비율을 계산

### 2-2-2. 메뉴 10에서의 활용 :
2-2-2-1. “배정 엔진 시뮬레이션” 화면에서
2-2-2-2. 후보군 리스트에 각 마스터의 지표를 함께 노출
2-2-2-3. 향후 “지표 기반 가중치”를 붙이기 위한 설계만 명시

### 2-2-3. 고도화 방향 :
2-2-3-1. 응답속도, 수락률 등에 따라 기본 가중치(1)를 0.5~1.5 범위에서 조정
2-2-3-2. 이 로직은 “슬롯 계산 단계 앞단”에 삽입 가능하도록 설계

## 2-3. 포인터/슬롯 고도화 설계

### 2-3-1. 슬롯 상한 설계 :
2-3-1-1. membership_slot_count가 N이더라도
2-3-1-2. 슬롯 수 = min(N, max_slot_limit) 형태로 조정 가능
2-3-1-3. max_slot_limit는 정책 테이블에서 관리

### 2-3-2. 포인터 스코프 확장 가능 설계 :
2-3-2-1. 현재 스코프 = service_category_mid + region_group
2-3-2-2. 향후 스코프를 “시/도” 또는 “시/군/구”로 확장할 수 있도록
2-3-2-3. 포인터 테이블에 generic scope_key 컬럼을 두고, region_group은 그중 하나의 타입으로 처리

## 2-4. 배정 실패/미배정 케이스 자동 분석 설계

### 2-4-1. 자동배정 실패 로그를
2-4-1-1. 필터 단계별 탈락 사유(카테고리, 권역, 상태, 멤버십, 상한 등)로 집계

### 2-4-2. 메뉴 10에서
2-4-2-1. “특정 기간 동안 필터 단계별 탈락 비율”을 조회하는 화면을 정의

### 2-4-3. 이를 통해
2-4-3-1. 특정 지역 인력 부족
2-4-3-2. 과도한 필터 강도
2-4-3-3. 멤버십 정책 실패
를 빠르게 진단할 수 있도록 설계

# 3. 메뉴 10 배정 로직 관리: 사용자 시나리오

## 3-1. 시나리오 1: 활성 배정 로직 버전 조회 및 변경

3-1-1. 관리자 로그인
3-1-2. 백오피스 → 메뉴 10 배정 로직 관리 진입
3-1-3. 상단 “배정 로직 버전 관리” 탭 선택
3-1-4. 현재 활성 버전 정보 확인
: version_id, 설명, 적용 시작일, 작성자 표시
3-1-5. 기존 버전 목록 조회
: 과거 version_id 리스트, 적용 기간, 주요 변경 요약 표시
3-1-6. 새로운 정책 초안 작성 완료 후 “새 버전 등록” 클릭
3-1-7. 정책 JSON 또는 설정 폼 입력
: 후보군 필터 기준, 멤버십 적용 여부, 슬롯 계산 방식, 포인터 스코프 등
3-1-8. 유효성 검사 실행
: 필수 필드 누락, 범위 오류, 포인터 스코프 중복 여부 확인
3-1-9. 저장 후 “활성 버전으로 전환” 체크
3-1-10. 활성 버전 전환 시 경고 팝업 노출
: “전환 시점 이후 모든 자동배정은 새 버전을 사용합니다.”
3-1-11. 관리자 최종 확인 후 적용
3-1-12. version_history 테이블에 변경 로그 기록
3-1-13. 주문관리 메뉴에서 이후 실행되는 자동배정이 새 버전으로 동작

## 3-2. 시나리오 2: 특정 서비스/권역의 배정 후보군 및 슬롯 구조 시뮬레이션

3-2-1. 관리자 로그인
3-2-2. 메뉴 10 배정 로직 관리 진입
3-2-3. “후보군/슬롯 시뮬레이션” 탭 선택
3-2-4. 테스트용 서비스 중분류 선택
3-2-5. 테스트용 region_group 선택
3-2-6. 테스트용 채널 선택(자사/제휴 등)
3-2-7. “시뮬레이션 실행” 버튼 클릭
3-2-8. 시스템이 현재 활성 정책 기준으로 후보군 필터링 실행
3-2-9. 결과 화면에 다음 정보 표시
3-2-9-1. 후보군 master_account 목록
3-2-9-2. 각 마스터의 membership_slot_count
3-2-9-3. 멤버십/일반 그룹 구분
3-2-9-4. 계산된 슬롯 수
3-2-9-5. 최종 순번 리스트(예: [M1, M2, M2, M3, N1…])
3-2-10. 운영자가 특정 전문가 선택 → 상세 지표 확인
: 평점, 수락률, 응답속도, 최근 배정 건수
3-2-11. 리드 몰림 징후 확인 시
: 멤버십 정책 또는 슬롯 상한 고도화 필요 여부 판단
3-2-12. 필요 시 정책 수정 후 다시 시뮬레이션 실행

## 3-3. 시나리오 3: region_group 포인터 상태 모니터링

3-3-1. 관리자 로그인
3-3-2. 메뉴 10 배정 로직 관리 진입
3-3-3. “포인터 상태 조회” 탭 선택
3-3-4. 서비스 중분류 선택
3-3-5. region_group 선택
3-3-6. 해당 조합의 포인터 레코드 조회
: current_index, last_assigned_at, 리스트 길이
3-3-7. 최근 배정 이력과 비교
: 마지막 배정된 master_account, 그때의 index
3-3-8. 포인터가 리스트 길이보다 크게 벗어나 있는지 검증
3-3-9. 후보군 급변 등으로 인한 리스트 길이 변경 시
: current_index 재매핑 또는 초기화가 필요한지 판단

: 개발/운영에 포인터 리셋 요청(고도화 시 관리자 UI에서 제한적 리셋 가능)

3-4-9-3. 최종 “미배정” 처리 시점 및 사유 확인
3-4-10. 이를 기반으로
3-4-10-1. 특정 region_group 인력 부족
