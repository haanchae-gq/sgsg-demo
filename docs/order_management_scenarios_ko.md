### 시나리오 12-1: 한 고객의 다중/다수 서비스 구매 처리 (엑셀 업로드 확장)
본 시나리오는 한 명의 고객이 동일 시점에 두 개 이상의 서비스를 구매한 경우를 엑셀 업로드 구조에 반영하는 방식을 정의한다.
1: 기본 원칙 정의
1-1: 한 행(row)은 하나의 서비스 주문을 의미한다
 1-2: 한 고객이 여러 서비스를 구매한 경우, 서비스 개수만큼 여러 행으로 작성한다
 1-3: 시스템 내부에서는 "서비스 1개 = 주문 1건"으로 생성한다
 1-4: 여러 주문을 하나의 묶음으로 식별하기 위해 order_group_key 및 order_group_id를 사용한다

2: 엑셀 템플릿 컬럼 확장 정의
2-1: 공통 컬럼에 다음 항목을 추가한다
 2-1-1: order_group_key: 동일 고객의 다중 서비스 주문을 하나의 묶음으로 식별하기 위한 그룹 키(선택 입력)
2-2: order_group_key 작성 규칙
 2-2-1: 동일 고객의 동일 묶음 주문에는 동일한 값을 입력한다
 2-2-2: 묶음이 아닌 단일 주문인 경우 비워둘 수 있다
 2-2-3: 예시 값
 2-2-3-1: 20250301-홍길동-1
 2-2-3-2: 외부 채널 주문번호와 동일 값

3: 업로드 시 그룹 처리 로직
3-1: 시스템은 업로드된 각 행을 순회하며 order_group_key 값을 확인한다
 3-2: order_group_key가 존재하는 경우
 3-2-1: 동일한 order_group_key를 가진 행들을 하나의 그룹으로 인식한다
 3-2-2: 내부적으로 order_group_id를 생성한다
 3-2-3: 동일한 order_group_key를 가진 모든 주문에 동일한 order_group_id를 저장한다
3-3: order_group_key가 없는 경우
 3-3-1: 해당 주문은 개별 주문으로 생성한다
 3-3-2: order_group_id는 null 또는 단독 생성된 값으로 처리한다

4: 서로 다른 카테고리 서비스 동시 구매 처리
4-1: 서로 다른 서비스 카테고리의 경우
 4-1-1: 각 카테고리별 템플릿을 사용하여 개별 업로드를 수행한다
 4-1-2: 동일 고객의 묶음 주문일 경우 동일한 order_group_key 값을 사용한다
4-2: 업로드 완료 후 시스템 처리
 4-2-1: 서로 다른 업로드 batch에서 생성된 주문이라도
 4-2-2: order_group_key가 동일하면 동일한 order_group_id로 매핑한다

5: 주문 생성 시 데이터 저장 구조
5-1: 주문 테이블에 다음 필드를 추가한다
 5-1-1: order_group_id: 내부 묶음 식별자
 5-1-2: order_group_key: 외부 입력 기반 그룹 키
5-2: 주문 생성 시 처리
 5-2-1: order_group_key가 존재하면 기존 동일 키의 order_group_id 조회
 5-2-2: 존재하면 동일 order_group_id 사용
 5-2-3: 존재하지 않으면 신규 order_group_id 생성

6-1: 주문 목록 화면
 6-1-1: order_group_id가 동일한 주문은 묶음 표시 아이콘 또는 태그로 표시한다
 6-1-2: 그룹 클릭 시 해당 그룹의 주문 목록만 필터링하여 조회 가능하도록 한다
6-2: 주문 상세 화면
 6-2-1: 동일 order_group_id를 가진 다른 주문 목록을 하단에 표시한다
 6-2-2: 운영자가 "묶음 전체 보기" 기능을 통해 관련 주문을 한 번에 확인할 수 있도록 한다

7: 배정 및 정산과의 관계
7-1: 배정은 주문 단위로 수행한다
 7-1-1: 같은 그룹이라도 각 주문은 독립적으로 배정된다
 7-1-2: 동일 전문가에게 배정되도록 운영자가 수동 조정 가능

8-2: 운영 편의 기능

### 시나리오 15: 일정 변경에 따른 주문 정보 수정
본 시나리오는 예약/방문 일정이 변경될 때
 주문 정보와 일정(캘린더) 정보를 어떻게 함께 수정·동기화할지에 대한 운영 절차를 정의한다.
1: 관리자 로그인
 1-1: 운영 계정으로 백오피스에 로그인한다
 1-2: 로그인 성공 후 메뉴 02 주문관리로 이동한다
2: 일정 변경 요청이 들어온 주문 검색
2-1: 검색 조건 설정
 2-1-1: 주문번호로 검색
 2-1-2: 고객 전화번호로 검색
 2-1-3: service_date(예약일) 기준으로 오늘/내일 일정 필터
 2-1-4: expert_progress_status IN (RESERVED, IN_SERVICE, AS_REQUEST) 등 일정이 연관된 상태만 필터
2-2: 검색 결과에서 일정 변경 요청이 들어온 주문을 선택한다
3: 주문 상세 화면 진입 및 현재 상태 확인
3-1: 주문 상세 화면 진입
 3-2: 기본 정보 확인
 3-2-1: order_status (ASSIGNED, CANCELED 등)
 3-2-2: expert_progress_status (RESERVED, IN_SERVICE 등)
 3-2-3: assignment_delivery_status (SENT/HOLD 등)
 3-2-4: 현재 service_date, service_time 정보
 3-2-5: 배정된 expert_master_id, expert_worker_id
3-3: 일정 이벤트 연동 여부 확인
 3-3-1: 해당 주문과 연결된 캘린더 이벤트 존재 여부
 3-3-2: event_type (SERVICE / AS)
 3-3-3: event_start_time, event_end_time
4: 일정 변경 요청 내용 파악
4-1: 고객 또는 전문가로부터 전달받은 변경 내용 확인
 4-1-1: 날짜 변경 요청 (예: 3/10 → 3/12)
 4-1-2: 시간대 변경 요청 (예: 오전 → 오후)
 4-1-3: 동일 일자 내 시간만 이동
 4-1-4: 날짜 + 시간 모두 변경
4-2: 추가 조건 확인
 4-2-1: 당일/당일 직전 변경인지 여부 (페널티/정책 연동 가능)
 4-2-2: 주소 변경 수반 여부 (region_group 변경 여부 확인)
 4-2-3: 기존 전문가 일정과의 충돌 가능성
5: 변경 가능 여부 검토
5-1: 상태 기반 검토
 5-1-1: expert_progress_status = RESERVED 인 경우: 일반적인 일정 변경 허용
 5-1-2: expert_progress_status = IN_SERVICE 인 경우: 원칙적으로 일정 변경이 아닌 "실제 서비스 중"이므로, 시간 변경보다는 이력 메모 위주 처리
 5-1-3: expert_progress_status = CONFIRMED 인 경우: 서비스 종료로 간주, 일정 변경 불가 (예외 시 강제 상태 수정 시나리오 연계)
5-2: 전문가 스케줄 기반 검토
 5-2-1: 동일 전문가의 해당 시간대 다른 서비스 일정 유무 확인
 5-2-2: 충돌 시, 다른 시간 제안 또는 재배정 필요 여부 판단
5-3: 채널/정책 기반 검토
 5-3-1: 채널별 일정 변경 가능 마감 시간 확인
 5-3-2: 일정 변경 수수료/제한 정책 존재 시 해당 여부 확인
6: 일정 변경 실행 – 기본 케이스 (주소/권역 불변, 전문가 유지)
6-1: "일정 변경" 액션 클릭
 6-2: 새로운 날짜/시간 입력
 6-2-1: service_date_new, service_time_new 선택
 6-2-2: 시스템에서 과거 시점, 영업 불가 시간대 차단
6-3: 시스템 처리 – 주문 정보 업데이트
 6-3-1: 주문의 service_date, service_time을 새로운 값으로 업데이트
 6-3-2: expert_progress_status는 RESERVED 상태 유지
 6-3-3: order_status, assignment_delivery_status는 변경 없음
6-4: 시스템 처리 – 캘린더 이벤트 업데이트
 6-4-1: 기존 이벤트를 삭제하지 않고 start_time, end_time을 수정한다
 6-4-2: event 변경 이력을 event_history에 기록한다(이전 시간 → 변경 시간)
 6-4-3: 전문가웹/앱의 일정 화면에서 해당 이벤트가 새 시간으로 표시된다
6-5: 알림 발송
 6-5-1: 고객에게 일정 변경 알림 발송 (SMS/알림톡 등)
 6-5-2: 전문가에게 일정 변경 알림 발송 (앱 푸시 등)
6-6: 메모 기록
 6-6-1: 주문 메모에 일정 변경 사유 및 변경 전/후 시간을 기록한다
 6-6-2: memo_type = SCHEDULE_CHANGE 로 저장
7: 일정 변경 실행 – 주소/권역 변경이 수반되는 케이스
7-1: 고객 주소 변경 요청 확인
 7-1-1: address, 상세주소, 동/호수 변경 여부 확인
 7-1-2: region_group이 변경되는지 여부 확인
7-2: region_group 변경 여부 판단
 7-2-1: 기존 주소 기반 region_group_old
 7-2-2: 변경된 주소 기반 region_group_new
 7-2-3: region_group_old ≠ region_group_new 인 경우, 기존 전문가가 해당 권역을 지원하는지 확인
7-3: 기존 전문가가 새 권역 지원 불가인 경우
 7-3-1: 현재 배정을 해제해야 할지 판단
 7-3-2: 정책에 따라 다음 중 하나 선택
 7-3-2-1: 기존 전문가 유지(예외 허용)
 7-3-2-2: 기존 배정 취소 후 UNASSIGNED로 되돌려 재배정(재배정 시나리오 연계)
7-4: 주소 정보 및 region_group 업데이트
 7-4-1: 주문의 address, region_group 값을 새로운 값으로 수정
 7-4-2: 고객 기본 주소와 동기화 여부는 별도 정책에 따름
7-5: 캘린더 이벤트 반영
 7-5-1: 동일 전문가 유지 시: 이벤트 시간/장소 메모만 수정
 7-5-2: 전문가 변경 시: 기존 이벤트 is_active = false 처리, 새 전문가 기준으로 신규 이벤트 생성
7-6: 메모 기록
 7-6-1: 주소/권역 변경 사유 및 전문가 유지/변경 여부 기록
 7-6-2: 정책 예외가 있었다면 반드시 명시
8: 일정 변경 실행 – 전문가 교체가 수반되는 케이스
8-1: 일정 변경 요청 + 기존 전문가 일정 충돌 발생
 8-1-1: 새 시간대에 해당 전문가의 다른 예약이 이미 존재
 8-1-2: 또는 전문가가 해당 시간에 불가 응답
8-2: 전문가 재배정 필요 여부 판단
 8-2-1: 고객이 날짜/시간 변경보다 전문가 교체를 우선하는지 확인
 8-2-2: 동일 권역 내 다른 전문가 후보 탐색
8-3: 전문가 변경 처리
 8-3-1: 기존 주문 배정 해제
 8-3-1-1: order_status: ASSIGNED → UNASSIGNED (필요 시)
 8-3-1-2: expert_master_id, expert_worker_id 초기화 또는 변경
 8-3-1-3: assignment_delivery_status: SENT → NONE
 8-3-1-4: order_assignment_history에 REASSIGN_FROM_SCHEDULE_CHANGE 기록
8-3-2: 새 전문가 배정
 8-3-2-1: 자동/수동 배정 로직을 사용해 새 전문가 선택
 8-3-2-2: ASSIGNED 및 SENT 상태로 전환
 8-3-2-3: 새 전문가 기준 캘린더 이벤트 생성
8-4: 고객/전문가 알림
 8-4-1: 전문가 교체 사실 + 변경된 방문 일정 고객에게 안내
 8-4-2: 새 전문가에게 신규 일정 알림 발송
8-5: 메모 기록
 8-5-1: 일정 변경으로 인한 전문가 교체 사유 기록
 8-5-2: 고객 동의 여부, 통화 내용 저장

9: 일정 변경 불가 케이스 처리
9-1: 일정 변경 요청이 정책 범위를 벗어나거나
 9-1-1: 너무 임박한 시간(예: 방문 1시간 전)
 9-1-2: 이미 서비스 완료/구매확정 상태
 9-1-3: 정산 확정 이후 특별 변경 요청
9-2: 변경 불가 안내
 9-2-1: 고객에게 불가 사유 설명
 9-2-2: 필요 시 취소/환불/재예약 대안 제시
9-3: 메모 기록
 9-3-1: 일정 변경 요청 거절 사유 기록
 9-3-2: 향후 CS 이력 확인용
10: 운영 리스크 및 정책 포인트
10-1: 일정 변경은
 10-1-1: 주문의 service_date/time
 10-1-2: 전문가 캘린더 이벤트
 10-1-3: 전문가 배정 상태
 를 동시에 다루므로, 반드시 트랜잭션/동기화 고려가 필요하다
10-2: 일정 변경 이력이 많을수록 CS/품질 이슈 가능성이 있으므로
 10-2-1: 주문별 일정 변경 횟수를 집계
 10-2-2: 특정 고객/전문가/채널에서 빈번하게 발생하는지 통계로 관리
10-3: 당일/임박 변경에 대한 수수료, 패널티, 페널티 점수 등은 별도 정책 문서와 연동
10-4: 강제 일정 변경(운영이 임의로 전문가나 일정을 바꾼 경우)은
 10-4-1: 반드시 메모와 함께 남기고
 10-4-2: 필요 시 "강제 상태 수정" 시나리오와 연계해 감사 가능해야 한다

### 시나리오 16: 서비스 금액 수정(상담 후 금액 조정)
본 시나리오는 상담 과정 또는 운영 개입에 의해 주문의 total_price를 변경해야 하는 경우를 정의한다.
본 기능은 결제·환불·정산에 직접적인 영향을 미치므로 권한 기반 접근 제어가 필요하다.
1: 관리자 로그인
1-1: 금액 수정 권한을 보유한 관리자 계정으로 로그인한다
 1-2: 권한이 없는 계정에는 "금액 수정" 버튼이 노출되지 않는다
2: 주문 상세 화면 진입
2-1: 주문관리 메뉴에서 대상 주문을 검색한다
 2-2: 주문 상세 화면으로 이동한다
 2-3: 현재 금액 상태 확인
2-3-1: total_price
 2-3-2: paid_amount_total
 2-3-3: refund_amount_total
 2-3-4: remaining_payable_amount = total_price - paid_amount_total
 2-3-5: 결제 트랜잭션 내역
3: 금액 수정 가능 여부 판단
3-1: 금액 수정 불가 상태
3-1-1: order_status = REFUNDED 이고 refund_amount_total = paid_amount_total 인 경우
 3-1-2: 정산 확정 완료 상태에서 환불 없이 금액 하향 조정 시
 3-1-3: 법적 분쟁으로 잠금 처리된 주문
3-2: 금액 수정 허용 상태
3-2-1: 결제 전 상태
 3-2-2: 일부 결제 상태
 3-2-3: 전액 결제 상태이지만 환불 가능 상태
 3-2-4: 서비스 진행 전 상태
4: 금액 수정 액션 실행
4-1: "금액 수정" 버튼 클릭
 4-2: 수정 입력 팝업 노출
4-2-1: 기존 total_price 표시
 4-2-2: 신규 total_price 입력 필드
 4-2-3: 변경 사유 코드 선택
4-2-3-1: OPTION_CHANGE
 4-2-3-2: PRICE_ADJUSTMENT
 4-2-3-3: DISCOUNT_APPLIED
 4-2-3-4: INPUT_ERROR
 4-2-3-5: INTERNAL_EXCEPTION
4-2-4: 상세 사유 메모 입력 필수
5: 시스템 사전 검증
5-1: 신규 total_price ≥ 0 확인
5-2: paid_amount_total와의 관계 검증
5-2-1: 신규 total_price ≥ paid_amount_total 인 경우
 → 추가 결제 필요 또는 정상 유지
5-2-2: 신규 total_price < paid_amount_total 인 경우
 → 초과 결제 발생
 → 부분 환불 필요 경고 표시
5-3: refund_amount_total와의 관계 검증
5-3-1: refund_amount_total > 신규 total_price 인 경우 차단
6: 금액 수정 분기 처리
6-1: 케이스 A: 결제 전 금액 수정
6-1-1: total_price를 신규 값으로 업데이트
 6-1-2: remaining_payable_amount 재계산
 6-1-3: 기존 발송된 결제 링크 자동 만료 처리
 6-1-4: 필요 시 신규 결제 링크 발송 가능 상태 유지
 6-1-5: order_status, expert_progress_status 변경 없음
6-2: 케이스 B: 일부 결제 상태에서 금액 상향 조정
6-2-1: total_price 증가
 6-2-2: remaining_payable_amount 증가
 6-2-3: 기존 결제 트랜잭션 유지
 6-2-4: 추가 결제 링크 발송 필요
 6-2-5: expert_progress_status = WAIT_PAYMENT 유지
6-3: 케이스 C: 일부 결제 상태에서 금액 하향 조정
6-3-1: 신규 total_price ≥ paid_amount_total 인 경우
 → remaining_payable_amount 감소
 → 추가 환불 없음
6-3-2: 신규 total_price < paid_amount_total 인 경우
 → 초과 결제 금액 = paid_amount_total - 신규 total_price
 → 자동 부분 환불 실행 여부 선택
6-3-2-1: 즉시 부분 환불 실행
 6-3-2-2: 환불 예약 상태로 전환
6-4: 케이스 D: 전액 결제 완료 상태에서 금액 하향 조정
6-4-1: 초과 금액 발생
 6-4-2: 초과 금액 = paid_amount_total - 신규 total_price
 6-4-3: 부분 환불 시나리오 9로 자동 연계
 6-4-4: refund_type = PARTIAL
7: 데이터 업데이트 처리
7-1: total_price 업데이트
 7-2: remaining_payable_amount 재계산
 7-3: 필요 시 refund_amount_total 업데이트
 7-4: 금액 변경 이력 기록
7-4-1: event_type = PRICE_UPDATED
 7-4-2: old_total_price
 7-4-3: new_total_price
 7-4-4: admin_id
 7-4-5: 변경 시각
7-5: 주문 메모 자동 생성
7-5-1: memo_type = PRICE_CHANGE
 7-5-2: 변경 전/후 금액 기록

### 시나리오 17: 수동 전문가 변경(운영 직접 교체)
본 시나리오는 자동 재배정이 아닌,  운영자가 특정 전문가를 지정하여 배정을 교체하는 경우를 정의한다.
본 기능은 일반 배정과 달리 "강제 배정" 성격을 가지며, 배정 정책을 우회할 수 있으므로 권한 통제가 필요하다.
1: 관리자 로그인
1-1: "전문가 수동 변경 권한"을 보유한 계정으로 로그인한다
 1-2: 권한이 없는 계정에는 "전문가 변경" 버튼이 노출되지 않는다
2: 주문 상세 화면 진입
2-1: 주문관리 메뉴에서 대상 주문을 검색한다
 2-2: 주문 상세 화면에 진입한다
 2-3: 현재 배정 상태 확인
2-3-1: order_status
 2-3-2: expert_progress_status
 2-3-3: assignment_delivery_status
 2-3-4: expert_master_id
 2-3-5: region_group
 2-3-6: service_date / service_time
3: 수동 변경 가능 여부 판단
3-1: 변경 불가 상태
3-1-1: order_status = CANCELED
 3-1-2: order_status = REFUNDED
 3-1-3: expert_progress_status = CONFIRMED (서비스 종료)
 3-1-4: 정산 확정 완료 상태
3-2: 변경 가능 상태
3-2-1: ASSIGNED 상태
 3-2-2: assignment_delivery_status = HOLD 또는 SENT
 3-2-3: UNASSIGNED 상태에서도 직접 지정 가능
4: 전문가 변경 액션 실행
4-1: "전문가 변경" 버튼 클릭
 4-2: 전문가 선택 팝업 노출
4-2-1: 필터 조건
 4-2-1-1: region_group 일치
 4-2-1-2: 서비스 카테고리 수행 가능 여부
 4-2-1-3: 활성 전문가만 표시
 4-2-1-4: 멤버십 여부 표시
4-2-2: 일정 충돌 체크 기능 제공
 4-2-2-1: 해당 날짜/시간 기존 예약 여부 표시
 4-2-2-2: 충돌 시 경고 메시지 노출
5: 변경 사유 입력
5-1: 사유 코드 선택 필수
5-1-1: CUSTOMER_REQUEST
 5-1-2: EXPERT_REQUEST
 5-1-3: SCHEDULE_CONFLICT
 5-1-4: QUALITY_ISSUE
 5-1-5: INTERNAL_EXCEPTION
5-2: 상세 사유 메모 입력 필수
6: 상태 전환 처리 로직
6-1: 기존 전문가 배정 해제
6-1-1: 기존 expert_master_id 저장
 6-1-2: 기존 assignment_delivery_status 종료 처리
 6-1-3: 기존 전문가 일정 이벤트 is_active = false
 6-1-4: order_assignment_history에 type = MANUAL_UNASSIGN 기록
6-2: 신규 전문가 배정
6-2-1: order_status = ASSIGNED 유지
 6-2-2: expert_master_id = 신규 전문가 ID
 6-2-3: assignment_delivery_status = SENT
 6-2-4: expert_progress_status = NEW_ASSIGN
 6-2-5: order_assignment_history에 type = MANUAL_ASSIGN 기록
 6-2-6: 배정 시각 기록
7: HOLD 상태 주문의 수동 변경
7-1: assignment_delivery_status = HOLD 상태인 경우
7-1-1: 기존 HOLD 배정 전문가 제거
 7-1-2: 신규 전문가 지정 시
 7-1-2-1: 멤버십 전문가일 경우 HOLD 유지 가능
 7-1-2-2: 일반 전문가일 경우 즉시 SENT 처리
7-1-3: HOLD 변경 이력 별도 기록
8: 캘린더 및 일정 처리
8-1: 기존 전문가 일정 삭제 또는 비활성화
 8-2: 신규 전문가 기준으로 신규 일정 이벤트 생성
 8-3: 전문가웹/앱 캘린더에 즉시 반영
9: 알림 처리
9-1: 기존 전문가에게 배정 해제 알림 발송
 9-2: 신규 전문가에게 신규 배정 알림 발송
 9-3: 고객에게 전문가 변경 안내 발송 여부는 정책 선택
10: 자동 배정 로직과의 관계
10-1: 수동 변경은 자동 라운드로빈 포인터에 영향을 주지 않는다
 10-2: 수동 변경된 주문은 재배정 자동 트리거 대상에서 제외 가능
 10-3: 멤버십 우선 정책을 우회할 수 있으므로 변경 이력 필수 기록
11: 정산 연동
11-1: 정산은 최종 수행 전문가 기준으로 계산
 11-2: 변경 전 전문가에게는 정산 영향 없음
 11-3: 이미 일부 서비스 진행된 경우 예외 처리 필요
12: 이력 및 감사 로그
12-1: event_type = MANUAL_EXPERT_CHANGE 기록
 12-2: 이전 전문가 ID
 12-3: 신규 전문가 ID
 12-4: 변경 사유 코드
 12-5: admin_id
 12-6: 변경 시각
12-7: 주문 메모 자동 생성
 12-7-1: memo_type = EXPERT_CHANGE

13: 운영 리스크 통제 포인트
13-1: 수동 변경은 배정 통계에 왜곡을 줄 수 있음
 13-2: 멤버십 우선 정책을 무력화할 수 있으므로 남용 방지 필요
 13-3: 일정 충돌 무시 배정은 시스템에 명확히 표시 필요
 13-4: 반복 교체 주문은 품질 이슈 가능성 모니터링
