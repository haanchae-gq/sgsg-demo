7. 메뉴 09 채널관리
시나리오 1: 신규 자사 채널 생성
1. 관리자 로그인
 : Admin 권한을 가진 사용자가 백오피스에 로그인한다.
2. 메뉴 이동
 : 상단 메뉴에서 “메뉴 09 채널관리”를 선택한다.
3. 채널 목록 화면 진입
 : channel 목록 화면에서 “채널 생성” 버튼을 클릭한다.
4. 채널 유형 선택
 : channel_type을 INTERNAL로 선택한다.
 : INTERNAL은 자사 채널을 의미한다.
5. 필수 입력값 입력
5-1. channel_code 입력
 : 내부 식별 코드이며 중복 불가 값이다.
 : 영문/숫자 조합을 권장한다.
5-2. channel_name 입력
 : 주문관리 및 전문가웹앱에 노출되는 공식 채널명이다.
 : 중복 여부를 검증한다.
5-3. channel_status 확인
 : 기본값은 ACTIVE로 설정된다.
6. 제휴 전용 필드 검증
6-1. channel_type = INTERNAL 이므로
 : partner_company_name 입력은 비활성 처리된다.
 : 제휴 담당자 정보 필드는 입력 대상이 아니다.
7. 저장 버튼 클릭
8. 서버 유효성 검증
8-1. channel_code 유일성 검증
 8-2. channel_name 유일성 검증
 8-3. channel_type 값 유효성 검증
 8-4. 필수값 누락 여부 검증
9. 저장 처리
9-1. channel_id 자동 생성
 9-2. created_at 자동 기록
 9-3. channel_status = ACTIVE 저장
 9-4. sort_order 자동 설정
10. 감사 로그 기록
10-1. 작업자 ID
 10-2. 작업 유형 = CREATE
 10-3. 생성 channel_id
 10-4. 생성 시각
11. 연동 영향 확인
11-1. 주문관리 연동
 : 신규 주문 생성 시 channel 선택 리스트에 해당 channel_id가 포함된다.
 : ACTIVE 상태이므로 주문 생성 허용된다.
11-2. 서비스관리 연동
 : 서비스관리 메뉴에서 채널 스코프 선택 시 해당 channel_id가 선택 가능하다.
11-3. 배정 로직 연동
 : channel_type = INTERNAL 값이 배정 조건 분기에서 참조 가능하다.
 : 기존 배정 로직에는 영향이 없다.
11-4. 전문가웹앱 연동
 : 주문 생성 후 해당 채널 주문이 배정될 경우
 : 전문가웹앱 주문 상세에서 channel_name과 channel_type만 노출된다.
12. 예외 케이스 처리
12-1. channel_code 중복 시
 : 저장 차단
 : “이미 사용 중인 channel_code입니다.” 오류 반환
12-2. channel_name 중복 시
 : 저장 차단
 : “이미 등록된 채널명입니다.” 오류 반환
12-3. 권한 부족 시
 : Admin 외 사용자 접근 시 생성 버튼 비노출
 : API 호출 시 권한 오류 반환
13. 최종 상태
13-1. channel_id 생성 완료
 13-2. channel_status = ACTIVE
 13-3. 신규 주문 생성 가능
 13-4. 타 메뉴에서 참조 가능
시나리오 3: 채널 기본 정보 수정
1. 관리자 로그인
 : Admin 또는 수정 권한이 있는 운영자가 백오피스에 로그인한다.
2. 메뉴 이동
 : 상단 메뉴에서 “메뉴 09 채널관리”를 선택한다.

4. 채널 상세 화면 진입
 : 수정하려는 channel_id를 클릭하여 상세 화면으로 이동한다.
5. 수정 가능/불가 필드 정의
5-1. 수정 가능 필드
5-1-1. partner_company_name
 : channel_type = PARTNER 인 경우에만 수정 가능
5-1-2. partner_contact_name
 : PARTNER 채널에 한해 수정 가능
5-1-3. partner_contact_email
 : 형식 검증 수행
5-1-4. partner_contact_phone
5-1-5. 내부 운영 메모(note)
5-2. 수정 불가 필드
5-2-1. channel_id
 5-2-2. channel_code
 5-2-3. channel_name
 5-2-4. channel_type
 5-2-5. created_at
6. UI 처리 정책
6-1. channel_name은 읽기 전용 필드로 표시한다.
 6-2. channel_code는 읽기 전용 필드로 표시한다.
 6-3. channel_type은 읽기 전용 필드로 표시한다.
 6-4. INTERNAL 채널일 경우 partner 관련 필드는 비노출 또는 비활성 처리한다.
7. 수정 입력 수행
7-1. PARTNER 채널의 경우
 : partner_company_name 수정 가능
 : 담당자 정보 수정 가능
7-2. INTERNAL 채널의 경우
 : note 필드만 수정 가능
8. 저장 버튼 클릭
9. 서버 유효성 검증
9-1. channel_id 존재 여부 검증
 9-2. 수정 불가 필드 변경 시도 여부 검증
 9-3. partner_email 형식 검증
 9-4. channel_type = PARTNER 인 경우 partner_company_name 필수 여부 재검증
10. 저장 처리
10-1. channels 테이블 UPDATE
 10-2. updated_at 자동 갱신
 10-3. updated_by 기록
11. 감사 로그 기록
11-1. 작업 유형 = UPDATE
 11-2. 변경 필드 목록 기록
 11-3. 변경 전 값 / 변경 후 값 기록
 11-4. 수정자 ID 및 시각 기록
12. 주문관리 연동 영향
12-1. channel_name이 수정 불가이므로
 : 기존 주문 및 신규 주문 모두 동일한 채널명 유지
12-2. 채널 코드 불변
 : 주문 엑셀 업로드 매핑 영향 없음
13. 서비스관리 연동 영향
13-1. channel_id 불변
 : 서비스-채널 스코프 매핑 영향 없음
14. 배정 로직 연동 영향
14-1. channel_type 불변
 : 배정 분기 조건 영향 없음
15. 예외 케이스
15-1. channel_name 수정 요청 API 호출 시
 : 400 오류 반환
 : “channel_name은 수정할 수 없습니다.” 메시지 반환
15-2. channel_type 수정 요청 시
 : 400 오류 반환
15-3. 권한 없는 사용자 수정 시
 : 403 오류 반환
16. 최종 상태
16-1. 채널 식별 핵심 필드(channel_id, channel_code, channel_name, channel_type)는 불변 유지
 16-2. 제휴 담당자 정보 및 내부 메모만 수정 가능
 16-3. 주문·서비스·배정 로직과의 정합성 유지

시나리오 6: 채널 목록 조회 및 필터 검색
1.관리자 로그인

2.메뉴 이동

3.채널 목록 기본 조회
: 채널관리 첫 진입 시 channel 목록이 기본 조건으로 조회된다.
: 기본 정렬은 created_at DESC 또는 sort_order ASC 정책 중 하나로 고정한다.
: 기본 필터는 channel_status = ALL 로 설정한다.
4.목록 컬럼 구성 확인
: 화면에는 다음 정보가 컬럼으로 표시된다.
4-1. channel_id
4-2. channel_code
4-3. channel_name
4-4. channel_type(INTERNAL, PARTNER)
4-5. channel_status(ACTIVE, INACTIVE)
4-6. partner_company_name(PARTNER 채널일 경우)
4-7. created_at
4-8. updated_at
5.검색 조건 입력 영역 확인
: 상단에 검색 조건 입력 영역을 표시한다.
5-1. 키워드 검색
5-1-1. channel_name 또는 channel_code 부분 일치 검색
5-2. 채널 유형 필터
5-2-1. channel_type = ALL, INTERNAL, PARTNER 선택 가능
5-3. 상태 필터
5-3-1. channel_status = ALL, ACTIVE, INACTIVE 선택 가능
5-4. 생성일 기간 필터
5-4-1. from_created_at, to_created_at 입력 가능
5-5. 제휴사명 필터
5-5-1. partner_company_name 부분 일치 검색(PARTNER 채널만 대상)
6.검색 조건 설정
6-1. 키워드 입력
: 운영자가 채널명 또는 channel_code 일부를 입력한다.
6-2. 채널 유형 선택
: 예를 들어 PARTNER만 보고 싶은 경우 channel_type = PARTNER 선택
6-3. 상태 선택
: ACTIVE 채널만 보고 싶은 경우 channel_status = ACTIVE 선택
6-4. 기간 지정
: 최근 30일 내 생성된 채널만 보고 싶은 경우 created_at 기간을 지정
7.검색 실행
7-1. “검색” 버튼 클릭
: 현재 입력된 조건으로 서버에 조회 요청을 보낸다.
7-2. 서버 처리
: WHERE 절에 다음 조건들을 조합한다.
7-2-1. channel_name LIKE 검색어 OR channel_code LIKE 검색어
7-2-2. channel_type 필터가 ALL이 아닌 경우 해당 값으로 필터
7-2-3. channel_status 필터가 ALL이 아닌 경우 해당 값으로 필터
7-2-4. created_at BETWEEN from_created_at AND to_created_at
7-2-5. partner_company_name LIKE 검색어(입력된 경우만 적용)
8.검색 결과 표시
8-1. 필터 적용 결과를 목록으로 출력한다.
8-2. 상단에 “총 N건”을 표시한다.
8-3. 페이지네이션을 적용한다.
8-3-1. page, page_size를 기반으로 서버에서 페이징 처리
8-3-2. 기본 page_size는 20 또는 50으로 설정
9.정렬 변경
9-1. 운영자는 컬럼 헤더를 클릭하여 정렬을 변경할 수 있다.
9-1-1. 가능한 정렬 대상: created_at, updated_at, channel_name, channel_type, channel_status
9-1-2. 기본은 created_at DESC
9-2. 정렬 기준을 변경하면
: 같은 검색 조건 내에서 정렬만 변경된 결과를 다시 조회한다.
10.상세 화면 진입
10-1. 목록에서 특정 row를 클릭하면
: 해당 channel_id의 상세 화면으로 이동한다.
10-2. 상세 화면에서 상태 전환, 정보 수정이 가능한 권한인지에 따라 버튼 노출 여부를 제어한다.
11.권한에 따른 표시 제어
11-1. Admin
: 전체 목록 조회 가능
: 상태 변경, 정보 수정 버튼 모두 노출
11-2. Operator(운영자)
: 전체 목록 조회 가능
: 상태 변경은 가능하지만 channel 생성은 제한 등의 정책을 둘 수 있다.
11-3. Viewer(읽기 전용)
: 목록 조회 및 상세 조회만 가능
: 생성/수정/상태 변경 버튼은 노출하지 않는다.
예외 케이스 처리
12-1. 검색 결과가 0건인 경우
: “검색 조건에 해당하는 채널이 없습니다.” 메시지를 표시
: 조건 초기화 버튼 제공
12-2. 비정상 필터 조합
: 예를 들어 from_created_at > to_created_at인 경우
: 검색 실행 전에 클라이언트 또는 서버에서 오류 메시지를 반환
12-3. 권한 오류
: 채널관리 메뉴에 접근 권한이 없는 사용자가 URL로 직접 접근할 경우
: “접근 권한이 없습니다.” 메시지와 함께 대시보드로 리다이렉트
13.최종 상태
13-1. 운영자는 채널 목록을 다양한 조건으로 조회·필터링할 수 있다.
13-2. ACTIVE/INACTIVE, INTERNAL/PARTNER 상태를 빠르게 파악할 수 있다.
13-3. 이후 세부 시나리오(생성, 상태 전환, 엑셀, 연계 검증)를 수행할 채널을 쉽게 선택할 수 있다.

시나리오 10: 주문관리 연계 검증
1.목적 정의
: 채널 생성, 상태 변경 이후 주문관리 메뉴에서 정상적으로 반영되는지 검증한다.
: channel_id, channel_code, channel_status가 주문 생성·조회 흐름에 정확히 적용되는지 확인한다.
2.사전 조건 설정
2-1. 테스트용 채널 존재
: channel_status = ACTIVE 인 채널 1개
: channel_status = INACTIVE 인 채널 1개
2-2. 권한
: 주문 생성 권한이 있는 운영자 계정 사용
3.ACTIVE 채널 주문 생성 검증
3-1. 관리자 로그인
3-2. 메뉴 02 주문관리로 이동
3-3. “주문 생성” 버튼 클릭
3-4. 채널 선택 리스트 확인
: channel_status = ACTIVE 인 channel_id만 노출되는지 확인한다.
: INACTIVE 채널은 선택 리스트에 표시되지 않아야 한다.
3-5. ACTIVE 채널 선택 후 주문 생성
: order.channel_id에 선택된 channel_id 저장
: order.channel_name_snapshot에 channel_name 저장
3-6. 주문 저장 후 상세 화면 확인
: 주문 상세 상단에 channel_name 표시
: channel_type 표시
: channel_order_no 입력 가능 여부 확인
4.INACTIVE 채널 주문 생성 차단 검증
4-1. 주문 생성 화면 진입
4-2. 채널 선택 리스트 확인
: INACTIVE 채널이 노출되지 않는지 확인
4-3. API 레벨 검증
: 클라이언트에서 channel_id를 수동으로 전달하여 주문 생성 시도
: 서버에서 channel_status = ACTIVE 여부 검증
: INACTIVE인 경우 400 오류 반환
: “비활성 채널로는 주문을 생성할 수 없습니다.” 메시지 반환
5.엑셀 업로드 주문 연계 검증
5-1. 주문 엑셀 업로드 파일 준비
: channel_code 컬럼 포함
5-2. 업로드 수행
5-3. 서버 처리 로직
5-3-1. channel_code → channel_id 매핑
5-3-2. channel_status = ACTIVE 여부 검증
5-3-3. INACTIVE 채널일 경우 해당 row 실패 처리
5-3-4. 부분 성공 정책 적용
5-4. 업로드 결과 검증
: ACTIVE 채널 주문은 생성
: INACTIVE 채널 주문은 실패 로그 기록
6.주문 상세 화면 연동 검증
6-1. 기존 주문 조회
: order.channel_id 유지
: order.channel_name_snapshot 유지
6-2. 채널이 이후 INACTIVE로 전환되었더라도
: 기존 주문의 조회 및 상태 전이 정상 진행
7.채널 상태 변경 이후 신규 주문 검증
7-1. ACTIVE → INACTIVE 전환 후
: 신규 주문 생성 시 채널 선택 불가
7-2. INACTIVE → ACTIVE 전환 후
: 채널 선택 가능 상태 복구
8.주문 필터 연동 검증
8-1. 주문 목록 필터에 channel_id 조건 적용
: 특정 channel_id 기준 주문 검색 가능
8-2. channel_type 기준 필터 확장 가능
: INTERNAL / PARTNER 기준 조회
9.데이터 무결성 검증
9-1. order.channel_id는 외래키로 유지
9-2. channel_id 삭제 개념이 없으므로 FK 무결성 보장
9-3. channel_name 변경 불가 정책으로 스냅샷 불일치 문제 없음
10.예외 케이스
10-1. channel_id 존재하지 않는 값으로 주문 생성 시도
: 400 오류 반환
10-2. channel_status NULL 또는 비정상 값 존재 시
: 주문 생성 차단
11.감사 로그
11-1. 주문 생성 시 channel_id 기록
11-2. 엑셀 업로드 실패 row 로그 기록
12.최종 상태
12-1. 채널 ACTIVE 상태만 신규 주문 생성 가능
12-2. INACTIVE 채널은 신규 유입 차단
12-3. 기존 주문 데이터 보호
12-4. 주문관리와 채널관리 간 정합성 확보

시나리오 11: 서비스관리 연계 검증
1. 목적 정의
 : 채널 생성 및 상태 변경이 서비스관리 메뉴(메뉴 03)의 채널 스코프 설정에 정확히 반영되는지 검증한다.
 : channel_id, channel_status가 가격/수수료 설정 스코프에서 정상 동작하는지 확인한다.


2. 사전 조건 설정
2-1. 테스트용 채널 준비
 : channel_status = ACTIVE 인 채널 1개
 : channel_status = INACTIVE 인 채널 1개
2-2. 테스트용 서비스 준비
 : 서비스 중분류 1개 이상 존재
 : 채널 스코프 설정이 가능한 상태
3. ACTIVE 채널 스코프 선택 검증
3-1. 관리자 로그인
 3-2. 메뉴 03 서비스관리 이동
 3-3. 특정 서비스 중분류 선택
 3-4. “채널별 가격/수수료 설정” 탭 진입
3-5. 채널 스코프 선택 드롭다운 확인
3-5-1. channel_status = ACTIVE 채널만 노출되는지 확인
 3-5-2. INTERNAL / PARTNER 모두 표시 가능
3-6. ACTIVE 채널 선택 후 가격/수수료 설정 저장
3-6-1. service_channel_scope 테이블에
 : service_category_mid
 : channel_id
 : 가격/수수료 값 저장
3-7. 저장 완료 후 목록 재조회
 : 정상 저장 여부 확인
4. INACTIVE 채널 스코프 선택 제한 검증
4-1. 동일 화면에서 채널 선택 드롭다운 확인
 : INACTIVE 채널은 선택 리스트에 노출되지 않아야 한다.
4-2. API 레벨 검증
 : INACTIVE channel_id를 강제로 전달하여 저장 요청
 : 서버에서 channel_status = ACTIVE 여부 검증
 : INACTIVE인 경우 400 오류 반환
 : “비활성 채널은 신규 스코프로 설정할 수 없습니다.” 메시지 반환
5. 기존 스코프 유지 정책 검증
5-1. ACTIVE 상태에서 서비스–채널 매핑 생성
 5-2. 해당 채널을 INACTIVE로 전환
5-3. 서비스관리 재조회
5-3-1. 기존 service_channel_scope 데이터는 유지되어야 한다.
 5-3-2. 기존 가격/수수료 값은 삭제되지 않는다.
5-4. 단, 신규 스코프 설정은 불가
6. 채널 재활성 시 스코프 복구 검증


6-1. INACTIVE → ACTIVE 전환

시나리오 12: 배정 로직 연계 조건 확인
1. 목적 정의
 : 채널 정보(channel_id, channel_type)가 배정 로직(메뉴 10)으로 정확히 전달되고, 채널을 조건으로 한 배정 정책이 의도대로 동작하는지 검증한다.
 : 현재 v1 기준 “채널 무관 동일 배정”이 기본이며, 향후 INTERNAL/PARTNER 분기 또는 특정 채널 예외 정책을 추가할 경우에도 검증이 가능하도록 한다.
2. 사전 조건 설정
2-1. 채널 준비
 2-1-1. channel_type = INTERNAL, channel_status = ACTIVE 채널 1개
 2-1-2. channel_type = PARTNER, channel_status = ACTIVE 채널 1개
2-2. 전문가 준비
 2-2-1. 동일 service_category_mid에서 배정 가능한 전문가 2명 이상
 2-2-2. 동일 region_group 내에 위치
 2-2-3. approval_status = APPROVED, active_status = ACTIVE
2-3. 서비스·배정 로직 준비
 2-3-1. 메뉴 03 서비스관리에서 테스트 서비스 중분류에 대해 정상 설정 완료
 2-3-2. 메뉴 10 배정 로직 관리에서 기본 배정 버전 활성화
 2-3-3. v1 기준 채널별 차등 배정은 사용하지 않는 상태를 기본으로 한다.
3. 주문 생성 시 채널 정보 전달 검증
3-1. 관리자 로그인
 : 주문 생성 권한이 있는 운영자가 백오피스에 로그인한다.
3-2. 메뉴 이동
 : 메뉴 02 주문관리로 이동한다.
3-3. INTERNAL 채널 주문 생성
 3-3-1. 주문 생성 화면에서 channel_id로 INTERNAL 채널을 선택한다.
 3-3-2. service_category_mid, region_group, 예약일 등을 입력한다.
 3-3-3. 주문 저장 시 order.channel_id, order.channel_type = INTERNAL 값이 저장되는지 확인한다.
3-4. PARTNER 채널 주문 생성
 3-4-1. 주문 생성 화면에서 channel_id로 PARTNER 채널을 선택한다.
 3-4-2. 동일한 조건의 서비스 중분류, region_group으로 주문을 생성한다.
 3-4-3. order.channel_id, order.channel_type = PARTNER 값이 저장되는지 확인한다.
3-5. 배정 엔진 입력 파라미터 검증
 3-5-1. 두 주문에 대해 자동배정 요청 시
 3-5-2. assignment_request에 service_category_mid, region_group와 함께 channel_id, channel_type이 포함되는지 확인한다.
4. 채널 무관 동일 배정 정책(v1 기본) 검증
4-1. INTERNAL 주문 자동배정 실행
 4-1-1. INTERNAL 채널 주문에 대해 자동배정 트리거를 실행한다.
 4-1-2. 배정 후보군 필터
 4-1-2-1. service_category_mid 일치
 4-1-2-2. region_group 일치
 4-1-2-3. approval_status, active_status, 패널티, 일일 상한 필터 적용
 4-1-3. 채널 기준 추가 필터가 없는지 확인한다.
 4-1-4. 멤버십 가중치 및 슬롯 로직이 채널과 무관하게 동일 적용되는지 확인한다.
4-2. PARTNER 주문 자동배정 실행
 4-2-1. PARTNER 채널 주문에 대해 자동배정 트리거를 실행한다.
 4-2-2. 동일한 서비스·권역 조건에서 후보군 구성이 INTERNAL 주문과 동일해야 한다.
 4-2-3. INTERNAL 주문과 동일한 배정 엔진 버전, 동일한 필터/가중치 로직이 사용되는지 확인한다.
4-3. 결과 비교
 4-3-1. INTERNAL 주문과 PARTNER 주문 모두에서, 동일 시점·동일 조건일 때 동일한 후보군과 가중치 결과가 나오는지 비교한다.
 4-3-2. 채널이 단지 메타 정보로만 사용되고, v1 기준 배정 우선순위에는 영향을 주지 않는다는 것을 검증한다.
5. 채널 타입 기반 조건 분기 설계 검증(고도화 플래그 ON 시)
5-1. 메뉴 10에서 채널 분기 옵션 활성화
 5-1-1. 배정 로직 설정에서 channel_type 기반 분기 플래그를 활성화한다.
 5-1-2. 예시 정책 1
 5-1-2-1. PARTNER 채널 주문은 멤버십 우선 배정을 사용하지 않음
 5-1-2-2. INTERNAL 채널 주문만 멤버십 가중치 적용
5-2. INTERNAL 주문 재배정
 5-2-1. INTERNAL 채널 주문에 대해 자동배정 실행
 5-2-2. 후보군 분리
 5-2-2-1. 멤버십 그룹과 일반 그룹으로 나뉘고, 멤버십 그룹 우선 라운드로빈이 작동하는지 확인한다.
5-3. PARTNER 주문 재배정
 5-3-1. PARTNER 채널 주문에 대해 자동배정 실행
 5-3-2. 후보군 분리
 5-3-2-1. 설정된 정책에 따라 멤버십 가중치가 무시되고, 모든 전문가가 동일한 기본 가중치로 라운드로빈 되는지 확인한다.
5-4. 채널 분기 로직 안정성 검증
 5-4-1. 설정 플래그를 다시 OFF 했을 때
 5-4-2. INTERNAL·PARTNER 모두 동일 로직으로 되돌아오는지 확인한다.
 5-4-3. 버전 관리(메뉴 10 배정 로직 버전) 기반으로 ON/OFF 상태가 명확하게 구분되는지 확인한다.
6. 특정 채널 예외 정책 검증(채널 단위 override 설계 시)
6-1. 예시 정책
 6-1-1. 특정 PARTNER 채널은 자동배정을 사용하지 않고 항상 HOLD 처리한다.
 6-1-2. 또는 특정 PARTNER 채널은 특정 전문가 풀만 사용한다.
6-2. 메뉴 10에서 채널 예외 설정

6-3. 해당 채널 주문 생성 및 배정

 6-3-2. 자동배정 실행 시

6-4. 다른 채널 영향 검증

7-1. 배정 로그에 channel 정보 포함 여부

7-2. 배정 실패 로그에서 채널 조건 확인

 7-2-2. failure_reason 또는 filter_step_stats에 channel 관련 사유가 명시되는지 확인한다.

8-1. 채널 유형과 무관하게

 8-1-2. INTERNAL / PARTNER에 따라 다른 상태머신을 사용하지 않는다는 것을 검증한다.
9. 예외 케이스 검증
9-1. 존재하지 않는 channel_id를 가진 주문
 9-1-1. 주문 생성 단계에서 차단되어야 하므로
 9-1-2. 배정 엔진에는 들어오지 않는지 확인한다.
9-2. channel_status = INACTIVE 인 채널 주문
 9-2-1. 설계 상 신규 주문이 생성되지 않아야 한다.
 9-2-2. 만약 과거에 생성된 주문이라면
 9-2-3. 배정 로직은 정상적으로 동작해야 하며, channel_status에 의해 배정이 차단되지 않아야 한다.
10. 최종 상태
10-1. 채널 정보는
 10-1-1. 주문 생성 시 정확히 저장되고

 10-1-3. 정책에 따라 분기 조건으로 활용될 수 있다.

10-3. 향후 고도화 시
 10-3-1. channel_type 또는 특정 channel_id를 조건으로 한 배정 예외 정책을 추가해도
